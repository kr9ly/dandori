# 付録: formal ゲート — プロパティベーステストと形式検証

dandori の品質ゲートの拡張タグ `formal` に関する付録。
発動条件・道具の選び方・運用上の罠をまとめる。プロセス本体ではなく、
条件に引っかかったときに開く参照ドキュメント。

元ネタ: [実装仕様化と形式検証ワークフロー（mizchi）](https://gist.github.com/mizchi/db7817e6fc077d567c41cd9d41bb1c53)

## 発動条件 — 「複雑さ」ではなく「主張の形」で判定する

形式検証の出番は複雑なシステムではなく、仕様の主張が全称形になった瞬間に来る。
dandori-spec で B 行にゲートタグを付ける際、以下に該当したら `formal` を検討する:

- Then が「**どんな入力でも**〜」という全称主張になっている
  （例: どの設定の組み合わせでも遮断が優先される / どんな金額列でも合計が上限を超えない）
- Then が「**どの実行順でも**〜」という主張になっている
  （例: リトライが何回どの順で走っても二重処理されない）
- `unit` タグを付けようとして「代表的な入力を数個試すだけでは不安が残る」と感じた

この感覚が出ない B 行に formal を適用しない。テストで十分なものはテストで担保する。

## 形式検証に向くバグの型（ダイジェスト）

複雑なコードではなく、ありふれたコードで起きる型が大半:

| 型 | 例 |
|----|-----|
| 矛盾設定 | 条件の組み合わせが充足不能（どう設定しても発動しないルール） |
| 被覆の穴 | どの条件にもマッチしない入力が存在する（デフォルト経路の意図しない発動） |
| fail-open/close 非一貫 | エラー時の振る舞いが機構ごとに違う |
| 否定の畳み込み | 「エラー」と「不一致」を同じ false に潰し、否定条件で素通り |
| 境界 | > と >= の取り違え（cap-1, cap, cap+1 は必ず検証対象に含める） |
| 空集合の意味 | AND(∅)=真 / OR(∅)=偽 の混在 |
| 冪等性喪失 | リトライで二重カウント |
| read-modify-write レース | 読取→判定→書込の分離による上限すり抜け |

上 6 つは述語の問題（Z3 / 全列挙向き）、下 2 つは実行順の問題（モデル検査向き）。

## 道具の選択 — 段階的エスカレーション

プロパティベーステスト（PBT）と SMT ソルバー（Z3）は**同じ全称命題を書いて、
検証エンジンだけが違う**。この同一性を利用して段階的に投入する:

| | PBT | Z3 |
|---|---|---|
| 実行対象 | 本物のコード | コードから抽出したモデル |
| 保証 | サンプリング（数千件で反例なし） | 全入力空間（決定可能な範囲で証明） |
| モデルとコードのギャップ | 原理的にゼロ | 存在する（最大の弱点） |
| 導入コスト | テストを書くのと同程度 | モデル記述 + SMT の知識 |

### Step 1: プロパティを1回だけ書く

B 行の Then を述語 `P(入力)` として言語化する。これが共有資産で、
PBT と Z3 の両方がこの同じ述語を検証する。

### Step 2: PBT で本物の実装に回す

モデル化不要・即日動く・CI に常駐できる。まずここまでで止めてよい。
`formal` タグの最低ラインは「PBT で担保」。

- Kotlin/Android: Kotest property testing / jqwik
- TypeScript: fast-check
- Go: pgregory.net/rapid

### Step 3: 決定可能ならZ3 に昇格する

述語のコアが区間・集合・bool 結合（決定可能断片）に落ちるなら、
サンプリングを証明に格上げできる。言語バインディングは薄いものが多いので、
**対象データを読んで SMT-LIB テキストを吐く小さなジェネレータ + `z3` CLI** が
最小構成。生成された SMT-LIB は人間が監査できる。

有限で決定的な入力空間なら、Z3 すら不要 — 素の全列挙が最安。

### 相互検証

PBT と Z3 を同じプロパティで両方回すと、結果の食い違いがモデルとコードの
ギャップ自体を検出する:

- PBT だけが反例を出す → モデルが現実より甘い（モデルの修正）
- Z3 だけが反例を出す → 実装がまだ踏んでいないだけの穴（先回りの修正）

## 運用上の罠（エージェントに任せる場合は必須ゲート）

### broken-variant 検証

わざと壊した実装（ガード外し・境界の反転）を用意し、検査が**赤で捕まえる**ことを
確認してから緑を信用する。エージェントは「もっともらしいモデルを書いて緑にして
検証済みと報告する」失敗を最もやりやすい — 実装を読まずに記憶からモデルを
組成すると、変換の誤りがそのまま偽証明になる。dandori-impl の
「テストを弱める変更の禁止」と同型の防御であり、formal ゲートでは省略不可。

### 撤退基準

主要な不変条件・危険な穴を押さえたら止める。確認的な命題の量産は CI 負荷にしか
ならない。形式化しないと決めた B 行は通常のゲートタグに戻し、判断を記録する。

### 成果物は台帳

証明の山ではなく「実装の主張 / 機械検査の結果 / 未解決の確認質問」の一覧を残す。
dandori では spec.md の B 行 + dandori-gate のトレース表がこの台帳を兼ねる。

## dandori への組み込み

- **dandori-spec**: ゲートタグに `formal` を追加可能。付与条件は本付録の発動条件
- **dandori-plan**: `formal` 行のゲートは「プロパティ定義 + PBT 実行コマンド」まで具体化する
- **dandori-gate**: `formal` 行は PBT（+ Z3 があればそれも）を再実行し、
  broken-variant が赤になることの確認記録を根拠欄に残す
