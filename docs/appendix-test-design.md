# 付録: テスト設計技法 — dandori が暗黙に使っている古典技法の対応表

dandori の spec / 状態モデル / 地雷リストの設計判断は、古典的テスト設計技法の適用に
なっている。この付録は「どの成果物のどの判断が、どの技法か」を明文化し、
執筆時の妥当性基準とレビュー時の監査観点に名前を与える。

appendix-formal / appendix-state-model と同じく、条件に引っかかったときに開く参照
ドキュメント。技法名を挙げるだけの儀式チェックリストにしないこと — 各技法には
**判定基準**（執筆時に自問する形）と**実例**（効いた事例・失敗事例）をセットで載せる。

## §1 同値分割 — 軸の値域設計の正体

軸に値を並べる行為は同値クラスの宣言である: **「このクラス内のどの具体値でも
Then が同じ」という主張**。B 行のパラメタライズ（(a)(b)(c) で複数具体値を 1 行に
束ねる）も同じ主張の B 行版。

執筆時の判定基準:

1. **クラス内一様性** — クラス内のどの値を選んでも Then が変わらないと言い切れるか。
   言い切れない差異を見つけたら値を分割する（例: cp_ok から cp_ok_scoped を分離 —
   対象限定の有無で Then が変わる）
2. **クラス漏れ** — 入力空間にどのクラスにも属さない値がないか。「その他」に落ちる
   値の挙動を必ず 1 クラスとして持つ（例: get_bool の未知文字列 → デフォルト値クラス。
   True/False の 2 クラスだけだと「"1" を渡したとき」が無宣言になる）
3. **多重度の罠（クラス内非一様性の典型）** — 「該当データが複数あるとき」は
   単数クラスと同値ではないことが多い。**複数候補から 1 件を選ぶ処理は選択順序が
   観測可能な仕様**（.first() の暗黙 pk 昇順、タイブレーク）。単数/複数を別クラスに
   するか、複数時の選択規則を Then に書く

実例: get-cart review R-21（2026-07-09）は「開催中 Sale / 適格 FixedDiscount /
同一 nell_product_id の ProductInfo が複数あるとき」の first-wins 3 本（B-69〜71）を
発掘した — 判定基準 3 の失敗をレビューアが偶発的に拾った事例。技法名で指示すれば
系統的に探せる（review の監査観点に組み込み済み）。

## §2 base-choice testing — ペアワイズをやらない理由

状態モデルのカバレッジ戦略「基準シナリオ（base）+ 各軸 1 変化 + 宣言された依存交点」は
組み合わせテスト理論の **base-choice coverage**（NIST ACTS の用語）に相当する。
全ペア網羅（pairwise / 2-way coverage）より弱いカバレッジを意図的に選んでいる。

その正当化が**直交宣言の reason**: 「この 2 軸は独立」という宣言に spec 本文から
導ける根拠がある限り、そのペアの交点テストは空虚（どちらかの軸のテストで代表される）。
つまり dandori は「pairwise − 根拠つき直交ペア = 依存交点のみ」という削減をしており、
reason 必須・review による直交監査（裁定のふり検査）はこの削減の安全弁である。

執筆時の判定基準: 直交宣言を書くとき「もしこのペアに交点挙動があったら、それは
どの技法でも検出されない」と自覚すること。reason が書けないなら ground 送り
（appendix-state-model の運用どおり）。

## §3 境界値分析 — Given の具体値の選び方

B 行の Given に具体値を選ぶとき、クラスの内部代表値ではなく**境界**を選ぶ:

- 個数: 0 / 1 / 複数（同値分割の単数・複数クラスと連動）
- 数値: 0・負値・丸め境界（.5 — 言語間で丸め方向が割れる値）・型の桁上限
- 日付: 月末（月加算のクランプ）、年末年始、うるう年、タイムゾーン境界
- 文字列: 空文字 vs null vs 欠落（3 態は Python/TS 間で最も割れる境界 —
  js-python-semantic-gaps 参照）

実例: get-cart B-47（丸め away-from-zero、-0.5→-1 が Math.round と割れる）、
B-62（relativedelta 月末クランプ、3/31 + 11 か月 → 2/28）。どちらも境界値でしか
検出できない言語間非互換で、内部代表値のテストでは緑のまま非互換が残る。

## §4 デシジョンテーブル — 異常系マトリクスの正体

dandori-spec §3 の「状態変数を列挙し、各変数 × 正常系シナリオを機械的に走査する」は
デシジョンテーブルテストの条件記入部の全数走査に相当する。直感で「ありそうなエラー」を
挙げる方式（= 技法なし）との違いは網羅の機械性にある — 工程として独立させている理由も
「エージェントはエラーパスの列挙を体系的にサボる」ため。

判定基準: 表の各セル（変数×シナリオ）に「挙動が変わる/変わらない」の判定を
書いたか。「変わらない」の判定も裁定であり、書かずに飛ばしたセルは検査されていない。

## §5 状態遷移テスト — chains と state-map

状態モデルの `chains:`（続落宣言）と state-map の遷移宣言は状態遷移テストの
遷移網羅に相当する。判定基準: 宣言した遷移それぞれに、遷移を跨ぐ B 行
（`chain:` Covers）があるか — チェッカー検査 5 が機械化済み。
未宣言の遷移は要求できない（appendix-state-model「検出できないもの」1）ため、
遷移の発見自体は ground / review の仕事。

## §6 エラー推測の構造化 — 地雷リストと precedents

古典の error guessing（経験者の勘によるバグ推測）を、dandori は 3 つの永続
カタログに構造化している:

- **地雷リスト**（plan.md）: このフィーチャーで「一般的に正しいパターン」が
  非互換になる箇所の列挙。実装エージェントの無知な勘を打ち消す
- **precedents**: プロジェクト横断の確定裁定。同種論点の再裁定と FP を防ぐ
- **意味論ギャップ集**（js-python-semantic-gaps 等）: 言語・フレームワーク間の
  既知の割れ目。境界値分析（§3）の値選択の情報源

判定基準: 勘で見つけた地雷は必ずどれかのカタログに落とす（会話にだけ存在する
知見を作らない — 台帳と同じ原則）。

## 運用 — どの工程で何を参照するか

- **dandori-spec §2〜4**: 軸の値域を切るとき §1、直交宣言を書くとき §2、
  Given の具体値を選ぶとき §3 の判定基準で自問する
- **dandori-review**: レビューア指示テンプレートの監査観点に §1 の判定基準
  （クラス漏れ・クラス内非一様性/多重度）が組み込まれている
- **dandori-plan**: 地雷リストの執筆は §6 — カタログ 3 種からの転記漏れを確認する
