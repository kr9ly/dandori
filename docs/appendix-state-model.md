# 付録: 状態モデル — Given 空間の宣言とテストパス抜けの静的検査

dandori-spec の §2（正常系バリエーション）/ §3（異常系マトリクス）が散文でやっている
「軸分け・直交宣言・交点列挙」を宣言的な状態モデルに格上げし、B 行の抜けを機械検査する。
appendix-formal.md と同じく、条件に引っかかったときに開く参照ドキュメント。

前例: category-partition method / TSL（Ostrand & Balcer 1988）、PICT、NIST ACTS/CCM。
実証: `experiments/state-model/findings.md` — R23 通過済み spec から新規指摘 6 件、
レビュー指摘（R4 blocker-2 型）のバックテスト完全ヒット。

## 発動条件

- §2 で**軸分けが発動した**（バリエーション目安 10 超）spec — 軸が 3 つ以上あれば元が取れる
- §3 で**フェーズ×リソース型の横断関心事**（ロールバック・補償処理・通知）が出た spec
- 該当しない小さい spec（B 行 10 本未満・軸なし）ではスキップしてよい。
  ただし状態変数が 2 個以上あるなら「2 変数の同時発生」の交点だけは意識すること
  （cancel-order-product B-38/B-39 はこの型の取り落としだった）

## spec.md への書き方

### 状態モデルブロック

`## 状態モデル` セクションに info string 付き fenced block で置く:

````markdown
## 状態モデル

```dandori-state-model
axes:
  payment:
    label: "軸1: 決済メソッド"
    base: S1
    values: [S1, S2, S3, S4, S5, S6]
  stock:
    label: "軸3: 在庫割り当て"
    base: S11?            # `?` = 推定 — 基準シナリオが軸を明言していない（ground 送り扱い）
    per_item: true        # 商品単位で値を取る — 混在の裁定が必要
    values: [S11, S12]
observations: { S14: { of: [stock] } }          # 他軸の帰結の観測 — 直積から除外
modifiers: { S22: { affects: [ERP, 購入通知] } } # 他シナリオの出力の修飾 — 同上
orthogonal:
  - { axes: [wms, payment], reason: 出庫結果は WMS レスポンスのみで決まる }
  - { axes: [similar_return, "*"], reason: 過去注文データのみに依存 }   # "*" = 全軸と直交
orthogonal_groups:
  - { group: [route, order_no, pickup], reason: 付帯状態は相互に独立 }  # グループ内全ペア直交
dependent:
  - { cells: { payment: S5, coupon: S7 }, note: pending は ERP スキップ }
only:
  - { axis: order_no, value: preassigned, requires: { payment: [S4, S5] }, note: WalletPay 限定 }
chains:
  - { from: S16, to: S18, note: 422 再送失敗は仮宛先チェーンへ続落 }
ground:
  - { axes: [wms, pickup], note: 直交か依存か spec からは裁定不能 — コードで確定 }
  - { axes: [stock, stock], note: 混在時の引当順序が未記述 — 自己ペア = per-item 混在の裁定送り }
```
````

- **共有状態に触れる軸には `ref:` を付ける**（`ref: order_status` — states.md の
  状態マップの canonical ID を参照）。`ref:` 付き軸の値は canonical 値 ID を使う。
  構文と検査は `docs/appendix-state-map.md` を参照。フィーチャーローカルな軸は従来どおり
- 値 ID は §2 の S 番号を正とする（S 番号のない発掘軸は S4a〜c のように命名して軸化する）。
  S 番号の意味（散文）は従来どおり軸説明に書く — モデルブロックは索引であり
  人間向けの正文を置き換えない。値に注記が要るときは `{ id: other, note: "..." }`
  （例:「値は入力に存在するが出力に反映されない」強制上書き型は note の散文で表現する）
- 異常系は**横断関心事のみ**軸化する（裁定 2026-07-06）: 失敗タイミング
  （フェーズ構造をそのまま値域に写す）× 確保済みリソース（Shipping 状態など）+ only 制約。
  エラーマトリクス全行の形式化はしない — E 行↔B 行の突合は gate トレースの仕事。
  形式化しない異常系 B 行の Covers は `out-of-model — 異常系 EXX` と書く
- **`ground:` は「裁定の先送り」の機械表現**。spec 本文から直交/依存を裁定できないペアは
  吟味なしで直交宣言せず ground 送りにする。base の `?` も同種 — どちらもチェッカーが
  指摘とは別枠の「ground 送り項目」として出力し、ground 工程の確認項目リストになる

### B 行の Covers フィールド

```markdown
### B-4: wallet_pay 与信 pending（S5）
- Covers: base + payment=S5
- Given: ...（散文はこれまでどおり）
```

- デルタ構文。未言及の軸は基準値（base）に展開される
- パラメタライズは `payment=S1|S2`（どの値でも同じ挙動 — 1 行で複数値をカバー）
- 混在は `stock=S11+S12`（per-item 軸のみ — 1 注文内に複数値が同居する Given）
- 観測・修飾・連鎖は `Covers: base + obs:S14` / `mod:S22` / `chain:S16`
- 単発バグ再現行は `Covers: one-off — 理由`（直積検査から除外 — 専用軸を立てない）
- モデル対象外の行（形式化しない異常系マトリクス等）は `Covers: out-of-model — 理由`。
  one-off（単発バグ再現専用）と意味を分ける（実験 5 の摩擦 1 の解消）。
  レポートの除外行列挙は 8 件超で件数のみに折りたたまれる

全構文の実例: `experiments/state-model/format-sample.spec.md`（チェッカーの回帰フィクスチャ兼用）

### flow（任意）— spec ツリーの背骨

基準シナリオのステップ列と、各軸が「どのステップで分岐するか」の錨付け。
カバレッジ検査には関与しない — spec ツリーレンダラー（後述）の描画情報:

```yaml
flow:
  - { step: "割引適用・金額計算", ask: "割引は？", axes: [coupon, promo, fixed_discount] }
  - { step: "決済オーソリ", axes: [payment] }   # ask 省略時は軸ラベルから生成
```

- 1 軸は 1 ステップにのみ錨付けできる（重複はチェッカーが定義エラーにする）
- flow に載せなかった軸は末尾に補完されるので、描画から軸が消えることはない
- 壁打ち §2 で基準シナリオを書き下ろす行為がそのまま flow の執筆になる — 追加コストはほぼない

## モデリングの実務ノウハウ（実験で判明した罠）

1. **排他値のふりをした独立変数を分解する**。cart-payment の「軸2: 割引（4態）」は
   実際は coupon / promo / fixed_discount の独立 3 軸だった（spec 自身が B 行内で
   組成を認めていた）。「軸5: 付帯状態（6態）」も独立ブールの束。
   分解の判定: 2 つの値が 1 リクエストで同時に成立し得るなら別軸
2. **observation / modifier を軸にしない**。直積に入れると無意味な組み合わせ要求で
   ノイズが爆発する。「Xは Y の結果の観測」「X は他シナリオの出力を変えるだけ」と
   言えるものは種別を分ける
3. **per_item 軸は混在の裁定を忘れない**。商品単位の軸（在庫割り当て等）は
   1 注文内の値混在が成立する — 混在時の挙動を直交と宣言するか交点として列挙する
4. **軸の分解作業自体が仕様バグの発見源**。Then の奥に散文で埋まった相互作用
   （「〜の場合のみ」「〜は除外」）は依存宣言への昇格候補 — 分類時に B 行の Then を
   注意深く読むこと

## チェッカー

spec.md を直接パースして検査する（中間ファイル不要）:

```bash
node <dandori-repo>/skills/dandori-spec/scripts/check-state-model.ts <spec.md へのパス>
# 終了コード: 0 = グリーン（ground 送り項目は残ってよい）/ 1 = 指摘あり / 2 = 定義エラー
```

| # | 検査 | 指摘の意味と対応 |
|---|------|-----------------|
| 0 | Covers 欠落 | B 行にモデル対応がない → Covers を書くか one-off / out-of-model 宣言 |
| 1 | 値カバレッジ | テストパスの抜け → B 行を追加 |
| 2 | 依存交点カバレッジ | 交点主張が空虚に真になるリスク → 組み合わせ Given の B 行を追加。per-item 軸は base 省略で満たせない（明示言及が必要） |
| 3 | ペア分類の全数性 | 未裁定の組み合わせ → 直交宣言（理由つき）or 交点列挙 or `ground:` 送り |
| 4 | only 制約違反 | 到達不能な Given → B 行の Given に前提軸の値を明記 |
| 5 | 連鎖カバレッジ | 宣言した続落に B 行がない → パラメタライズで追加 |
| 6 | 観測・修飾カバレッジ | 検証する B 行がない → 追加 |
| 7 | per-item 混在 | 混在が未裁定 → 混在 Covers（`+`）or 自己ペア直交宣言 or `ground:` 自己ペア |

指摘とは別枠で「ground 送り項目」（`ground:` 宣言 + base の `?`）を出力する。
定義エラー（未知の軸/値参照・構文エラー・パースできない B 見出し）は黙殺せず exit 2 で止まる。

状態マップ（`docs/appendix-state-map.md`）を使うプロジェクトでは、`ref:` 付き軸に
対して検査 M3（ref 整合）が追加で走る。**状態マップはオプショナル** — `ref:` を
使わない spec では従来動作と完全に同一。

## spec ツリーレンダラー — 全 B 行の分岐図

fix 前の全体確認・gate 前の全体把握用に、全 B 行を「基準シナリオの分岐の先の想定挙動」
として 1 枚の Mermaid フローチャートに導出する:

```bash
node <dandori-repo>/skills/dandori-spec/scripts/render-spec-tree.ts <spec.md> > <出力.md>
```

- 背骨 = `flow:`（無ければ軸の宣言順で退化描画）、菱形 = 分岐の問い、紫枠の葉 = B 行、
  赤破線 ❓ = 穴（未カバー値・未カバー依存交点・ground 送り）、太矢印 = 連鎖
- **人間の承認用ビュー** — ユーザーは葉を全部なぞって「この分岐の先の想定挙動はこれでよいか」を
  確認する。品質ゲートの一覧を縦に読むより全体を把握しやすい（2026-07-08 の探索で採用）
- 出力は生成物であり手編集しない。B 行の増減後は再生成する。検査の正はチェッカー —
  レンダラーの ❓ は簡易導出（検査 2 相当のみ）で、検査 3〜7 は表示しない
- 描画確認は Mermaid 対応プレビュー（Zed / VS Code / GitHub）

## ゲート運用

- **spec fix の前提条件 = チェッカーグリーン（exit 0）**（裁定 2026-07-06）。
  裁定不能なものは `ground:`（ペア・per-item 混在）と `?`（base）で ground 送りを
  **機械可読に宣言**すれば exit 0 で通る。チェッカーが強制するのは**裁定の存在**
  （裁定するか、先送りを明示するか）であり、正しさは壁打ちと ground で担保する
- **機械的直交裁定の圧力に注意**（実験 4 摩擦 5）: 検査 3 を黙らせるためだけの
  吟味なし直交宣言は「裁定のふり」— チェッカーの意図の逆転。防波堤:
  (a) 直交の reason には spec 本文から言える根拠を書く（書けないなら `ground:` 送り）、
  (b) 同種の軸はグループ宣言でまとめ、裁定の痕跡を 1 箇所に残す、
  (c) 検査 3 の指摘はそのまま壁打ちの議題リストとしてユーザーに出す
  （実測: 30 件中 27 件は spec 本文から即裁定、残 3 件が ground 送り）、
  (d) review 工程のレビューアに直交宣言の監査（reason が spec 本文から導けるか +
  反例交点の探索）を指示する — (a)〜(c) は執筆側の自己規律であり、
  裁定のふりの第三者検査点は review が唯一（dandori-review の指示テンプレート参照）
- ground 工程は「ground 送り項目」を確認項目として消化し、確定結果をモデルに
  書き戻す（`ground:` エントリを `orthogonal`/`dependent` に昇格、`?` を外す）
- gate 工程では B 行の増減後にチェッカーを再実行する（末尾追加・改番禁止の
  既存ルールとそのまま整合）。**gate 時点で ground 送り項目が残っていたら要裁定** —
  未消化の確認項目はトレース表の ⚠️ と同格に扱う

## 検出できないもの — 過信防止

このモデルは**レビューループの代替ではない**。以下の型は原理的に守備範囲外で、
従来どおり ground / review / formal レーンで扱う:

1. **軸内の逐次連鎖の発見** — 宣言されていない chains は要求できない
2. **分類述語の値域の穴**（「どの文言にも一致しない 422」型）— formal レーン
   （全列挙 / Z3、appendix-formal「被覆の穴」）の守備範囲
3. **未発見の状態変数**（optout 型）— 変数の発見は ground / review の仕事
4. **値内部のセマンティクス**（照合条件の非対称、キー付け替え等）— コード読解でしか出ない
5. **エラー優先順位の導出**（2 条件同時はどちらが勝つか）— 異常系の形式化を
   横断関心事に限定した裁定の帰結。散文の評価順序宣言とレビューで扱う
