# アイディア: spec レンズレビュー — 生成 spec をレビュー観点として使う軽量レビューフロー

status: idea（未実施 — 実プロジェクトでの手動1周による較正待ち）

## 一言で

要件から spec.md（B 行）を **blind 生成**し、それをレビュー観点（照合チェックリスト）として
既存コード / PR をレビューする。トレース表・ゲートは作らない。
目的は全数認証ではなく**実装の穴の発見** — dandori の言葉でいうと、
gate（全数証明）を捨てて finder レーン（recall 優先）だけを残す構成。

## なぜ成立するか

- dandori-spec の価値の中核は §3 状態変数マトリクスによる**エラーパスの体系的列挙**。
  トレース表はそれを認証に格上げする装置にすぎず、穴の発見が目的なら認証は不要
- spec をレビュアに渡すと、問いが「diff を読んでバグを探せ」（開放形）から
  「B-n の Given/When に対して Then を満たすコードがあるか」（閉じた照合問題の列）に変換される。
  codereview の「照合対象を全数列挙し1件ずつ walk」ルールの列挙リストとして B 行が機能し、
  レーン内部で疑似トレースが起きる — 失うのは永続記録とサイレント通過の禁止だけ

## フロー（5手）

1. **spec 生成（blind）** — 要件ドキュメントだけから B 行を生成。コードは見ない
2. **タイトル一覧スキャン** — B 行タイトルだけユーザーが眺め、「要件にない発明」を線引き
   （フル壁打ちはしない）
3. **レーン並列** — spec レンズレーン（B 行をチェックリストに照合）+
   blind レーン（spec を渡さない素のバグ探し — 「仕様どおりだから OK」バイアス対策と
   spec の射程外の穴の保険）
4. **反証フェーズ** — codereview と同じ verifier。生成 spec はユーザー fix を経ないため
   「要件にない挙動を発明した B 行」由来の偽指摘率が本流より高く、精度ゲートは削れない
5. **指摘提示** — 生き残った指摘をユーザー裁定へ

## 残すもの / 削るもの

| 判断 | 対象 | 理由 |
|------|------|------|
| 残す | spec 生成の blind 規律 | 緩めると spec がコードの言い換えになり照合が恒真化。価値の全部がここ |
| 残す | §3 状態変数マトリクス | レビュアの直感が拾わないエラーパス欠落の検出源。短縮するならここ以外を削る |
| 残す | 反証フェーズ | 発明 B 行由来の偽指摘を落とす唯一の精度ゲート。verifier は安い |
| 削る | ゲートタグ | 検証コスト申告はレビューでは無意味 |
| 削る | 状態モデルチェッカー | fix しない spec に機械的厳密性は過剰 |
| 削る | 壁打ちの往復 | タイトル一覧スキャン（30秒）で代替 |
| 削る | トレース表・修正ループ | 出力は指摘の裁定リストで止まる。直す/要件側を直す/受容はユーザー裁定 |

## 発展形（重量版 — 別アイディア）

トレース工程を足すと spec-drift 検出（認証）になる:
B 行ごとに証拠（既存テスト・コード読解・実行結果）を探索し
`✅一致 / ❌乖離 / ⚠️未実装 / ❓検証不能` のトレース表を作る。
順方向トレースは「コードがやっているのに要件にない挙動」を構造的に見逃すため、
逆方向レーン（spec を渡さずコードから観測挙動を書き起こして突合 —
EC 適用時代の BT レビューの一般化）を足して両側を閉じる。
軽量版で較正してから検討する。

## 較正してから固めること

- **発明 B 行率**（= 反証で落ちる指摘の率）を実 PR 1本の手動1周で計測する。
  高すぎるならスキャン工程を厚くする / 出自マーク（要件由来 / 推定）を B 行に付けて
  推定行の指摘は「質問」としてユーザーに出す
- spec レンズレーンの分割単位（正常系/異常系の軸ごと? — codereview のレーン内二次分割と同じ方針）
- スキル名は dandori-specreview（仮）。spec 生成部は dandori-spec の §1〜3 を参照で流用
