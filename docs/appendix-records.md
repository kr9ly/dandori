# 付録: コミット記録運用（クローズの正準定義 / .dandori/records.md）

dandori の**既定のクローズ運用**。`.dandori/specs/<feature>/` のフィーチャードキュメントは
**実装が終わったらすべて消える** — spec.md も例外ではない。残った文書はコードとの乖離で
いずれ嘘をつき、混乱のもとになる。正はこの3層だけ:

1. **ソースコード + テスト** — いま真であることのすべて（振る舞いはテストに、
   コード行に anchor がある Why は annotate 工程のコメントに転写済み）
2. **コミット履歴** — 過去の判断。クローズ時の**墓碑コミット**に裁定情報を蒸留する
3. **ADR 等の長寿命ドキュメント**（プロジェクトが持っていれば）— アーキテクチャレベルの
   判断。所在は `.dandori/resources.md` で宣言する

コミットメッセージがアーカイブ層に向く理由: **drift しない**（スナップショットに完全に
紐づき「この時点ではこうだった」以上を主張しない）、**メンテナンスコストゼロ**（verify も
刈り込みも不要）、**機械検索できる**（固定キー + `git log --grep`）。

ただしコミットメッセージは **pull 型**（疑問を持った読者が探しに来る）の保存先であり、
**push 型が必要な情報 — 未解決リスク・分離タスク — は載せてはいけない**(誰も開けない
引き出しに地雷マップをしまうのと同じ)。push 型はタスク置き場（下記の必須宣言）へ送る。
この境界が本運用の唯一の禁則。

## .dandori/records.md — プロジェクト宣言ファイル

運用に必要な宣言を持つファイル。**初回のクローズ（annotate 着手）までに整備する** —
無ければ annotate / cleanup 工程がユーザーと壁打ちして作成する。

```markdown
# コミット記録運用

- 方式: tombstone
- タスク置き場: GitHub Issues の label:from-dandori 【必須 — 未解決リスク・分離タスクの push 先】
- マージ方式: merge(履歴保存)【squash 運用なら墓碑メモを PR 本文 / merge コミットに複製すること】
- ADR: docs/adr/ 【任意 — アーキテクチャレベルの裁定の行き先】
- 追加キー: 旧実装非互換:（移行元システムとの裁定済み乖離）
- 検索: git log --grep '^Decision:' -- 全裁定 / git log --grep '^旧実装非互換:' -- 非互換一覧
```

- **方式**: `tombstone`（既定 — spec.md 含め全処分 + 墓碑コミット）/ `retain`
  （spec.md を長寿命ドキュメントとして残す例外運用 — 理由を併記する。
  この場合も design.md 等の使い捨て文書の処分と annotate は通常どおり）
- **タスク置き場**（tombstone では必須）: push 型情報の行き先。ゴールグラフ・
  issue tracker・プロジェクトの backlog 等、「次の作業を選ぶときに必ず見る場所」であること。
  **宣言できないプロジェクトは tombstone を選べない**（push 型の行き先がないまま
  spec を消すと未解決リスクが消滅する）— retain に倒す
- **マージ方式**（必須）: 墓碑コミットが履歴に残ることの確認。squash 運用の
  プロジェクトでは墓碑メッセージが潰れるため、PR 本文 / merge コミットへの複製を宣言する
- **追加キー**（任意）: 下記の共通キーに加えるプロジェクト固有の固定キー

## 墓碑コミット形式

spec.md を含むフィーチャードキュメント一式を**削除する専用コミット**。strip の
B-ID 除去コミットとは分ける — 削除コミットを `git log --diff-filter=D -- <spec.md のパス>`
で特定すれば、**蒸留メモ（メッセージ）と全文書の最終版（親コミット）に同時に届く**。

```
chore(cleanup): <feature> クローズ

Decision: FK 未解決トークンは一律「認証エラー」に正規化 — 旧実装は 500/空応答に非対称
  （病的データ。probe 実測は親コミット probes/xxx_probe.py）
Decision: 契約突合は現行方式の拡張で実現 — 生成スキーマ比較への作り替えは規模大で別タスク化
Dropped: B-19 スタブEP認証テスト — 登録済みスタブ 0 本と機械判定、対象消滅
Test-Strategy: 87本×権限の全組み合わせはせず代表EP + 機械検査で全数担保

Spec-Tombstone: .dandori/specs/<feature>/
```

### 共通固定キー

| キー | 中身 | 主な源 |
|------|------|--------|
| `Decision:` | 確定した裁定（採った方式・棄てた選択肢・理由） | spec スコープ外の裁定注記、台帳の却下行 |
| `Dropped:` | 消したもの（取り消し線 B 行・軸）とその理由 | spec の取り消し線、feedback の却下 |
| `Test-Strategy:` | テストしない範囲とその担保方法 | 状態モデルの直交裁定、テスト設計の骨格 |
| `Spec-Tombstone:` | 処分した spec ディレクトリのパス（trailer） | — |

キーは**行頭固定**（`git log --grep '^Decision:'` を成立させる）。1 記録 1 キー行 +
続き行はインデント。証拠（probe・実測）は「親コミットの <パス>」で参照する — 墓碑の
親には全文書・probe が揃っている。

### 書く価値の判定

墓碑メモは全文書の要約ではない。載せるのは「**将来の読者・エージェントが同じ論点を
再演しうる裁定**」だけ — Given/When/Then の本体はテストスイートに転写済みであり
重複させない。コード行に anchor がある Why は annotate 工程がコードコメントとして
残しており、墓碑はフィーチャー単位・裁定単位の疑問に答える層。アーキテクチャレベルの
裁定（複数フィーチャーを拘束する）はコミットより ADR が適切 — records.md の ADR 宣言先へ。

## 復元手順（done からの再開）

改訂は「fix 済み spec への差分」として入るため、クローズ済みフィーチャーに結論が
持ち込まれたら dandori-feedback が**まず spec.md を復元する**:

```bash
git log --diff-filter=D -1 --format=%H -- .dandori/specs/<feature>/spec.md  # 墓碑コミット特定
git show <墓碑sha>^:.dandori/specs/<feature>/spec.md > .dandori/specs/<feature>/spec.md
```

復元した spec は作業コピー — 改訂サイクルが終わり再び cleanup に至ったら、
新しい墓碑コミット（前回墓碑以降の Decision を追記）とともに再処分する。

## 検索レシピ

- 過去裁定の横断検索（spec 壁打ちの前提集め）: `git log --grep '^Decision:.*<キーワード>' --format='%h %s'`
- 特定フィーチャーの墓碑: `git log --diff-filter=D -1 -- .dandori/specs/<feature>/spec.md`
- 消したテスト・B 行の理由: `git log --grep '^Dropped:'`
- カバレッジ戦略の確認: `git log --grep '^Test-Strategy:' -- <テストディレクトリ>`

## 工程への影響（サマリ）

| 工程 | 挙動 |
|------|------|
| spec | §1 で過去裁定を `git log --grep` で検索し壁打ちの前提に含める（records.md の追加キーも使う） |
| annotate | 候補源に spec.md を含む全処分予定文書。処置語彙に「コミット記録へ」「タスク置き場へ」。records.md 未整備ならここで作成を壁打ち |
| strip | B-ID 除去の専用コミット（墓碑とは別 — 親側に gate コミットのトレース可能なスナップショットを残す） |
| feedback | done からの再開時、spec.md を墓碑コミットから復元してから通常手順。取り込みゼロ経路の却下・保留は「## Why 保全」節へ自分で追記（annotate は再訪しない） |
| cleanup | 墓碑コミット（spec.md 含む全処分 + 蒸留メモ）でクローズ。push 型項目のタスク置き場転送を確認してから処分。retain 宣言のプロジェクトのみ spec.md を残す |
