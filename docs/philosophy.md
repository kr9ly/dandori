# dandori の理論的背景

dandori は経験知から作られたプロセスであり、理論から演繹されたものではない。
この文書は、後から確認された理論的な系譜と根拠を整理したものである。
プロセスの正当性を説明する必要が生じたときの参照用。

理論が後付けであることは弱みではない。理論から演繹したプロセスは現場で死ぬが、
現場で生き残ったものに理論を張る作業は、検証済みの主張しか生まない。

## 1. 系譜 — 何が先行し、どこからが固有か

「検証基準を仕様と同時に定義する」という発想には確立した先行がある:

- **ATDD（受け入れテスト駆動開発）/ Specification by Example**（Gojko Adzic）—
  「仕様を書くとは検証可能な例を書くことだ」という主張そのもの
- **V字モデル / W字モデル** — 要件定義の各段階で対応するテストを設計する構図
- **Kiro / spec-kit / BMAD** — スペック駆動をエージェント向けの具体的なコマンド群に
  落とした先行実装。Kiro は EARS 記法の受け入れ基準を要件定義時に書かせる

したがって「品質基準を早期に定義する」ことも「それを具体化する」ことも新規ではない。
**先行はすべて前半（早く定義する）で止まっている**、というのが正確な差分である。
定義された基準が実装完了まで生き残り、行単位で照合されることを強制する仕組み —
ゲートタグ → B行トレース → サイレント通過の禁止、および「実行検証済 / 読解のみ」の
区別 — を持つものは（2026年時点の調査では）見当たらない。

> 反論への備え: 「Kiro にもあるのでは」と言われたら、受け入れ基準の**生成**と、
> 実装完了時にその基準表と突き合わせて**自己通過を禁止する工程**の有無を確認させる。
> 前者はある。後者はない。

一言で言えば: **V字モデルの右側（検証）を先に設計し、エージェントにそれを強制する**。
仕様書として書いているものは、実際にはテスト仕様書である。spec.md がテスト仕様書と
ほぼ一対一に写像できるのは偶然ではなく、この転倒の構造的な帰結である。

## 2. 貫通原理 — 主張の検証状態の会計

§1 の差分（定義ではなく照合の強制）を一般化すると、dandori の全工程はひとつの
原理の現れである: **エージェントの自己申告を信用しない**。すべての主張に「検証状態」
という属性を持たせ、未検証のまま通過することをプロセスが禁止する。

- 土台調査の主張 → 「実行検証済 / 読解のみ」の等級タグ（ground / survey）
- 仕様の充足 → B行トレース表とサイレント通過の禁止（gate）
- レビュー指摘 → 反証フェーズ。指摘もまた未検証の主張である（review / codereview）
- ドキュメントの記述 → 証拠アンカーと verify による腐敗検出（survey）
- 検証の副産物 → 使い捨て隔離。検証されていないコードを主張の地位から外す（spike）

この見方は競合比較を機能列挙の水掛け論から救い出す。Kiro / spec-kit は仕様を
**生成する**システムであり、dandori は主張の検証状態を**追跡する**会計システムである。
仕訳（基準の定義）はどのツールでもできる。監査（照合の強制）を持つかが差分であり、
§1 の「先行は前半で止まっている」はこの言い換えである。

なお「人間裁定待ち」も検証状態のひとつである。§9 の弁別装置は、この会計の台帳に
人間の署名が記帳されることの帰結にすぎない。

## 3. 経済の逆転 — なぜ廃れたプラクティスが今成立するのか

Specification by Example が人間のチームで定着しなかった主因は思想の誤りではなく
コストである。仕様策定の段階で検証方法まで書き切ると仕様会議が倍になり、実装前に
例が陳腐化した。規律の維持コストも高い — 締切前に「今回はいいか」で照合を飛ばす
誘惑に、人間のチームは負け続けた。

エージェントは二つのコストを同時に変えた:

1. **定義コストの暴落** — Given/When/Then + ゲートタグを書くコストがほぼゼロになった
2. **規律の機械化** — サイレント通過の禁止はプロセス側が強制する。人間に要求される
   のは規律ではなく判断に純化される

「段取り八分・仕事二分」は職人の世界では重要度の比喩だった（実際の工数は石を切る
作業が大半を占めた）。エージェントコーディングはこれを文字通りの工数配分にした。
実装が吸収された結果、人間の時間の8割を段取りに使うことが実測になる。

## 4. 動くコードの証拠価値の暴落（攻めの根拠）

人間が書いていた時代、「動くコード」には証拠価値があった。動くところまで持って
いくのが大変だから、動いている事実がある程度の理解と品質を担保した。

エージェントは動く**ように見える**コードを無限に安く生産する。この証拠価値は
暴落した。「動いた気がするが仕様を満たしていない」という失敗モードは、この暴落の
直接の帰結である。勝負は動くコードができた時点で決まっている — 段取りが良ければ
勝ちが確定しており、悪ければ**負けているのに動くコードがそれを隠す**。

真実の確定は、もはや仕事二分の側では行えない。全部段取り側でやるしかない。

## 5. 誤りの汚染モデル（守りの根拠）

人間が間違ったコードを書いたとき、意図の記憶は本人の頭に残る。「書いたもの」と
「書きたかったもの」を別々に保持しているので、間違いは差分として認識でき、修復は
局所で済む。

エージェントには意図の独立な保存先がない。後続のエージェントにとってコードベース
そのものが意図の一次証拠であり、間違ったコードは単なるバグではなく**偽の前提**として
機能する。後続はそれを読んで意図を逆算し、パターンを模倣して横展開し、その上に積む。
人間の間違いは減価するが、エージェント環境の間違いは複利で増殖する。

歴史的な含意: アジャイルが BDUF（事前の重厚な設計）を否定できた根拠は、変更コスト
曲線（Boehm）が寝たことだった — 修正が安いなら、早く作って直す方が得。エージェントは
編集コスト自体はさらに下げたが、**認識汚染のコストで変更コスト曲線をもう一度立てた**。
スペック駆動の再流行は懐古趣味ではなく、コスト曲線の再逆転への合理的な適応である。

dandori-spike が「使い捨てコード」を強調するのはこのモデルの帰結。検証の副産物が
偽の前提として後続に参照されるのを防ぐ隔離措置 — 行儀作法ではなく防疫である。

## 6. correct by construction — 思想的源流

「正しく動くべくコードを書いて、その通り動くように作れ」という原則には名前がある。
**correctness by construction**（Dijkstra、後に英 Praxis 社の方法論として確立。
Martyn Thomas, Anthony Hall, Roderick Chapman）。対義語は
**construct by correction** — 直しながら動くところに漂着させる作り方。

- correction（正しい状態に向かうこと）は誤りの存在を前提しないと成立しない概念で
  ある。construct by correction は名前の中に「誤りが原材料である」という世界観を
  組み込んでいる
- 向かうことは到達を証明しない。バグを100個直したコードが101個目を持たない根拠は
  何もない。「テストはバグの存在を示せるが、不在は示せない」（Dijkstra）は、
  correction と correct の間の越えられない溝を言った言葉である
- construct by correction で出来たコードには**なぜ正しいのかの導出が残らない**。
  導出のないコードは、コードから意図を逆算するエージェントにとって解釈不能な遺物になる

この教義は40年間、経済が合わずに辺境（航空宇宙・鉄道・暗号）にいた。導出を書く
コストを人間が払えなかったからである。エージェントは導出を書くコストを暴落させ、
同時に導出のないコードのコストを暴騰させた。両側から挟まれて、40年遅れで
Dijkstra の経済が成立した。

correction パラダイムが社会的に勝った理由も押さえておく: **動きは観測可能だが、
状態は観測不能**。直したバグの数、バーンダウンチャートは進捗の動画的証拠を生むが、
correct by construction は完成まで静止して見える。「段取り八分」がサボりと誤解
される危険はこの構造から来る。B行トレースは、状態の言語（Given/When/Then は差分では
なく状態の述語である）に進捗の可視性を与える装置であり、correction 側の唯一の実務的
優位を状態側の土俵で再現したものである。

## 7. 暗黙知の外在化 — ground と目視ゲートの独自性の理由

dandori が独自に言語化している工程は、人間のプラクティスでは**暗黙にやっていたから
名前がつかなかった**部分である:

- **ground** — 経験ある開発者は設計の前に必ずコードを触って前提を接地する（動かす、
  grep する、print を仕込む）。暗黙知だったので人間向けプロセスに工程として存在しない。
  エージェントは自分の読解への確信度を較正できない（読んだだけで分かった気になる）
  ので、外在化して「実行検証済 / 読解のみ」のラベルを強制するしかない
- **目視ゲート** — TDD はオラクル（正否の判定者）を自動テストに限定し、自動化でき
  ない検証（佇まい、文言、ログの読みやすさ）をプロセス外に落とした。dandori は
  オラクルを一般化し、B行ごとに検証手段（自動テスト / 人間の目視）を宣言させ、
  目視の行も同じトレース表で照合する。これは TDD より古い実務 — 手動確認項目と
  確認欄を持つ古典的なテスト仕様書 — の復活だが、エージェント文脈では必然性がある:
  エージェントの失敗モードは「人間に見せるべき箇所を自分で通過させる」ことなので、
  人間の担当行を事前に割り当て、その行の自己通過を禁じることが自律性への制御弁になる

一般原理: 人間の開発プロセスは、人間の暗黙知に依存していた部分を書いていない。
エージェントには暗黙知がないので、そこを全部明示的な工程にする必要がある。
既存のスペック駆動ツールに ground 相当がないのは、人間の SDLC の**書かれていた
部分**だけを移植したからである。

## 8. 散文の限界と漸進的形式化 — 機械検査層の根拠

この節の主張だけは演繹ではなく測定から立っている。23 ラウンドの独立レビューを通過した
spec（cart-payment、830 行）に対して、状態変数の直積を走査する機械検査が実質的な
新規指摘を 6 件出した — 宣言済み交点の未カバー×2、制約矛盾×1、ペア全数性×2、
軸混在の未検討×1（`experiments/state-model/findings.md`）。

散文レビューには構造的な上限がある。レビューアを独立させ、ラウンドを重ねても、
直積は走査できない。実際に見つかったのは「クーポン要素を欠いた Given が ERP 連携
『なし』の主張を空虚に真にしている」「到達不能な Given」といった、文章として読む限り
正しく見える種類の欠陥である。この上限の突破が機械検査層（状態モデル・状態マップ）
だが、導入形態に二つの裁定が効いている:

- **外部の形式言語を導入しない**。PICT / TSL 流の別ファイル形式化ではなく、spec.md 内の
  宣言ブロック。「spec.md が単一の正」という思想を守り、散文の実例から段階的に形式化
  できる。§6 の経済がここでも働く — 導出（軸と直交性の宣言）を書くコストをエージェント
  が暴落させたから、この層を維持できる
- **チェッカーが強制するのは正しさではなく裁定の存在**。未分類ペア（何も分からない
  状態）を「依存を宣言済みだが未カバー」という実行可能なタスクに変換し、「分からない」
  にも記帳可能な勘定科目（ground 送り）を与える。§2 の会計原理の機械実装である

状態マップは競合差分の第二段でもある。既存のスペック駆動ツールはフィーチャー単位で
閉じており、機能境界をまたぐ共有状態への影響 — SSOT の多重定義、外部専有状態への
書き込み、変更の回帰範囲 — は全ツールの盲点である。系譜は CRUD マトリクス、
frame condition（JML `assignable` / Dafny `modifies`）、separation logic の frame rule。
過大近似で運用する（影響しうる範囲を漏らさず、偽陽性は許容する）という健全性の選択が、
台帳を数十行に保ち、機械検査を実装可能にしている。

## 9. 人間に残る仕事 — 判断・引き受け・専門性

dandori がユーザーとの接点として残す3箇所（仕様の壁打ち、レビュー結果の裁定、
最終ゲートの裁定）は、いずれも古典的なテストエンジニアリングの中核能力に対応する:
同値分割・境界値・状態遷移の勘、深刻度トリアージ、残存リスクの受容判断。

エージェントは構造的に「どちらの判断をしても責任を問われない」存在であり、判断の
演算はできても引き受けができない。そして事後に「責任を取る」の中身を分解すると、
修復・補償・辞任謝罪の三種類しかなく、ソフトウェアで価値があるのは修復だけ —
つまり能力の行使そのものである。**判断ができる ＝ 引き受けができる ＝ 専門性がある**
は同じものの三つの言い方であり、専門性の裏付けのない「責任を取ります」は不渡り
確定の手形である。

したがって dandori は「回せる人を増やす民主化装置」ではない。判断が行単位で記録され
結果と突き合わされるため、裏付けのない署名は数フィーチャーで露呈し、裏付けのある
署名は台帳で証明されていく — **署名に裏付けがあるかを可視化する弁別装置**である。
エージェント時代に人間の寄与は引き受けの一点に濃縮され、残った仕事の専門性密度は
上がる。生き残るのは、その署名を裏付けられる専門性 — テスト工学 — を持つ人間である。

## 10. 出自

dandori は、QA チーム不在の少人数モバイル開発チームが週次リリースで
クラッシュフリーレート 99.9% を維持するために蒸留した実践知のダンプである。
検証機能の引き受け先が組織に存在しない環境では、開発者が自分の中にオラクルを
内蔵するしかない — そしてエージェントコーディングとは、全員がその環境に放り込まれる
ことを意味する（エージェントに QA チームはついてこない）。

効かない儀式を維持する余裕のない制約環境は、実効性のあるプラクティスだけを残す
蒸留器として働く。毎日ビルドして触る（人間の知覚という最強のオラクルを最短の
フィードバックループで回す）、1px のズレで警報を鳴らす（チームの異常検出閾値の較正）、
読んだだけの前提を信用しない — dandori に入っているのは、この蒸留を通ったものだけである。

名前の出自も同型である。「段取り八分・仕事二分」は特定の提唱者を持たない現場伝承の
ことわざ — 無名の職人たちの暗黙知が蒸留されて生き残った言葉であり、語源の一説は
石工が石段を作る前に坂全体を見て段数を確定させる「段を取る」から来る。
石を切り始める前に段数を確定させ、出来上がった石段で段取りの良し悪しが事後評価される。
このプロセスがやっていることと同じである。
