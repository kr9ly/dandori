# dandori — 段取り八分・仕事二分

エージェントコーディングで高品質な実装を実現するための、仕様駆動プロセスを提供する
Claude Code スキル群。実装前の仕様策定・前提検証に工数の8割を投じることで、
「動いた気がするが仕様を満たしていない」実装を構造的に防ぐ。

全工程を貫く原理はひとつ — **エージェントの「できました」を信用しない**。
仕様の充足、コード読解の前提、レビュー指摘 — すべての主張に検証状態
（実行検証済 / 読解のみ / 人間裁定待ち）を持たせ、未検証のまま通過することを
プロセス側が禁止する。この原理は次の仕組みで担保される:

- 土台調査の全主張に「実行検証済 / 読解のみ」の等級を付ける。読んだだけの前提は、
  検証スパイクで動かして確かめるまで格上げされない
- 最終ゲートで仕様の全行を検証状況つきで照合する。「たぶん通っている」行は
  サイレントに通過できず、未検証として列挙される
- 自動テストで判定できない行（佇まい・文言・ログの読みやすさ）は人間の目視行として
  仕様時点で宣言され、エージェントの自己通過が禁止される
- レビュー指摘そのものも検証対象。反証を生き残った指摘だけが還流し、
  もっともらしいだけの指摘を遮断する
- 検証スパイクの産物は本体に混ぜない。未検証コードが後続エージェントの
  「偽の前提」として参照されることを防ぐ

## 設計思想

エージェント実装の品質は仕様の品質で上限が決まる。dandori は以下の失敗モードへの
対策をプロセスに組み込んでいる:

| 失敗モード | 対策 |
|-----------|------|
| ゴールが検証不能な美辞麗句になる | 品質ゲートを要件定義と**同時に**定義（Given/When/Then + ゲートタグ） |
| エラーパスの洗い出しをサボる | 状態変数×正常系シナリオのマトリクス走査を独立工程に |
| コードを読んだだけの前提が外れる | 「実行検証済 / 読解のみ」の区別を強制 + 検証スパイク |
| ドキュメントレビューが机上で完結する | レビューアに「主張をコードで検証せよ」を指示 |
| レビューループが収束しない | 深刻度3分類 + blocker/major ゼロで通過 + 収束兆候の消失で escalate |
| 実装エージェントが制約を読み飛ばす | マイルストーンごとの部分開示（コンテキストマニフェスト） |
| 実装中の発見がドキュメントに還流されない | design.md の発見ログ + worker への報告義務 |
| テストが通れば正しいと錯覚する | テスト緑後のセルフコードレビュー（テスト忠実度・不変条件・blind・周辺整合の4レーン） |
| 名前だけ仕様に対応した弱いテストが緑で通る | diff スコープのミューテーションテスト（生存ミュータント = 実行証拠）+ テスト忠実度レーン |
| レビュアの過剰検出でノイズが溢れる | 指摘ごとの反証フェーズ。反証を生き残った指摘だけを還流 |
| 却下した指摘が会話にしか残らず、収束判定（再燃・振動）が記憶頼みになる | 指摘台帳（review-ledger.md）。却下・反証破棄も永続化し、台帳照合で機械判定 |
| チェッカーを黙らせるための吟味なし直交宣言（裁定のふり） | review 工程で直交宣言を監査（reason の根拠検証 + 反例交点の探索） |
| 最終確認が「全部やった気がする」で終わる | 全仕様行の検証状況トレース表。サイレント通過の禁止 |
| 状態変数の交点抜けが散文レビューで拾いきれない | 状態モデルの直積走査による機械検査（任意） |
| 機能横断の共有状態への影響が漏れる | 状態マップによる影響 spec の機械導出と SSOT 検査（任意） |
| 実装エージェントがマニフェスト外の既存資産を知らずに再発明する | リファインメント工程の再発明レーン（コードベース全体との照合） |
| 書き味の指摘が正しさレビューに混ざる / 好みの議論が発散する | リファインメント工程の分離。接地必須 + 1ラウンド限り、検証は反証でなくゲート再実行 |
| デザイン変更で指示外の画面領域が巻き添えで変わる /「デザイン通りにした」の自己申告 | スケッチ工程で全領域を変更/不可侵に全数分類 + 不可侵領域の非変化を gate でトレース |
| デザイン指示の空白（エラー時・空データ時の見た目）をエージェントが勝手に補う | 指示の読み取りに「明記 / 推測」マークを強制。推測は全件ユーザー裁定 |
| プロセスの追跡装置（テスト名の B-ID・spec 参照コメント）がプロダクトコードに残り、プロセスを知らない読者に伝わらない | クリーンアップ工程で言及を除去（residue 機械検査 + `dandori-ok:` 裁定マーカー）。gate コミットがトレース可能なスナップショットとして履歴に残る |
| 設計判断の Why / Why not（採らなかった選択肢・意図的にやらないこと）がドキュメント処分と同時に消え、後続の読者が意図的なコードを「直して」しまう | コメント保全工程（annotate）— 処分の前に消える判断を全数走査し、読者が出会う場所（コードコメント / 墓碑コミット / タスク置き場 / ADR）へ振り分けてから cleanup へ進む |
| 実装後も残した仕様書がコードとの乖離でいずれ嘘をつき、混乱のもとになる | フィーチャードキュメントは実装が終わったらすべて処分。正はコード + テスト、過去の判断は墓碑コミット（固定キーで機械検索可能）と ADR |
| テストの軸がコード上で散在し、交点特例の if が乗算で増殖する | ground の軸対応 — 状態モデルの全軸をコード構造（型/leaf/分岐点）に対応付け、機械検査（D5）。dependent 宣言だけが散在の免罪符 |
| テストの Given が行データの手組みに埋まり、どの座標を検証しているのか読めない | Given 空間ビルダー + fixture 語彙 — spec の Covers デルタとビルダー引数を同型にする（`base + coupon=cp_ok` ↔ `{ coupon: 'valid' }`） |
| 型に値が増えても仕様の同値分割が較正されず、どのクラスでもテストされない値が黙って生まれる | partition_of 宣言（軸の値域 = 型の分割）+ `Record<型, 軸値>` のコンパイル時強制 + ground の全域性突合 |
| 単体で固定したい振る舞いが e2e テストに流れ、分解されないまま太る | Gate 乖離マーク（`e2e→unit（阻害要因）`）— チェッカーが別枠列挙し、ground の seam 検討議題になる |

これらを貫く終端不変条件がひとつある: **痕跡を残さず、判断だけを残す**。
完成後のリポジトリは、dandori を知らない読者にとって理想的な開発者の仕事と区別できて
はならない。成果物の恒久保存先はコードとコミット記録だけであり、プロセスの語彙は
成果物に残らない。プロセス痕跡は書いた瞬間から腐り始めること、そしてプロセス自体の
陳腐化が速い時代には**捨てられないプロセスは採用する価値がない**ことがその根拠である
（詳細は [理論的背景 §10](docs/philosophy.md)）。

## プロセス

```
dandori（入口・トリアージ）
  │
  ├─ フルコース
  │   dandori-spec ──→ (dandori-sketch) ──→ dandori-ground ──→ dandori-review ──→ dandori-spike
  │   （仕様壁打ち）   （デザイン接地・UI時）   （前提の接地）      （独立レビュー）    （前提の実地検証）
  │        │                                                    │
  │        │   dandori-feedback ←── dandori-gate ←── dandori-refine ←── dandori-codereview ←── dandori-impl ←── dandori-plan
  │        │   （継続改善の裁定点）  （最終ゲート） （リファインメント） （セルフコードレビュー）  （MS単位実装）    （計画）
  │        │
  └─ 短縮コース: dandori-spec（短縮） ──→ dandori-impl ──→ dandori-gate ──→ dandori-feedback

  継続改善: dandori-feedback で裁定 —
            結論の取込: （spec | ground | impl に巻き戻し）──→ 通常フローに再合流 ──→ gate ──→ feedback
            結論待ち:   phase: feedback のまま待機（安定点）
            完全に fix: dandori-annotate（消える Why のコメント保全）──→ dandori-cleanup（B-ID 除去 = 不可逆な店じまい）──→ done
```

ユーザーとの接点は3つに集約される: 仕様の壁打ち、レビュー結果の裁定、最終ゲートの裁定。
それ以外の工程は自律で進む。

## 状態管理

進捗はすべてフィーチャーディレクトリのドキュメントに永続化され、セッションをまたいで
再開できる。`dandori` と一言打てば現在地から再開する。

```
<project>/.dandori/specs/<feature>/
├── state.yaml      # 現在フェーズ・進捗
├── spec.md         # 要件 + 振る舞い仕様 + 品質ゲート【fix 後の正 — クローズ時に墓碑コミットへ蒸留して処分】
├── sketch.md       # デザイン指示の読み取り + 変更/不可侵の領域マップ【UIタスクのみ・使い捨て】
├── design.md       # 前提・改変箇所・不変条件 + 発見ログ【実装完了まで】
└── plan.md         # マイルストーン + コンテキストマニフェスト【使い捨て】
```

フィーチャードキュメントは**実装が終わったらすべて消える**
（[docs/appendix-records.md](docs/appendix-records.md)）。残した文書はコードとの乖離で
いずれ嘘をつくため、正はソースコード + テストに一本化し、過去の判断はクローズ時の
**墓碑コミット**（`Decision:` / `Dropped:` / `Test-Strategy:` の固定キー — `git log --grep`
で機械検索でき、drift せずメンテナンスコストゼロ）と ADR に蒸留する。未解決リスク・
分離タスクなど push 型の情報だけはコミットに載せず、`.dandori/records.md` で宣言した
タスク置き場へ送る。仕様の構造情報（テスト座標系・同値分割・交点の置き場）は処分より
前にコードの構造として実装へ移されている（「仕様とコードの構造対応」参照）— 処分で
消えるのはプロセスの足場だけで、構造は残る。

3ドキュメントは寿命で分かれている。1本に追記し続けると実装フェーズで全文を
コンテキストに入れることになり、肝心の制約が希釈されるため。

## プロジェクト設定 — リソースマップ（任意）

プロジェクト固有の情報参照先（規約・設計ドキュメント・正準コマンドなどの所在）を
`.dandori/resources.md` に列挙しておくと、各工程スキルが工程開始時に読んで活用する:

```markdown
# dandori リソースマップ

- docs/architecture.md — モジュール構成と依存ルール。土台調査の起点
- docs/coding-guide.md — 命名・エラーハンドリングの規約。実装・レビューの基準
- `make test` — unit ゲートの正準コマンド
```

ground は調査の起点に、review はレビュー観点に（パスのみ渡して独立性を維持）、
impl はブリーフの参照指示に、gate はゲートコマンドの正準定義に使う。
ファイルが無ければ従来どおり各工程がその場で探索する。

## 足場調査 — dandori-survey（任意）

既存コードベースの分析結果（アーキテクチャ・状態一覧・振る舞いフロー）を
`.dandori/map/` 配下のドキュメント群として生成・維持する工程。プロセスフローの
工程ではなく足場整備で、成果物は resources.md 経由で各工程に供給される。
ツールで再生成可能な事実（モジュール依存・ライブラリ一覧）は散文化せず
再生成コマンドだけを記録し、解釈だけを証拠アンカー + 検証等級つきで永続化する。
verify モードでコード変更による主張の腐敗を機械的に検出できる。
promote モードで、dandori フロー中に判明した永続的な知見（実行検証済の既存コード挙動、
不変条件、既存理解の訂正）を使い捨てドキュメントの処分前に map へ昇格できる。

## 状態モデルと状態マップ — 機械検査層（任意）

散文の仕様レビューには上限がある。実測では、23 ラウンドの独立レビューを通過した spec
からも、状態変数の直積を機械走査すると交点の抜けが 6 件出た
（`experiments/state-model/findings.md`）。この上限を突破するために、spec の散文の上に
機械検査可能な宣言層を漸進的に載せられる:

- 状態モデル（spec ローカル）— spec.md 内の fenced block に軸・直交/依存の宣言を書き、
  チェッカー（`skills/dandori-spec/scripts/check-state-model.ts`）が B 行のカバレッジ穴・
  制約違反・未裁定ペアを検査する。チェッカーが強制するのは正しさではなく**裁定の存在** —
  未分類ペアは「直交か依存か不明」という実行可能なタスクに変換され、ground の確認項目に
  そのまま接続される
- 状態マップ（プロジェクト全域）— states.md（survey 成果物）に機能境界をまたぐ共有状態の
  台帳（保存場所・owned/shared/borrowed 区分・writers/readers・SSOT）を宣言する。
  「この変更が書く状態 → その状態を Given に持つ spec」の機械導出で回帰範囲を確定し、
  SSOT の多重定義や borrowed 状態へのアプリ内書き込みを検査で弾く

どちらも任意の追加層で、外部の形式言語は導入しない。spec.md / states.md が単一の正の
まま、必要になった箇所から段階的に形式化する。発動条件と記法は付録を参照。

## 仕様とコードの構造対応 — テスト設計を設計に還流する

テスト項目を先に洗い出すと、その軸と交点の構造は**実装への構造要求**として読める。
軸がコード上の単一構造に対応し、交点の振る舞いが部品の合成から導出できるコードは、
テストマトリクスとモジュール構造が**同型**になっている — 逆に軸を1本増やすと分岐が
乗算で増えるコードは、この写像が壊れている。この対応が守るコード側の性質は
5つに分解でき、頭文字で **DRIVE** と呼ぶ:

- **D 直交性** Decomposability — 場合分けの切り口ごとに置き場が1つ
- **R 再現性** Reproducibility — どの状況も基準+差分でテスト上に再現できる
- **I 独立性** Isolation — 振る舞いを小さいテストで単独に固定できる
- **V 検証性** Verifiability — 合否判定を機械に任せられる
- **E 網羅性** Exhaustiveness — どの入力もどれかの場合に落ち、値が増えたら見直しが強制される

理論的背景とレバレッジ経済は [philosophy §11](docs/philosophy.md) 参照。
dandori は spec の語彙ごとにコード側の対応物を要求し、対応が書けない場所を
設計の警告として実装前に議題化する:

| spec の語彙 | 要求されるコードの対応物 | 守る性質 | 検査点 |
|------------|------------------------|---------|--------|
| 軸 | 単一の構造（型 / パラメータ / 分岐点 / leaf） | D 直交性 | ground の軸対応（design.md）+ D5 機械検査 |
| 軸の値域 | 型の分割（商）・分類述語の 1 関数集約 | E 網羅性 | partition_of 宣言 + ground 全域性突合 + impl 型強制 |
| Covers デルタ（base + 差分） | Given 空間ビルダーの引数（デフォルト = 基準シナリオ） | R 再現性 | fixture 語彙 + impl テスト規約 |
| dependent 交点 | 名前のある置き場（どの関数がその交点を担うか） | D 直交性 | 軸対応エントリに明記 |
| 直交宣言（列所有権型の reason） | モジュール境界（テスト削減の前提を守る独占） | D 直交性 | 軸対応 + codereview 不変条件レーン |
| Gate タグ | 固定単位の局所性（unit で固定できない = seam 不足） | I 独立性 | 乖離マーク（`e2e→unit`）→ ground の seam 議題 |

V（検証性）だけは対応表に行を持たない — ゲートタグの語彙そのもの（`manual` /
`visual` の比率と、spec 工程の manual 過多警告）が守っている。

**使う側が意識すべきこと**: spec 工程で軸を発見する行為は、設計の因数分解を実装前に
行っているのと同じ。だから「この交点だけ特別」が出たら、それは散らばった if ではなく
名前のついた置き場を要求するドメインの本物の相互作用であり、「軸に分けられない」は
仕様側の発見漏れか設計側の絡まりのどちらかのサインになる。

この層はクローズ運用と噛み合っている。Covers・基準シナリオ・同値分割・交点の置き場は
ドキュメントでなく**コードの構造**（ビルダー引数・デフォルト値・分類 Record・named unit）
として実装に移り、spec.md を処分した後も dandori を知らない読者に「よく整理された
コード」として残る。プロセスの追跡装置（テスト名の B-ID）は cleanup で消えるが、
構造の同型性は消えない — 仕様書を保守し続ける代わりに、仕様の構造情報をコードへ
移し切ってから仕様書を処分する、という解き方。

これとは別に、正準フォーマット層は常時機械検査される —
`skills/dandori/scripts/check-docs.ts` が spec.md の形式（必須セクション・B 行フィールド・
ゲートタグ語彙・fix 後の改番検知）と spec ↔ plan の B 行カバレッジ（未カバー B 行 =
実装されない仕様、の機械検出）を突合し、spec の fix / plan の確定の前提条件になる。
design.md の検証マーク・B 行対応・軸対応の検査（ground の完了条件）、B-ID ↔ テストコードの
突合による gate の初期トレース表生成、review-ledger の収束判定（review / codereview）、
map の証拠アンカー死活検査（survey verify）、state.yaml の整合検査（ルーターの再開判定）も
同じチェッカーが担う。

## インストール

```bash
git clone https://github.com/kr9ly/dandori.git

# グローバル導入（全プロジェクトで有効）
cp -r dandori/skills/* ~/.claude/skills/

# またはプロジェクト単位で導入
cp -r dandori/skills/* <project>/.claude/skills/
```

## 使い方

```
> /dandori 通知機能を作りたい
```

入口スキルがトリアージ（フル/短縮コースの判定）を行い、以降は各工程スキルが
順に案内する。中断しても `/dandori` で state.yaml から再開する。

## スキル一覧

| スキル | 役割 |
|--------|------|
| `dandori` | 入口・ルーター。トリアージと再開 |
| `dandori-spec` | 仕様策定。要件・振る舞い仕様・品質ゲートを壁打ちで一体定義 |
| `dandori-sketch` | デザイン接地。デザイン指示を読み取り、画面を変更/不可侵に全数分類（UIタスクのみ） |
| `dandori-ground` | 前提接地。土台・改変箇所・不変条件を「動かして」確認 |
| `dandori-review` | 独立レビューループ。収束条件・回数上限つき |
| `dandori-spike` | 検証スパイク。最高リスク前提を使い捨てコードで実地検証 |
| `dandori-plan` | 実装計画。マイルストーン分割 + コンテキストマニフェスト |
| `dandori-impl` | 実装。マイルストーン単位のディスパッチ + 発見の還流 |
| `dandori-codereview` | セルフコードレビュー。テスト緑の diff を4レーン並列レビュー + 反証フェーズ |
| `dandori-refine` | リファインメント。機械検査 + 3レーン（イディオム一致・再発明/簡素化・テスト品質）を1ラウンド適用 |
| `dandori-gate` | 最終ゲート。全仕様行のトレース表 → コミット裁定 |
| `dandori-feedback` | 継続改善の裁定点。外部の結論を台帳化 → フェーズ巻き戻しで改訂サイクル再走、完全 fix の裁定で annotate へ |
| `dandori-annotate` | コメント保全。design.md 等の処分で消える Why / Why not を全数走査し、コードコメントとして接地 |
| `dandori-cleanup` | クリーンアップ。annotate 完了後にプロセス言及（テスト名の B-ID 等）を除去 → フィーチャーのクローズ |
| `dandori-survey` | 足場調査。コードベース分析を .dandori/map/ に生成・維持（工程外） |
| `dandori-doctor` | 前提条件診断。各工程が依存する機械検査の基盤を実行確認し、整備の要/不要を提案（工程外） |
