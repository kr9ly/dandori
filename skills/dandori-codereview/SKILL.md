# dandori-codereview — セルフコードレビュー

impl と gate の間で、**テストが検証できない領域**をコードを読んで潰す工程。
gate が機械的に検出できるもの（テストの失敗、B 行の未検証）はこの工程の対象外 —
そのためテスト緑を入口条件にする。レビュアの独立性が品質の源泉であることは
dandori-review と同じで、同じ独立性ルール（パスのみ渡す・要約や弁解を混ぜない）に従う。

## 入口条件（テスト緑）

plan.md の unit / e2e ゲート（`.dandori/resources.md` に正準コマンドがあればそちらを優先）を
**この工程で再実行**し、すべて緑であることを確認する。赤があれば dandori-impl の
修正ループ（§2〜3 を単発で回す）へ差し戻し、この工程を開始しない。

- テスト緑はレビューの前提条件であって、レビュアに伝える情報ではない
- trace.md が存在してもレビュアに**渡さない** — 「検証済み」という先入観が
  テストの穴を探す動機を消す

## レビュー対象の確定

フィーチャーの実装差分を特定する（dandori はコミットを gate まで遅らせるため、
通常は working tree の全変更 + untracked ファイル）。レビュアには差分の
**取得コマンド**（例: `git diff` と untracked 一覧）を渡す。メインエージェントが
diff を要約・抜粋して渡すことはしない（解釈の混入）。

## 機械検査先行 — ミューテーションテスト（任意）

`.dandori/resources.md` にミューテーションテストの正準コマンドが宣言されていれば、
レーン起動前に **diff スコープ限定**で実行する（改変行だけを変異させる。
フルコードベースへの実行はしない — 実行時間が破綻する）。

- **生存ミュータント = 実行して得た証拠**。「この行の振る舞いはどのテストにも
  固定されていない」という具体的な指摘であり、エージェントの読解ベースの指摘より強い
- 生存ミュータントは major 相当として台帳（C-n）に記録し、**反証フェーズにかける**。
  反証の争点は等価ミュータント（意味を変えない変異で、殺しようがない）かどうか
- 生き残ったものはテスト追加として修正ループへ還流する
- 未宣言ならこのステップはスキップする。ツールの導入作業はこの工程でやらない
  （前提条件の診断と整備提案は dandori-doctor が担う）

ミューテーションテストが走った場合、テスト忠実度レーンの問いは機械が見られない
残余に絞る: テストが**仕様の意図した条件**を符号化しているか（実装の現状を追認する
だけのオラクルになっていないか）、生存ミュータントはどの B 行の固定の弱さか。
assertion の強さ・境界の網羅そのものは機械検査の結果を正とする。

## 4レーン並列ディスパッチ

観点 × 照合先で分割した4レーンを、それぞれ**独立のサブエージェント**として並列起動する。
各レーンの問いが狭いため、モデルは実装エージェントと同等以下（Sonnet クラス）で足りる。

レーンは**発見係（finder）**であり、精度の担保は反証フェーズ（verifier）に全委譲する。
精度をレーンに求めると自己検閲が起き、反証フェーズが飢える
（2026-07-08 実測: 4 レーンの生報告が計 blocker/major 1 件・minor 2 件・反証破棄 0 —
反証フェーズに仕事がない状態で「毎ラウンド major 1 件」のプラトーが続いた）。
見落としは反証フェーズでは回復できないが、偽陽性は反証フェーズが破棄できる —
レーンには recall を、反証には精度を、と役割を分ける。

| レーン | 渡すもの | 問い |
|--------|----------|------|
| テスト忠実度 | spec.md のパス | 各 B 行の緑のテストが本当にその振る舞いを固定しているか — assertion の強さ、境界条件の網羅、セットアップが仕様の条件を再現しているか。「実装が spec を満たすか」自体は impl / gate の管轄なので対象外 |
| 不変条件 | design.md のパス | 不変条件それぞれについて、diff がそれを破る経路がないか呼び出し元まで遡って検証せよ |
| blind バグ探し | **なし**（diff とコードベースのみ） | バグ・エッジケース欠落・並行性問題・リソースリークはないか |
| 周辺整合 | design.md + resources.md 記載の規約のパス | 呼び出し元への波及、規約違反、design が改変箇所と宣言していない場所への影響はないか |

blind レーンには spec.md / design.md を渡さない。仕様を知るレビュアは
「仕様どおりだから OK」バイアスがかかるため、仕様を知らない目を1枚必ず混ぜる。

共通テンプレート（レーン固有の問いと渡すものを差し替える）:

```
あなたは実装コードの独立レビューアです。コードベースへの読み取りアクセスがあります。
レビュー対象の差分: <取得コマンド>
<レーン固有: 照合先ファイルのパス + 問い + 走査対象の全数列挙
（テスト忠実度 = 全 B 行 / 不変条件 = 全不変条件 / blind = 全ファイル・全 hunk /
周辺整合 = 改変ファイルごとの呼び出し元）>

ルール（あなたは発見係 — 指摘の白黒は後段の独立反証フェーズが付ける）:
- 見落としは反証フェーズでは回復できないが、偽陽性は反証フェーズが破棄できる。
  疑いは自己検閲せず列挙すること。「確信が持てないから黙る」は禁止
- 照合対象をまず全数列挙し、1 件ずつ疑いを探すこと。目についた 1 件で走査を止めない
- 各指摘に疑いの根拠（ファイル:行）と、可能なら「何を確認すれば白黒つくか」を付けること
- 深刻度は「指摘が真だった場合」の深刻度で付ける。確信度で下げない
  - blocker: 仕様を満たさない / データ破壊・欠損につながる
  - major: エッジケースの欠落、不変条件違反の可能性、隠れた波及
  - minor: 改善提案・可読性。minor だけは反証フェーズを通らずユーザーに直接届くため、
    確信のあるもののみ報告すること
- 修正は行わない。指摘の列挙だけを返すこと
```

diff が大きい場合の二次分割は**レーン内**で行う（例: テスト忠実度レーンを spec.md の
正常系/異常系の軸ごとに複数エージェントへ分割）。ファイル単位の空間分割を主軸に
しない — 全体像を失い、横断的な問題を構造的に見逃す。

## 反証フェーズ

レーンは recall 優先で偽陽性を意図的に含む — **この工程が唯一の精度ゲート**。
blocker / major の各指摘について、**新しいサブエージェント**に反証を試みさせる:

```
以下のコードレビュー指摘を反証してください。指摘は recall 優先の発見係によるもので、
偽陽性を多く含む前提です — この反証が唯一の精度ゲートです。指摘が誤りである可能性 —
事実誤認、実際には到達不能なパス、既存仕様として正当な挙動 — を
コードを読んで確認すること。
「到達不能」を反証根拠にする場合は、コード経路だけでなくデータ由来の到達可能性
（nullable なカラム・スキーマ変更前から残る古いレコード・部分書き込み・外部からの入力値）を
確認してからにすること — 型注釈や現行コードパスのみを根拠に到達不能と断定しない。
真とも偽とも確定できない場合は生存（反証不成立 — 安全側）とする。
反証の成否と根拠（ファイル:行）を報告すること。
```

- 反証されなかった指摘だけを還流に回す。反証された指摘は破棄する
- minor は反証フェーズを通さず、重複排除して最後に一括でユーザーに提示する
  （ループの駆動源にはしない）

## 指摘台帳（review-ledger.md）

dandori-review と同じ台帳 `.dandori/specs/<feature>/review-ledger.md` を共用する
（この工程の ID 接頭辞は `C-n`。形式・処置語彙・照合ルールは dandori-review を参照）。
レーンから受け取った blocker / major は**反証フェーズにかける前に全件追記**し、
反証の結果を処置に記録する（`反証破棄` + 反証根拠、または `反映済`）。
破棄した指摘こそ記録する — 後続ラウンドのレーンが同じ指摘を再生産したとき、
台帳照合で「反証済み」と即断でき、反証エージェントの重複起動を省ける。
台帳をレビュア・反証エージェントに渡さないのは dandori-review と同じ。

## 還流と修正ループ

生き残った blocker / major を dandori-impl の修正ループ（§2〜3 単発）で修正する。

- 指摘が spec / design の記述とのズレを含む場合は design.md の発見ログに追記する（既存経路）。
  spec の振る舞いに波及するものはユーザーに裁定を仰ぐ
- ユーザー裁定（却下・一般裁定）は適用対象の近傍（コードコメント・design の該当節）に
  明文化する — 各ラウンドのレーンは台帳を読まないため、台帳の記録だけでは再指摘を
  止められない（詳細は dandori-review の台帳節「裁定の明文化」）
- 修正後、影響を受けたゲートを再実行して緑を確認し、修正に関係するレーンだけ
  **新しいサブエージェントで**再レビューする（前ラウンドの記憶を持たせない）
- state.yaml の `codereview.rounds` をインクリメント

### 収束条件

dandori-review と同じ判定を使う。チェッカー
（`node <dandori-repo>/skills/dandori/scripts/check-docs.ts ledger <review-ledger.md>`）が
C-n 系列のラウンド推移と判定（passed / escalated / 継続）を機械出力する
（反証破棄の行は生存数から除外され、反証破棄済み論点の再指摘は escalate 材料に数えない）:

- **通過**: 反証を生き残る blocker / major がゼロのラウンドが出たら `codereview.status: passed`
- **escalate**: 台帳に `再燃→` が付いた（反映済の論点の再指摘 — 反証済みの再生産は除く）、
  または3ラウンド以上連続で件数が減らずかつ最新ラウンドに未解消の再燃が含まれる →
  `codereview.status: escalated` にしてユーザーに裁定を仰ぐ

台帳に行を追記しなかったラウンド（指摘ゼロ / 全て反証済みの再生産）は、チェッカー実行前に
`<!-- round: C Rd=7 指摘なし -->` 形式のマーカー行を台帳末尾に追記する — 行が残らない
ラウンドはチェッカーから観測できず、過去の停滞パターンから escalated を返し続ける
（形式の正準定義は dandori-review の台帳節）。

## Workflow スクリプト（任意 — 対応環境のみ）

この工程の制御フロー（入口検査 → レーン並列 → 台帳追記 → 反証 → 修正 → 収束判定のループ）は
完全に決定的なので、サブエージェントのオーケストレーション機構（Claude Code の Workflow ツール等）が
使える環境では、同梱の `workflow.js`（このスキルと同じディレクトリ）で機械駆動してよい。
利益は速度ではなく**情報隔離の構造的な強制** — 「パスのみ渡す」「台帳をレビュア・反証エージェントに
渡さない」「blind レーンに spec を渡さない」がプロンプトテンプレートに固定され、実行のたびの
自制に依存しなくなる。

- args: `specDir`（specs/<feature>）/ `diffCommand` / `gates`（正準コマンド配列）/
  `checkDocs`（check-docs.ts の実行プレフィックス）、任意で `resources` / `mutationCommand` /
  `maxRounds` / `workRoot`。JSON オブジェクトで渡す（文字列で届く環境でもスクリプト側で
  JSON.parse に正規化される）
- `workRoot`: コードの作業ルート。worktree 並列レーン等、コードがセッションの主作業ディレクトリと
  別の場所にあるとき指定する（サブエージェントは主作業ディレクトリで動くため、指定しないと
  相対パスのゲートが失敗し、他 worktree を誤読・誤編集するリスクがある）。レビュー・反証・修正・
  入口検査のプロンプトに「コードの読み書きとゲート実行を workRoot 内に閉じる」指示が注入される
- スクリプトは「収束 or 脱出」まで回して結果を返す。**裁定はスクリプト内で解決しない** —
  `needs_adjudication`（spec 波及）・`escalated`・`gate_red`・`blocked` で停止したら、
  minor の採否・state.yaml の更新とあわせて本文の該当節に従ってメインエージェントが処理する。
  処理後の再開は**新規実行**で行う（台帳の C-n ラウンドから続きが導出される）。
  **resume（resumeFromRunId 等のキャッシュ再生機構）は使わない** — プロンプトが不変のため
  全呼び出しがキャッシュ再生され、ディスクを読み直さずに停止時の結果をそのまま返す
  （dandori-review で 2026-07-08 実測）
- 使えない環境では本文の手順をそのまま手で回す。挙動は同一（スクリプトは本文の写しであり、
  乖離したら本文が正）

## 完了条件

- `codereview.status: passed`（または escalated 後のユーザー裁定で通過）
- 修正がすべて working tree に反映され、ゲートが緑
- minor の採否がユーザーに提示済み
- state.yaml: `phase: refine`、`phases_done` に codereview 追加

短縮コースではこの工程はデフォルトでスキップする（`codereview.status: skipped`）。
ユーザーが求めれば短縮コースでも実施してよい。
