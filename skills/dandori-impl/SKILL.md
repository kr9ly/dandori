# dandori-impl — マイルストーン実装

plan.md に従い、マイルストーン単位で実装エージェント（サブエージェント）に作業を
ディスパッチする工程。メインエージェントは実装せず、ブリーフ組み立て・ゲート検証・
発見の還流というメタ作業に徹する（コンテキストを実装詳細で消費しない）。

## マイルストーンごとのループ

state.yaml の `impl.milestones_done` から次のマイルストーンを特定し、以下を繰り返す:

### 1. ブリーフ組み立て

plan.md のマニフェストに従い、実装エージェントに渡す情報を組み立てる:

- マニフェスト記載のセクション**だけ**を渡す（spec/design の全文を渡さない — 希釈防止）
- 不変条件は常に全文含める
- design.md の**発見ログのうち、このマイルストーンに関係する項目**を含める
  （前のマイルストーンで見つかったズレを知らずに実装させない）
- マニフェストにリソースマップ由来の参照先（規約等）があれば、その**パス**をブリーフに
  含め、「実装前に読んで従うこと」を指示する（内容は貼り込まない — 希釈防止）

### 2. ディスパッチ

サブエージェントに以下の趣旨で指示する:

```
以下のブリーフに従って実装してください。

<ブリーフ: 対応 B 行 / 関連する設計エントリ / 不変条件全文 / 手順概略 / ゲート>

ルール:
- 不変条件に抵触する変更が必要になったら、実装せず停止して報告すること
- 実装中に「仕様・設計の記述と現実のコードが食い違う」箇所を見つけたら、
  勝手に解釈して進めず、構造化して報告に含めること:
  [発見] <何が> <どう食い違うか> <あなたの対応（回避した/保留した）>
- 完了時に指定のゲートを自分で実行し、結果（生の出力）を報告に含めること
- テストを通すためにテスト側を弱める変更は禁止
- 作成・変更するテストの名前（test タイトルまたは describe）に、検証対象の
  B 行 ID を含めること（例: test("B-3: 在庫ゼロは 409 を返す", ...)）。
  1 テストが複数 B 行を検証するなら全 ID を列挙する
```

テスト名への B-ID 埋め込みは工程の追跡装置（gate の trace が grep で機械突合する）で
あって、プロダクトコードの命名規約ではない — feedback 工程で「実装は完全に fix」の
裁定が出た後、dandori-cleanup が除去する（改訂サイクルが回っている間は現役のまま残る）。

### Given 空間ビルダー（状態モデルがある spec のみ）

状態モデルがある spec では、**最初のテストを書くマイルストーンでシナリオビルダー**
（テストユーティリティ）を立て、以降の全テストの Given 構築をビルダー経由にする。
Covers のデルタ構文とビルダー引数が同型になり（`Covers: base + coupon=cp_ok` ↔
`setupScenario(ctx, now, { coupon: 'valid' })`）、テストがどの座標を検証しているかが
テスト名でなく**構築コードそのもの**に現れる — cleanup が B-ID を除去した後も
トレース構造がコードに残る。

- パラメータ = 軸、値の語彙 = 状態モデルの `fixture:` 宣言（appendix-state-model 参照）。
  **S 番号・B 番号をビルダーに持ち込まない** — ドメイン語彙で命名する
- 省略時のデフォルト = base（基準シナリオ）。基準の変更がデフォルト値 1 箇所に局所化される
- per_item 軸は行仕様の**配列**で表現する（count 系の軸は配列長として畳む。
  混在 Covers の `+` は配列要素の混在と同型）
- only 制約はビルダーの構築時に強制する（到達不能 Given をフィクスチャでも作れなくする）
- `partition_of:` 宣言のある軸（appendix-state-model「型対応の宣言」）は分類を型で強制する —
  値域が有限型なら `Record<型, 軸値>`（satisfies）で分類表を書き、**型に値が増えたとき
  spec 較正までコンパイルが割れる**ようにする。無限値域（文字列の商等）は分類述語を
  1 関数に集約し、パーティションテスト（全クラス実在 + 代表値の分類一致）で固定する
- 時刻依存の値（開催中・期限内）があるなら基準時刻 `now` をビルダー引数にする
- ブリーフには「Given の構築はシナリオビルダー経由。必要な軸値がビルダーにないときは
  ビルダーに追加してから使う（テスト内で直接 insert しない）」を含める。ビルダーの
  所在（または新設指示）はマニフェスト経由で全マイルストーンに開示する

### 3. 検証と還流

報告を受けたら:

- **ゲート結果を検証する**。エージェントの「通りました」を信用せず、報告された出力を確認し、
  疑わしければ自分で再実行する
- **[発見] を design.md の発見ログに追記する**。その上で影響を判定:
  - design.md の記述修正で済む → 修正して続行
  - spec.md の振る舞いに波及する → 一時停止してユーザーに裁定を仰ぐ
    （軽微なら B 行を修正して続行、重大なら plan からやり直し）
- state.yaml の `impl.milestones_done` を記録する — 整数カウンタのインクリメント（逐次）または
  完了マイルストーン ID の追記（`[M1, M2]` 形式 — 並列実装で完了順が M 番号順と一致しないときはこちら。
  workflow.js はこの形式で記録する）。check-docs.ts state は両形式を受理する

### 4. 進捗報告

各マイルストーン完了時にユーザーへ1〜3行で報告する（M番号、ゲート結果、発見の有無）。
裁定が必要な場合以外はユーザーの応答を待たず次へ進む。

## 並列化

複数マイルストーンを並列にディスパッチしてよいのは、**依存がなく、かつ触るファイル集合が
重複しない**ことを plan.md から確認できる場合のみ。判断に迷ったら逐次で進める。
並列時も還流（§3）はメインエージェントが直列に処理する。

## Workflow スクリプト（任意 — 対応環境のみ）

マイルストーンループ（ブリーフ組み立て → ディスパッチ → 独立ゲート検証 → 発見の還流）は
決定的なので、サブエージェントのオーケストレーション機構（Claude Code の Workflow ツール等）が
使える環境では、同梱の `workflow.js`（このスキルと同じディレクトリ）で機械駆動してよい。
利益は希釈防止の構造的な強制 — ブリーフ組み立て係（マニフェスト記載セクションだけを抽出）と
実装エージェントを分離し、実装エージェントのプロンプトに spec/design のパスを存在させない。

- args: `specDir`（specs/<feature> — plan.md / state.yaml もこの直下）、任意で `maxFixRounds`。
  JSON オブジェクトで渡す（文字列で届く環境でもスクリプト側で JSON.parse に正規化される）
- 逐次実行のみ。§並列化の判断（依存なし・ファイル非重複）はメインの領分なので
  スクリプトには持ち込まない — 並列にしたい場合は本文の手順で
- 戻り値は `done` / `needs_adjudication`（spec 波及の [発見]）/ `halted`（不変条件抵触）/
  `gate_red` / `blocked`。`impl.milestones_done` はマイルストーンごとにスクリプトが記録する。
  裁定と最終フェーズ遷移（`phases_done` 等）はメインが処理し、裁定後に**新規実行**すれば
  残りのマイルストーンから再開される（進捗は state.yaml の `milestones_done` がディスク上で
  保持し、入口確認が続きを導出する）。**resume（resumeFromRunId 等のキャッシュ再生機構）は
  使わない** — プロンプトが不変のため全呼び出しがキャッシュ再生され、ディスクを読み直さずに
  停止時の結果をそのまま返す（dandori-review で 2026-07-08 実測）
- 使えない環境では本文の手順をそのまま手で回す。挙動は同一（スクリプトは本文の写しであり、
  乖離したら本文が正）

## 完了条件

- 全マイルストーンのゲートが通過している
- すべての [発見] が design.md の発見ログに記録され、裁定済みである
- state.yaml: `phases_done` に impl 追加。次フェーズはフルコースなら `phase: codereview`、
  短縮コースなら `phase: gate`
