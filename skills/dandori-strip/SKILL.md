# dandori-strip — プロセス言及の除去（対外化準備）

B-ID や spec/design への参照は**プロセスの追跡装置**であって、プロダクトコードの語彙ではない。
gate のコミットで「トレース可能なスナップショット」が履歴に固定された後、この工程が
プロセス由来の言及を取り除く。dandori を使っていない読者がテスト名やコメントを読んで、
前提知識なしに意味が取れる状態がゴール — feedback の安定点で受ける**外部レビューは、
dandori の痕跡がないコードに対して行われる**。

この工程は**可逆**である — B-ID を剥がしても trace.md は現役のまま残り（処分は cleanup）、
旧 B 行↔テストの対応は trace.md と strip コミットの rename 差分でいつでも再現できる。
改訂サイクルの再 gate は feedback の trace_scope 裁定（dandori-feedback §5）に従う。
不可逆な店じまい（フィーチャードキュメント処分・墓碑コミット）は cleanup の仕事であり、
この工程はコードの表面だけを整える。

annotate（付与）→ strip（除去）の順序は固定 — annotate で付与したコメントに dandori 語彙が
うっかり混入しても、この工程の residue 検査が機械検出できる。

## 入口条件

- state.yaml: `phase: strip`（gate 通過・コミット済み + annotate 完了 —
  `phases_done` に annotate）
- trace.md が存在する — B 行↔テストの対応の正であり、この工程の作業リスト
  （処分はしない — cleanup まで生存する）

## 手順

### 1. 作業リストの確定

trace.md の根拠列（file:line）から対象ファイル集合を確定し、残存言及を機械列挙する:

```
node <dandori-repo>/skills/dandori/scripts/check-docs.ts residue <対象ファイル|ディレクトリ...>
```

B-ID トークンと dandori への言及が file:line つきで列挙される。**対象は今回の
フィーチャーが作成・変更したファイルに限る** — 並行フィーチャーのテストは
まだ gate 前で B-ID が現役のため、プロジェクト全体への一括適用はしない。

対象ファイル集合は目視で列挙せず**機械生成する**（ディレクトリの手打ち列挙は
取りこぼす — get-cart 初回クローズで実証）。2 系統の和集合を取る:

```bash
grep -oE '[A-Za-z0-9_/.-]+\.tsx?' trace.md | sort -u        # trace.md 根拠列の全ファイル
git show --name-only --format= <gateコミット> | sort -u      # gate コミットの作成・変更ファイル
```

（テストファイルだけでなく、その実装側ファイル・Input 型・index.ts にもコメントの
言及が残るため、gate コミット側のリストが実装ファイルを補完する）

改訂サイクルの再訪では、今回のサイクルの gate コミットが触れたファイル
（新 B-ID 付きテストを含む）が対象に加わる — 作業リストを上記の 2 系統から
機械生成していれば自然にカバーされる。

### 2. 言及の除去

- **テスト名**: B-ID トークンを剥がし、残りが自然文として成立すればそれを採用する。
  成立しなければ spec の該当 B 行（タイトル・Given/When/Then）から、振る舞いを
  説明する名前に書き直す
- **コメント**: dandori 言及（B-n / R-n / C-n / spec.md / design.md / dandori）を含む
  コメントは、内容が読者に必要なら言及なしの自然文に書き換え、プロセス管理のため
  だけなら削除する
- **機能的依存は削除しない**: dandori 成果物をデータとして読むコード・テスト
  （例: `.dandori/` 配下のマップをパースする整合テスト）は削除でなく**ユーザー裁定に
  浮上させる** — 成果物を長寿命ドキュメントとして `.dandori/` 外へ昇格するか、
  依存を残すか。残す裁定なら該当行に `dandori-ok: <理由>` コメントを付ける
  （residue 検査の除外マーカー — 裁定の記録を兼ねる）。マーカーは**その行自身と
  直後の 1 行**を除外する — 守りたい参照の行内に置くか、直前のコメント行に置く。
  2 行以上に及ぶ参照は各行（または 1 行おき）にマーカーが必要
- **振る舞いに触れない**: 変更はテスト名・コメントに限る。アサーション・ロジック・
  テストの追加/削除は禁止（それは strip ではなく別フィーチャーの仕事）

#### 並列分割の安全規約

対象ファイルが多く、除去をサブエージェントに並列分割する場合は以下を守る
（並列バッチの 1 体が他バッチ編集中のファイルを「無関係な巻き添え」と誤認して
`git checkout --` を実行し、未ステージ作業を破壊した実績がある）:

- **バッチ間のファイル集合は素集合にする** — 分割前にバッチごとの担当ファイルを
  明示列挙し、重複がないことを機械確認する（`sort | uniq -d` が空であること）
- 各エージェントのプロンプトに**担当ファイルの明示列挙**と
  「**担当ファイル以外への git 操作（checkout / restore / stash / clean 等）は禁止。
  担当外ファイルに差分が見えても無視すること — 他バッチの編集中である**」を必ず注入する

### 3. 検証（機械確認）

- residue を再実行して **exit 0** を確認する（`dandori-ok:` 行は除外される）
- ゲート（unit / e2e — `.dandori/resources.md` の正準コマンド）を再実行して
  **緑 + skipped/todo 0 件**を確認する — テスト名の変更が test filter や
  snapshot 名に波及していないことの確認

### 4. strip コミット

gate のコミットとは**分離した専用コミット**にする — 直前の履歴に B-ID つきで
トレース可能なスナップショットが残り、B 行↔テストの対応は後からでも git で
再現できる（trace.md 処分後の唯一の再現手段）。

### 5. 遷移

state.yaml を更新して dandori-feedback へ:
`phase: feedback`、`strip.status: done`、`phases_done` に strip 追加。
feedback は安定点 — 外部レビュー・運用の結論を待つ場合はここでセッションを終えてよい。

## スキップ

B-ID つきの名前をコード上に残す方針のプロジェクト（トレーサビリティを履歴でなく
現物で保ちたい等）では、この工程をスキップして `strip.status: skipped`（理由つき）とし、
`phases_done` に strip を追加して feedback へ進む。この場合 B-ID が現役のまま残るため、
再 gate は通常の全行機械突合が効く（feedback の trace_scope 裁定は不要）。

## 完了条件

- residue exit 0 + ゲート緑 + strip コミット済み（または skipped 裁定）
- state.yaml: `phase: feedback`、`phases_done` に strip 追加
