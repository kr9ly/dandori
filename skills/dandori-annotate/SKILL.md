# dandori-annotate — 消える Why の行き先確定

spec.md を含むフィーチャードキュメント一式は cleanup で処分される（既定運用 —
`docs/appendix-records.md`）。そこに記録された設計判断のうち、「コードを読んだ第三者が
『なぜこうなっている？』『なぜ素直な方法 X でないのか？』と疑問を持つもの」への回答は、
処分と同時に読者の手から消える。この工程は処分の**前に**、消える Why / Why not の
行き先を全数確定する — **読者が出会う場所から逆算して置く**: コード行に anchor がある
判断はコードコメントへ、フィーチャー単位の裁定は墓碑コミットの記録へ、push 型の情報
（未解決リスク・分離タスク）はタスク置き場へ。

cleanup（プロセス言及の**除去**）と対になる**付与**の工程であり、必ず cleanup より先に
行う — この工程で付与したコメントに dandori 語彙がうっかり混入しても、後続の cleanup の
residue 検査が機械検出できる（逆順だと検査済みのコードに未検査のコメントが増える）。

## 入口条件

- state.yaml: `phase: annotate`（gate 通過・コミット済み + feedback で完全 fix の裁定済み）
- 処分予定ドキュメント（design.md / review-ledger.md / sketch.md）が1つ以上存在する
  — 1つも無ければ「スキップ」へ

## 手順

### 1. 候補の抽出

処分予定のドキュメントを走査し、「消えると困る判断」を列挙する:

| 源 | 見るもの |
|----|---------|
| design.md「不変条件」 | 変えてはいけない理由がコード側から自明でないもの |
| design.md「発見ログ」 | 仕様と現実のズレの発見 — 「一見バグ・冗長に見えるが意図的」型の対応 |
| design.md「土台/改変箇所」 | 採らなかった選択肢（別 API・別方式を検討して棄てた記録） |
| review-ledger.md の却下行 | 「こうすべきでは？」への反証・裁定理由 — 将来のレビューアも同じ指摘をする |
| sketch.md の不可侵領域 | 「触ってよさそうに見えるが触ってはいけない」UI コードがあれば |
| spec.md | スコープ外の裁定注記、取り消し線 B 行、状態モデルの直交裁定、テスト設計の骨格、未解決事項の残存項目 |

`.dandori/records.md`（運用宣言 — タスク置き場・マージ方式等。形式は
`docs/appendix-records.md`）が未整備なら、**この工程の冒頭でユーザーと壁打ちして作成する**。
タスク置き場を宣言できないプロジェクトは spec 処分を選べない（`方式: retain`）—
その場合 spec.md は候補源から外れる（残るため）。

判定基準は1つ: **その判断を知らない読者が、該当コードを「直したくなる」か。**
直したくなる（そして直すと壊れる・退行する・過去の議論を再演する）なら Why not
コメントが必要。ならないなら不要 — 過剰付与はノイズ（コメントはコードが示せない
制約だけ、が大原則。判断の由来や工程の記録はコメントにしない）。

### 2. 照合と付与

候補ごとに該当コード位置を特定し、処置を決める:

- **既にコメントがある**（impl 工程が書いた等）→ 内容が Why / Why not に答えているか
  確認する。dandori 語彙（B-n / design.md 参照）混じりでも可 — cleanup が自然文化する
- **ない** → 付与する。書き方:
  - dandori 語彙を使わない自然文で書く（cleanup の作業を増やさない）
  - 周辺コードのコメント密度・言語・イディオムに合わせる
  - 「何をしているか」でなく「なぜか / なぜ X でないか」を書く。棄てた代替案は
    「X だと〜が壊れる」まで書いて初めて再演を防げる
- **コミット記録へ**（pull 型 — 疑問を持った読者が探しに来る情報）: フィーチャー単位・
  裁定単位の Why / Why not（確定裁定・消したもの・テスト戦略）。固定キー
  （`Decision:` / `Dropped:` / `Test-Strategy:` + records.md の追加キー）で**下書きを作り**、
  「## Why 保全」節に溜める — cleanup が墓碑コミットのメッセージに転写する
- **タスク置き場へ**（push 型 — 次の作業を選ぶ人が必ず見る場所）: 未解決リスク・
  分離した独立タスク。records.md 宣言のタスク置き場へこの工程で転送する。
  **push 型をコミット記録に載せるのは禁則** — コミットメッセージは誰も能動的に開かない
- **長寿命ドキュメントへ**: アーキテクチャレベル・複数フィーチャーを拘束する判断は
  ADR（records.md / resources.md で宣言された置き場）や `.dandori/map/`、プロジェクトの
  docs/ へ。`.dandori/map/` への昇格は cleanup の survey promote でも拾われるが、
  **コード読者が出会う場所に置くべきものはこの工程が主担当**
- **不要** → 理由を一言記録（コードから自明 / テストに転写済み / 読者が疑問を持たない）

処置の記録（付与 file:line / 既存で十分 / 長寿命ドキュメントへ / 不要=理由 —
コミット記録運用ではコミット記録の下書きも含む）は design.md 末尾に「## Why 保全」節
として追記する — design.md はこの後処分されるので汚してよく、中断時の再開点になる。
design.md が無い短縮コースでは trace.md 末尾に置く（trace.md は cleanup まで必ず生存し、
同じく処分される使い捨て）。

### 3. 検証とコミット

- 変更が**コメントと長寿命ドキュメントに限られている**ことを diff で確認する —
  ロジック・テスト・アサーションには触れない（それは annotate ではなく別フィーチャーの仕事）
- ゲート（unit — `.dandori/resources.md` の正準コマンド）緑を確認する
  （コメントのみの変更でも doctest・コメント lint・snapshot に波及しうる）
- **annotate 専用コミット**にする（gate / cleanup のコミットと分離 — 付与の diff が
  それ自体「この実装の Why 一覧」として履歴に残る）。付与ゼロならコミット不要

### 4. 遷移

state.yaml を更新して dandori-cleanup へ:
`phase: cleanup`、`annotate.status: done`、`annotate.annotated: <付与数>`、
`phases_done` に annotate 追加。

## スキップ

既定運用（tombstone）では spec.md が常に候補源のためスキップしない。
`方式: retain` のプロジェクトで、処分対象ドキュメント（design.md / review-ledger.md /
sketch.md）が**1つも存在しない**場合のみ `annotate.status: skipped`（理由つき）として
cleanup へ進む。候補を走査した結果ゼロだった場合は skip ではなく `done`（annotated: 0）—
「消える判断は無い」と確認した事実に価値がある。

## 完了条件

- `.dandori/records.md` が整備済み（この工程で作成した場合を含む）
- 候補全数に処置が付いている（付与 / 既存で十分 / コミット記録へ / タスク置き場へ /
  長寿命ドキュメントへ / 不要=理由）
- push 型項目（未解決リスク・分離タスク）が全件タスク置き場へ転送済みで、
  コミット記録の下書きが「## Why 保全」節に揃っている
- ゲート緑 + annotate コミット済み（付与ゼロならコミット不要）
- state.yaml: `phase: cleanup`、`phases_done` に annotate 追加
