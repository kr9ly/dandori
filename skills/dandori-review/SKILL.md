# dandori-review — 独立レビューループ

仕様と設計を第三者視点で検証する工程。レビューアの独立性が品質の源泉なので、
**メインエージェントの解釈・補足・弁解をレビューアに一切渡さない**。

## 独立性の担保（厳守）

- サブエージェントには **spec.md / design.md のファイルパスだけ**を渡す。
  プロンプトに内容の要約・言い換え・背景説明を埋め込まない（無意識の答え合わせ・バイアス注入を防ぐ）
- 調査して初めて分かった原因、メインエージェントの仮説、「ここは意図的にこうした」等の
  弁解は渡さない
- レビューアはコードベースへの読み取りアクセスを持つ（検証に必要）
- `.dandori/resources.md`（リソースマップ）に規約・設計ドキュメント・バグパターン集が
  あれば、その**ファイルパスを**指示に追加してよい（レビュー観点の項目として列挙する）。
  内容の要約・抜粋は渡さない — 要約はメインエージェントの解釈を混入させる

## レビューアへの指示テンプレート

サブエージェント（読み取り専用が望ましい）を以下の趣旨で起動する:

```
あなたは実装ドキュメントの独立レビューアです。以下の2ファイルを読み、批評してください。
- <.dandori/specs/<feature>/spec.md への絶対パス>
- <.dandori/specs/<feature>/design.md への絶対パス>

観点:
1. 要件不達 — この設計を実装しても spec のゴールを満たさない箇所はないか
2. エラーパス欠落 — 状態変数の洗い出しに漏れはないか。spec の異常系 B 行で
   カバーされないエラーパスを、実際のコードから逆算して探せ
3. 前提の誤り — design.md の「土台」「不変条件」の主張を鵜呑みにせず、
   コードベースを自分で読んで検証せよ。特に [読解のみ] マークの前提を重点的に疑え。
   [実行検証済] も証拠形式を監査せよ: 実行コマンドと観測結果の併記がないもの、
   コマンドを再実行して主張どおりにならないものは、マークの降格を指摘せよ
4. 整合性 — spec と design の間の矛盾、design が触ると言っていない箇所への隠れた影響

各指摘に深刻度を付けること:
- blocker: 要件を満たさない実装になる / 前提が事実と異なる
- major: エラーパス・状態変数の欠落、不変条件の見落とし
- minor: 改善提案・表現の曖昧さ

出力: 深刻度別の指摘リスト（各指摘に根拠となるファイル:行を付ける）
```

spec.md に状態モデル（`dandori-state-model` ブロック）がある場合、観点 2 の
組み合わせ網羅・交点カバーはチェッカーが機械的に担保している — レビューアには
モデルが**原理的に検出できない型**への集中を指示に追記する:
軸内の逐次連鎖の発見 / 分類述語の値域の穴 / 未発見の状態変数 /
値内部のセマンティクス（照合条件の非対称等）/ エラー優先順位の導出
（詳細: `docs/appendix-state-model.md` の「検出できないもの」）。

さらに**直交宣言の監査**を指示に追記する。チェッカーが強制するのは裁定の存在であって
正しさではない — 「裁定のふり」（チェッカーを黙らせるためだけの吟味なし直交宣言）は
チェッカー自身には検出できず、ここが唯一の第三者検査点になる:

```
状態モデルの orthogonal / orthogonal_groups の各宣言について:
- reason が spec 本文から実際に導けるか検証せよ。本文に根拠のない reason は
  それ自体を指摘せよ（major — 裁定のふりの疑い。ground 送りへの差し戻しを提案）
- 反例となる交点を探せ: その 2 軸の値の組み合わせで挙動が変わるケースを、
  spec の Then の散文（「〜の場合のみ」「〜は除外」）とコードの両方から探すこと。
  見つかれば依存宣言への昇格 + 交点 B 行の追加を指摘せよ（blocker）
```

## 指摘台帳（review-ledger.md）

全指摘を `.dandori/specs/<feature>/review-ledger.md` に永続化する。会話にしか存在しない
指摘（特に**却下したもの**）を作らないため、また収束判定（再燃・振動）を記憶と裁量でなく
**台帳との照合**という機械的作業に落とすため。dandori-codereview と共用
（ID 接頭辞で区別: この工程は `R-n`、codereview は `C-n`）。

```markdown
| ID | Rd | 深刻度 | 論点（一行） | 処置 | 根拠・理由 |
|----|----|--------|--------------|------|-----------|
| R-1 | 1 | blocker | B-3 の Given が既存 API の実挙動と矛盾 | 反映済 | design.md 土台節を修正 |
| R-2 | 1 | major | 在庫ゼロ×クーポンの交点 B 行がない | 却下 | スコープ外（spec §外）— ユーザー裁定 |
| R-3 | 2 | major | R-1 と同根: 修正後も戻り値の型が不一致 | 再燃→R-1 | escalate 判定の材料 |
```

- 処置の語彙: `反映済` / `却下`（理由必須） / `保留`（minor の採否待ち） / `再燃→<ID>` /
  `反証破棄`（codereview のみ — 反証根拠必須）
- ラウンドごとに、受け取った指摘を**まず全件台帳に追記してから**処置に入る。
  その際、既存行と同一論点かを照合し、同一なら新規行にせず `再燃→<ID>` を記録する

## ループの回し方

1. レビューアを起動し、指摘リストを受け取る
2. **全指摘を台帳に追記する**（既存行との照合込み — 上記）
3. **blocker / major をドキュメントに反映する**。反映は自分で判断できるものは自律で行い、
   spec の意図に関わるもの（ゴール定義・スコープの変更を伴うもの）はユーザーに裁定を仰ぐ。
   処置を台帳に記録する
4. minor は一括でユーザーに提示し、採否をまとめて決める（ループの駆動源にはしない）
5. 反映後、**新しいサブエージェント**で再レビュー（前ラウンドの記憶を持たせない。
   台帳もレビューアには渡さない — 前ラウンドの結論による先入観を防ぐ）
6. state.yaml の `review.rounds` をインクリメント

### 収束条件

台帳から機械的に判定する。判定はチェッカーが機械化している —
`node <dandori-repo>/skills/dandori/scripts/check-docs.ts ledger <review-ledger.md>`
がラウンド推移と判定（passed / escalated / 継続）を出力し、あわせて台帳の形式
（未処置の行・理由なし却下・保留の誤用・再燃参照切れ・ID 重複/欠番）を検査する。
各ラウンドの処置記録後に実行する。判定基準:

- **通過**: blocker と major が両方ゼロのラウンドが出たら通過。`review.status: passed`。
  blocker / major が残っている状態でのラウンド数による打ち切りはしない（ユーザー裁定 2026-07-04）
- **escalate（脱出弁）**: ラウンド数ではなく**収束の兆候の消失**で判定する。以下のいずれかに該当したら
  `review.status: escalated` にしてユーザーに裁定を仰ぐ:
  - 台帳に `再燃→` が付いた（反映済の論点が後続ラウンドで再指摘された —
    レビューアと反映の間で解釈が振動している）
  - 3ラウンド以上連続で blocker+major の件数が減っていない（台帳の Rd 列で数える）
  （典型: 要件自体が曖昧 → spec に差し戻し / レビューアが過剰検出 → 指摘を却下して通過扱い）
- 各ラウンドの新規指摘が「未知の事実の発見」である限りは健全な収束過程 — ラウンドを継続する

## Workflow スクリプト（任意 — 対応環境のみ）

このループ（独立レビュー → 台帳追記 → 反映 → 収束判定）は決定的なので、サブエージェントの
オーケストレーション機構（Claude Code の Workflow ツール等）が使える環境では、同梱の
`workflow.js`（このスキルと同じディレクトリ）で機械駆動してよい。利益は独立性ルールの
構造的な強制 — 「レビューアにはパスだけを渡す」「台帳を渡さない」「各ラウンドは新しい
レビューア」がプロンプトテンプレートに固定され、実行のたびの自制に依存しなくなる。

- args: `specDir`（specs/<feature>）/ `checkDocs`（check-docs.ts の実行プレフィックス）、
  任意で `reviewDocs`（観点に加える参照ドキュメントのパス配列）/
  `checkStateModel`（check-state-model.ts の実行プレフィックス — 状態モデルつき spec では
  **必ず渡す**。反映エージェントが反映後にチェッカーを exit 0 まで回すことを強制し、
  軸値未定義・Covers 未知値・単軸 dependent 等の形式エラーの持ち込みを工程内で検出する）/
  `maxRounds`。
  JSON オブジェクトで渡す（ハーネスによっては文字列で届くが、スクリプト側で JSON.parse に
  正規化される — 失敗時は `args に specDir / checkDocs が必要` エラー）
- 戻り値は `passed` / `escalated` / `needs_adjudication` / `blocked`。**裁定はスクリプト内で
  解決しない** — spec の意図に関わる指摘と「指摘の却下」は `needs_adjudication` で返る。
  minor の採否・却下の台帳記録・state.yaml の更新は本文の該当節に従ってメインが処理し、
  裁定後に再実行すれば台帳の Rd から次ラウンドとして再開される
- 使えない環境では本文の手順をそのまま手で回す。挙動は同一（スクリプトは本文の写しであり、
  乖離したら本文が正）

## 完了条件

- `review.status: passed`（または escalated 後のユーザー裁定で通過）
- 反映内容がすべて spec.md / design.md に書き込まれている（会話にだけ存在する修正はゼロ）
- 全指摘が台帳に記録され、処置列が埋まっている（`保留` はユーザー裁定済みの minor のみ —
  `check-docs.ts ledger` の exit 0 で機械確認する）
- state.yaml: `phase: spike`、`phases_done` に review 追加
