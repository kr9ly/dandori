# dandori-sketch — デザイン接地（スケッチ工程）

UI・デザイン変更を含むタスクで、spec（振る舞いの語彙）に乗らない**空間の語彙** —
「画面のどこを変え、どこを変えてはいけないか」— を確定する工程。
エージェントの UI 実装で多い事故は2つ: 指示外の領域を巻き添えで変えること、
「デザイン通りにした」という検証なしの自己申告。どちらも「変えてほしい部分以外」が
暗黙のままだと起きる。全領域を変更/不可侵に分類し、未分類を残さないことで構造的に防ぐ。

## いつ実施するか

ルーターのトリアージで `sketch.status: pending` が設定された UI タスクのみ。
spec 完了後、ground の前。判定基準: デザイン指示（Figma・デザインスペック・画像・
口頭の見た目指示）がある、または spec に visual/manual タグの B 行があり画面の
見た目が変わる。UI に触れないフィーチャーでは skipped。

## 手順

### 1. デザイン指示の収集と読み取り

`.dandori/resources.md` にデザイン指示の参照先（Figma エクスポート、デザインスペック
置き場等）があれば読む。指示から読み取った主張には検証状態マークを付ける —
dandori の教義どおり、指示に「書いてある」と「読んで補った」を混同しない:

| マーク | 意味 |
|--------|------|
| `[指示に明記]` | デザイン指示にそのまま書いてある。フレーム名・ページ等の所在を併記 |
| `[指示から推測]` | 指示にないが文脈から補った。根拠を併記 |

`[指示から推測]` は**全件ユーザー裁定に出す**。デザイン指示は往々にして正常系の
1画面しか描いていない — エラー時・空データ時・長文/多件数時の見た目、余白の数値、
既存画面とのなじませ方は推測になりがちで、ここが壁打ちの主戦場になる。

### 2. 画面インベントリと領域の全数分類

対象画面/コンポーネントを列挙し（アプリ内での到達手順つき）、各画面を領域に分割して
**全領域を `変更` か `不可侵` のどちらかに分類する**。未分類の領域を残さない —
状態モデルの全数走査と同じ思想で、「変えてほしい部分以外がどこか」を暗黙にしない
ことがこの工程の核。

- `変更` 領域は対応する spec の B 行（visual/manual）と**双方向に**突合する。
  対応 B 行のない変更領域はスコープ外の変更（spec に B 行を足すか、変更をやめる）。
  逆にどの領域にも対応しない visual/manual B 行は、インベントリに画面の抜けがある
- `不可侵` 領域に理由は不要（デフォルトが不可侵）。ただし変更領域に隣接し
  巻き添えリスクが高い領域には ⚠ を付け、gate の確認を重点化する

### 3. スケッチの提示（壁打ち）

変更マップを視覚化してユーザーに提示する。表現は2段構え:

- **簡単な変更**（1画面・領域数が少ない）: ASCII レイアウト図で足りる
- **画面構成が絡む変更**: HTML モックを書いて Artifact でプレビュー提示する。
  変更領域を強調色・不可侵領域をグレーで塗った領域マップと、変更後の想定レイアウト。
  自己完結 HTML（外部リソース参照不可）で書く

**スケッチは使い捨て・正は sketch.md の変更マップ**（spec のシナリオフロー図と同じ扱い —
図を正準成果物にすると drift の検査対象が増えるだけ）。壁打ちで得た裁定は
sketch.md に反映してから次へ進む。

### 4. 検証手段の割り当て

- `変更` 領域: 対応 B 行の visual/manual ゲートで検証する（gate 工程でトレース）
- `不可侵` 領域: 「変わっていないこと」の確認手段を宣言する。機械検証
  （スクリーンショット diff、レイアウト計測等）は**プロジェクトごとに最適なツールが
  違う** — `.dandori/resources.md` に宣言されたツールがあれば使い、なければこの時点で
  整備するかをユーザーと裁定する（基盤の有無は dandori-doctor の診断項目）。
  整備しない場合は gate での目視確認となり、確認手順（どの画面をどの状態で見るか）を
  ここで書いておく

目視確認が多くなりすぎる場合は spec の `manual` と同じく警告する — 最終ゲートの
検証コストがユーザーに跳ね返る。

### 5. 書き出しと fix

sketch.md 全体をユーザーに提示し、確認を得て fix。state.yaml を更新する
（`progress` は工程中も逐次更新 — 他工程と同じ）。

## sketch.md 形式（正準定義）

```markdown
# <feature> デザインスケッチ

## デザイン指示
- 参照先: <Figma フレーム / 画像パス / 指示の所在>
- 読み取り:
  - <要点> [指示に明記: <所在>]
  - <要点> [指示から推測: <根拠>] → 裁定: <承認 / 修正内容>

## 画面インベントリ
- <画面/コンポーネント名>: 到達手順 <アプリのどこからどう操作して出すか>

## 変更マップ

### <画面A>
| 領域 | 分類 | 変更内容 | 対応 B 行 |
|------|------|----------|-----------|
| 商品カードの価格表示 | 変更 | 定価に打ち消し線 + セール価格を併記 | B-3 |
| ヘッダー | 不可侵 | — | — |
| カード下部のボタン列 | 不可侵 ⚠ 変更領域に隣接 | — | — |

## 不可侵領域の検証手段
- <機械: `<コマンド>`（resources.md 宣言）/ 目視: <確認手順>>

## 未解決事項
- ...
```

## 後工程との接続

- **ground**: 不可侵領域を不変条件の候補源にする。各領域を描画するコードを特定し、
  画面の不変条件をコードの不変条件に接地する
- **plan**: UI に触れるマイルストーンのマニフェストに該当画面の変更マップを含める。
  **不可侵領域は常に全文**（不変条件と同じ部分開示の例外）
- **gate**: 変更領域は B 行トレースで検証し、不可侵領域は「非変化確認」行として
  trace.md に足す

## 完了条件

- sketch.md がユーザー確認済みで fix されている
- 全領域が変更/不可侵に分類され、変更領域と visual/manual B 行が双方向に対応している
- `[指示から推測]` が全件裁定済み（裁定できないものは「未解決事項」に明示的に残す）
- 不可侵領域の検証手段（機械 or 目視手順）が宣言されている
- state.yaml: `phase: ground`、`phases_done` に sketch 追加、`sketch.status: done`
