# dandori-ground — 前提の現実接地

spec.md の要件をコードベースの現実に接地させる工程。
「読んで整理する」ではなく「動かして確認する」。エージェントがコードを読んで立てた前提は
読み間違い・古いコードパスの誤認・実行時にしか分からない挙動でかなりの頻度で外れる。
机上の前提と検証済みの前提を明確に区別することがこの工程の存在意義。

## 手順

### 1. 調査

spec.md の各 B 行に対して:

- **土台**: そのまま利用できる既存実装はどこまであるか
- **改変**: 既存コードのどこを変更する必要があるか
- **新規**: 何をゼロから実装するか
- **不変条件**: 変えてはいけないもの（公開 API の互換性、既存テストが守る振る舞い、
  データ形式、他モジュールが依存する挙動）

コードベースの探索には利用可能なツールを使う（構造把握 → 該当箇所の精読）。
`.dandori/resources.md`（リソースマップ）があれば先に読み、アーキテクチャ・設計
ドキュメントを調査の起点に、規約類を不変条件の候補源にする。

sketch.md があるフィーチャー（UI タスク）では、変更マップの `不可侵` 領域を
不変条件の候補源にする — 各領域を描画するコード（コンポーネント・スタイル・
レイアウト定義）を特定し、「画面の不変条件」を「コードの不変条件」として
design.md に接地する（⚠ 付き領域は特定を省略しない）。

**状態モデルの ground 送り項目の消化**: spec.md に状態モデル
（`dandori-state-model` ブロック）がある場合、チェッカー
（`node <dandori-repo>/skills/dandori-spec/scripts/check-state-model.ts <spec.md>`）を実行し、
「ground 送り項目」（未裁定ペア・per-item 混在・base 推定 `?`）を確認項目に加える。
コードを読んで/動かして直交か依存かを確定し、**結果をモデルに書き戻す**
（`ground:` エントリを `orthogonal`（理由つき）か `dependent`（+ 交点をカバーする B 行）に
昇格、`?` を外す）。書き戻し後にチェッカーを再実行し、新たな指摘（昇格した依存交点の
未カバー等）が出たら B 行追加として spec に反映する。詳細: `docs/appendix-state-model.md`

**軸対応の接地**（状態モデルがある spec のみ — 無ければこの項は丸ごとスキップ）:
spec の各軸について、それを実現するコード構造（型 / パラメータ / 分岐点 / leaf モジュール）を
特定し、design.md の「軸対応」節に**全軸分**を書く（`- <軸キー>: <対応物> <判定>` — 軸キーは
状態モデルの axes のキーを使う）。判定語彙は3つ:

| 判定 | 意味 |
|------|------|
| `[1箇所]` | 軸が単一のコード構造に対応する |
| `[散在: <理由>]` | 消費点が複数だが正当 — 理由には dependent 宣言等、相互作用の実在根拠を書く |
| `⚠ <発見と行き先>` | 正当化できない散在 = 設計への警告 |

- **dependent 宣言だけが散在の免罪符** — 裁定済みの相互作用は複数箇所への波及を正当化できる。
  宣言なき散在は smell として ⚠ を付け、行き先（新規実装の切り方の見直し / リスクランキング /
  ユーザー裁定）を確定してから先へ進む。ここがテスト軸の因数分解を設計に還流する接点 —
  B 行単位の対応（改変箇所・新規実装の既存欄）は交点を見るが、軸の散在は見えない
- dependent 交点には明示的な置き場（どの関数・分岐がその交点挙動を担うか）を軸エントリ内に書く
- `partition_of:` 宣言のある軸（appendix-state-model「型対応の宣言」）は、対応物の特定に加えて
  **全域性**を実コードで突合する — 型・値集合の全メンバーがちょうど 1 つの軸値クラスに落ち、
  空クラスがないこと。未分類メンバー（どのクラスの Given でもテストされない値）・空クラス
  （到達不能 Given）は spec への還流としてユーザー裁定に上げる
- 直交宣言の reason が「フェーズ P_n は X 列のみ触る」型（**列所有権の主張**）の場合、それは
  実装のモジュール境界への構造要求 — どの境界がその独占を守るかを対応物の記述に含める。
  直交宣言は base-choice テスト削減の正当化根拠であり、実装が所有権を破ると削減の前提が黙って崩れる

**Gate 乖離行の消化**（乖離マーク `<現状>→<希望>` がある spec のみ — 無ければスキップ）:
spec の乖離行（`check-docs.ts spec` が別枠で列挙する）ごとに、希望の固定単位を可能にする
seam（切り出し・分離点）を design に置けるか検討する:

- 置ける → 新規実装/改変箇所（と軸対応）に seam を設計し、spec のタグを希望側に昇格する
  （`e2e→unit（...）` → `unit（<seam> で固定）`）
- 置けない → 阻害要因が構造的に解消不能である理由を注記に足して乖離のまま残す
  （黙認不可 — 理由の薄い残存は review 工程の批評対象になる）

**共有状態への影響の宣言**（状態マップ — `.dandori/map/states.md` に
`dandori-state-map` ブロック — があるプロジェクトのみ。無ければこの項は丸ごとスキップ）:

1. 今回の改変・新規実装が**書く**共有状態（マップの行）を特定し、design.md に宣言する
2. 書く状態ごとに影響導出クエリで既存 spec の回帰候補を列挙し、design.md に記録する:
   `node <dandori-repo>/skills/dandori-spec/scripts/check-state-model.ts --impact <状態ID> <既存spec...>`
3. 区分による設計チェック: **borrowed な状態に書こうとしていたら即違反** — 設計の
   見直しか区分の再裁定（マップ更新）へ。shared なら外部起因の遷移と失敗時補償が
   spec で裁定されているか確認する
4. 今回のフィーチャーが新しい書き手/読み手になる場合はその旨を design.md に記録する
   （マップへの反映は gate クローズの survey update で行う）

### 2. 実行検証

§1 で「土台」と見なした実装は、**可能な限り実際に動かして**期待どおりの挙動を確認する:

- 既存テストがあれば実行する
- なければ使い捨てスクリプトで対象関数・API を直接呼び、入出力を観測する
- アプリとして起動可能なら起動して該当パスを通す

すべての前提に以下のいずれかのマークを付ける:

| マーク | 意味 |
|--------|------|
| `[実行検証済]` | 実際に動かして確認した。**実行コマンド + 観測結果の要点**を併記 |
| `[読解のみ]` | コードを読んだだけ。実行できない理由を併記 |

「実行できない理由」が単なる手間なら実行すること。環境が必要・破壊的・外部依存など
正当な理由がある場合のみ `[読解のみ]` を許す。

`[実行検証済]` の併記は**第三者が再実行・反証できる形式**で書く — 「テスト通過」だけでは
検証の主張であって証拠ではない。何を実行し（コマンド）、何を観測したか（期待値と一致した
出力の要点）まで書いて初めて、review 工程のレビューアがマークの正当性を監査できる。

### 3. リスクランキング

`[読解のみ]` の前提を「外れたときの被害 × 外れる可能性」で降順に並べる。
上位の前提が dandori-spike の検証対象になる。全前提が `[実行検証済]` なら
spike はスキップ可能 — その旨を design.md に明記する。

### 4. 書き出し

design.md を作成したら形式検査を実行し、exit 0 にする:
`node <dandori-repo>/skills/dandori/scripts/check-docs.ts design <spec.md> <design.md>`
（必須セクション・検証マークの有無と証拠形式・B 行対応の突合 — 未対応 B 行や
幽霊参照を機械検出する）。その上で state.yaml を更新する。この工程にユーザー壁打ちは不要 — 
成果物は次工程の dandori-review が第三者検証する。

## design.md 形式（正準定義）

```markdown
# <feature> 実装設計

## 土台（利用する既存実装）
- <モジュール/関数>: <何に使うか> [実行検証済: `npm test -- xx.test` 12 passed — 戻り値が B-2 の期待形式と一致]
- <モジュール/関数>: <何に使うか> [読解のみ: 外部 API のため]

## 改変箇所
- <ファイル/関数>: <何をどう変えるか> → 影響する B 行: B-3, B-7

## 新規実装
- <コンポーネント>: <責務> → 実現する B 行: B-1, B-2

## 不変条件（変えてはいけないもの）
- ...

## 軸対応（状態モデルがある spec のみ）
- payment: PaymentMethod 型 + authorize() のディスパッチ [1箇所]
- coupon, promo: 検証 leaf + 対象判定 leaf + タイブレーク [散在: dependent {coupon, promo} — 交点の置き場は getDiscountInfoForItem]
- fd: ⚠ 最終状態が3書き込みパスの直列合成でしか定義されない — applyDiscountPhases に切り出して交点を単体固定（発見ログ参照）

## 共有状態への影響（状態マップがあるプロジェクトのみ）
- 書く状態: <状態ID>（<区分>/<storage>）→ 影響 spec: <spec> B-n, B-m（--impact 出力）
- 読む状態: <状態ID> — <staleness/sync の前提>
- マップ更新の要否: <新しい書き手/読み手になるか>

## リスクランキング（読解のみ前提の降順）
1. <前提>: 被害=<高/中/低> 可能性=<高/中/低> — spike 候補
2. ...

## 発見ログ
<!-- 以降の全工程（spike, impl）で発見した仕様と現実のズレをここに追記する。 -->
<!-- 形式: - [日付/工程] 発見内容 → 対応（design修正 / spec要修正 / 影響なし） -->
```

## 完了条件

- design.md が存在し、全前提にマークが付いている
- 改変箇所・新規実装が spec.md の B 行と対応付いている（どの B 行にも土台/改変/新規の
  対応がない場合、spec か調査のどちらかに穴がある — 解消してから完了とする）
- 上記2点を形式検査（`check-docs.ts design`）の exit 0 で機械確認している
- **spec.md の「未解決事項」を走査し、ground の調査で解消したものを spec.md に書き戻している**
  （取り消し線 + 解消内容 + 根拠の所在。design.md に書くだけでは spec 側が未解決のまま残る）
- 状態モデルがある spec: ground 送り項目を消化してモデルに書き戻し、チェッカーが
  exit 0 かつ ground 送り項目ゼロ（消化しきれないものはユーザー裁定の上で残してよいが黙認は不可）
- 状態モデルがある spec: design.md に「軸対応」節があり全軸を判定つきでカバーしている
  （`check-docs.ts design` の D5 が機械検査）。⚠ が残る場合はユーザー裁定を経る（黙認不可）
- Gate 乖離行がある spec: 全乖離行が消化されている（タグ昇格、または解消不能理由つきの残存）
- 状態マップがあるプロジェクト: design.md に「共有状態への影響」節があり、書く状態の
  宣言と影響 spec の列挙が済んでいる（書く共有状態がゼロならその旨を明記）
- state.yaml: `phase: review`、`phases_done` に ground 追加
