# dandori-spec — 仕様策定（壁打ち工程）

要件・操作バリエーション・エラーパス・品質ゲートを**一体で**定義する。
「何が満たされればゴールか」の答えはそのまま「何で検証するか」に変換できるはずであり、
分離すると検証不能な美辞麗句がゴール定義に紛れ込む。ここが dandori 全体で唯一の重い壁打ち工程。

## 進め方の原則

- **たたき台を先に出す**。白紙の質問をユーザーに投げない。元ドキュメント・会話・コードから叩き台を作り、フィードバックをもらうループで回す
- **合意した内容は都度 spec.md に書き出す**。§5 の fix を待たず、各段の壁打ちが終わるたびに spec.md へ追記する。冒頭に `status: draft` と各セクションの確定/未確定を明記し、fix 時に外す。セッション断で壁打ち結果を失わないため、また会話の中にしか存在しない合意を作らないため
- **逐次追記でも正準フォーマットを崩さない**。追記は既存セクションの中身を更新する形で行い、同名セクションを新設しない（「未解決事項」が2つできる等）。書き出し後にフォーマット検査（`node <dandori-repo>/skills/dandori/scripts/check-docs.ts spec <spec.md>`）を実行し、セクション構成・B 行フィールド・ゲートタグが正準定義と一致することを確認する
- **spec.md を書くたびに state.yaml も更新する**。`progress`（工程内の現在地一行、例: `§3 異常系マトリクスの壁打ち中`）と `updated` を書き換え、state を常に現状と一致させる
- **一度に全部聞かない**。§1→§4 を段階ごとに壁打ちする。ただし各段の往復は1〜2回で切り上げ、疑問は「未解決事項」に積んで先へ進む
- **議題のシナリオはフロー図で出す**。§2〜3 で交点・異常系・連鎖を検討するときは、対象シナリオ単位の Mermaid フローチャートをたたき台にする（「シナリオフロー図」節参照）
- **検証不能な要件を書かない**。「使いやすい」「高速」が出たら、観測可能な条件（操作列と期待結果、数値閾値）への言い換えをその場で提案する

## 手順

### 1. ゴール定義

元ドキュメント（要件メモ、issue、会話ログ）があれば読み込む。
`.dandori/resources.md`（リソースマップ）があれば読み、ドメイン仕様・既存要件
ドキュメントの所在を確認して壁打ちの材料にする。その上で:

- この実装で**何を実現するべきか**（1〜3文）
- **何が満たされればゴールと言えるか**（観測可能な条件の箇条書き）
- **スコープ外**（やらないことの明示 — 後工程のスコープクリープを防ぐ防波堤）

たたき台を提示し、壁打ちで確定する。

### 2. 正常系 — 操作・入力のバリエーション

ユーザー操作や API 入力のバリエーションを列挙する。操作シナリオとして
「どのような操作の結果、何が起きるべきか」をシミュレーションする。
たたき台を提示し、抜けをユーザーに指摘してもらう。

バリエーションが多い場合（目安: 10 を超える）は、独立した**軸**（例: 決済方法 /
割引種別 / 在庫状態）に分けて整理する。軸に分けたら、**軸同士の関係を必ず明記する**:

- **完全直交**: どの組み合わせも独立に成立する。基準シナリオ + 各軸1変化のテストで足りる
- **依存関係あり**: ある軸の値が別の軸の挙動を変える・前提になる・特定の値に限定される。
  該当する交点を列挙し、明示的にテスト対象へ含める

直交と宣言したものだけが組み合わせテストを省略できる。関係を書かずに軸へ分けると、
交点の挙動（例: 決済 pending 時だけ通知がスキップされる）が §3 でも §4 でも
拾われずに漏れる。

### 3. 異常系 — 状態変数マトリクス

結果に影響する**変数・状態**を列挙する（例: ログイン有無、ネットワーク状態、事前操作の有無、
データの空/満杯、権限）。次に **各変数 × §2 の正常系シナリオ** のマトリクスを機械的に走査し、
挙動が変わる交点をすべて拾う。これがエラーハンドリングパスの洗い出しになる。

直感で「ありそうなエラー」を挙げるのではなく、マトリクスから導出すること。
エージェントはエラーパスの列挙を体系的にサボる — この工程を独立させているのはそのため。

交点が多い場合（目安: 10 を超える）は、§2 の軸分けと同様に**対象の構造に即した軸で
分割して整理する**。軸は対象から発見するものであり、決め打ちしない。例:

- **処理フェーズ**: 入力検証 → 排他 → 前提確認 → 外部照合 → 確定、のような実行順序。
  「どこまで進んで失敗したか」がロールバック要件を規定する場合に特に有効
- **リソース種別 / 外部依存先**: 失敗源ごとの分類
- **権限・認証状態**: 主体ごとの分類

フェーズで分けた場合は、**フェーズ横断の関心事**（ロールバック、補償処理、通知）を
独立した横断グループとして最後に置く。「失敗タイミング × 確保済みリソース」で挙動が
変わるなら、その組み合わせを個別の交点として拾う。

**状態モデルへの格上げ**: §2 で軸が 3 つ以上になった場合、または本節で
フェーズ×リソース型の横断が出た場合は、軸・直交宣言・交点を機械可読な状態モデル
（spec.md 内の `dandori-state-model` fenced block + 各 B 行の `Covers:` フィールド）に
書き起こし、B 行の抜け（未カバー値・空虚に真な交点主張・未裁定ペア・到達不能 Given）を
チェッカー（`node <dandori-repo>/skills/dandori-spec/scripts/check-state-model.ts <spec.md>`）で
静的検査する。
§4 の B 行執筆と**同時に** Covers を書く（後付けの遡及は恣意性が高い — 実験 4 で実証）。
書き方・チェッカー・ゲート運用はリポジトリの `docs/appendix-state-model.md` を参照。

状態マップ（`.dandori/map/states.md` に `dandori-state-map` ブロック — 無いプロジェクト
では無視してよい）がある場合、共有状態に触れる軸には `ref: <状態ID>` を付けて
canonical 値 ID を使う。触れた共有状態がマップに無ければ行の追加をユーザーに提案する
（サイレントに追加しない）。詳細: `docs/appendix-state-map.md`

チェッカーの検査 3（未分類ペア）の指摘は壁打ちの議題リストとしてそのままユーザーに出す。
**全数性を満たすためだけの吟味なし直交宣言をしない** — 直交の reason には spec 本文から
言える根拠を書き、書けないペアは `ground:` 送りを宣言する（裁定のふりの量産はチェッカーの
意図の逆転）。

### 4. 振る舞い仕様化 + ゲートタグ付け

§1〜3 の結果を Given/When/Then 形式に落とす。**各行に品質ゲート種別をタグ付けする**:

| タグ | 意味 |
|------|------|
| `unit` | ユニットテストで担保 |
| `e2e` | 結合・E2E テストで担保 |
| `visual` | レイアウト・表示の確認で担保 |
| `manual` | ユーザー目視・手動操作で担保 |
| `formal` | プロパティベーステスト・形式検証で担保（拡張タグ） |

**`formal` の発動条件は主張の形**: Then が「どんな入力でも」「どの実行順でも」という
全称主張で、`unit` を付けようとして「代表入力を数個試すだけでは不安」と感じたら
`formal` を検討する。詳細はリポジトリの `docs/appendix-formal.md` を参照。

`manual` が多すぎる場合は警告する — 最終ゲートの検証コストがユーザーに跳ね返る。
自動化可能な形への仕様の言い換えを提案すること。

### 5. 確定

spec.md 全体をユーザーに提示し、最終確認を得て fix。state.yaml を更新する。

**フォーマット検査グリーン（`check-docs.ts spec <spec.md>` が exit 0）が fix の前提条件**。
必須セクション・B 行フィールド・ゲートタグ語彙・B-ID の重複/欠番を機械検査する。

状態モデルを書いた spec では、加えて**状態モデルチェッカーグリーン（exit 0）が fix の前提条件**。
指摘が残ったままの fix は不可 — 裁定不能なものは `ground:` / `?` で ground 送りを
機械可読に宣言すれば通る（ground 送り項目は ground 工程の確認項目リストになる）。

## シナリオフロー図 — 壁打ちの視覚たたき台

壁打ちの議題を1件検討するとき（§3 の異常系交点の挙動決定、依存交点の裁定、
fix 前の B 行の Then 検証）、対象シナリオを **1 本のパス**として Mermaid
フローチャートにして提示する。モデル全体の関係図は描かない — 関係の整合は
チェッカーの領分で、人間の意思決定に効くのは「シナリオを頭で歩く」
シミュレーションの外部化だけ（実験 2026-07-08: `experiments/state-model/per-scenario-sample.md`）。

- **1 シナリオ = 1 図 = 1 議題**。ノード 10 個前後が上限。基準シナリオと同じ
  ステップは「（基準どおり）」の1ノードに畳む
- **視覚語彙（固定）**:
  - 太枠（`stroke-width:3px`）= このシナリオで基準と挙動が変わるステップ
  - 菱形 = 依存交点の条件分岐（対応する dependent 宣言・S 番号を注記）
  - 赤破線 + ❓ = 未裁定。そのまま壁打ちの議題になる
  - ラベル内の HTML 強調タグ（`<b>` 等）は使わない — 枠スタイルと二重符号化になる
- **問いの型**: 「太枠の集合は B 行の Then の記述と一致するか（言い漏れ検出）」/
  「図の末端が裸で終わる場所 = spec の記述が切れている場所（後続の欠落検出）」/
  「❓ は今裁定するか ground 送りか」
- **図は使い捨て**。spec.md には残さない — 正は B 行 + 状態モデルであり、図を
  正準成果物にすると drift の検査対象が増えるだけ。一時ファイルに書いて
  Mermaid 対応プレビュー（Zed / VS Code / GitHub）で開いてもらう提示が読みやすい

## spec.md 形式（正準定義）

```markdown
# <feature> 仕様

## ゴール
<1〜3文> + 満たすべき観測可能な条件の箇条書き

## スコープ外
- ...

## 振る舞い仕様

### B-1: <タイトル>
- Given: <前提状態>
- When: <操作・入力>
- Then: <観測可能な期待結果>
- Gate: unit

### B-2: ...（異常系も同形式。Given に異常状態を置く）

## 未解決事項
- ...（壁打ちで確定しなかった点。plan までに解消するか、明示的にスコープ外へ）
```

B-n の ID は以降の全工程（plan のマイルストーン対応、gate のトレース表）から参照される。
**fix 後の改番は禁止**。追加は末尾、削除は取り消し線 + 理由。
fix 済み spec を再編集したときは改番検知を機械で掛ける:
`git show HEAD:<spec.md> > /tmp/base.md` で旧版を取り出し、
`node <dandori-repo>/skills/dandori/scripts/check-docs.ts spec <spec.md> --baseline /tmp/base.md`
（タイトルのすり替え・取り消し線なしの削除・末尾以外への挿入を検出する）。

## 短縮コース時

短縮コース（state.yaml の `course: short`）では §1〜4 を1パスで通し、壁打ちは
最後の全体確認1回に圧縮する。ただし §3 の状態変数マトリクスは省略しない — 
短縮コースで事故るのは大抵ここ。完了後は dandori-plan / dandori-ground を飛ばし
dandori-impl へ直行する（マイルストーン1個の plan を暗黙生成する扱い）。

## 完了条件

- spec.md がユーザー確認済みで fix されている
- 全 B 行にゲートタグが付いている
- フォーマット検査（`check-docs.ts spec`）が exit 0
- 状態モデルを書いた spec: 状態モデルチェッカーが exit 0（ground 送り項目は残ってよい）
- state.yaml: `phase: ground`（短縮コースは `phase: impl`）、`phases_done` に spec 追加
