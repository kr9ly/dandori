# dandori-spec — 仕様策定（壁打ち工程）

要件・操作バリエーション・エラーパス・品質ゲートを**一体で**定義する。
「何が満たされればゴールか」の答えはそのまま「何で検証するか」に変換できるはずであり、
分離すると検証不能な美辞麗句がゴール定義に紛れ込む。ここが dandori 全体で唯一の重い壁打ち工程。

## 進め方の原則

- **たたき台を先に出す**。白紙の質問をユーザーに投げない。元ドキュメント・会話・コードから叩き台を作り、フィードバックをもらうループで回す
- **合意した内容は都度 spec.md に書き出す**。§5 の fix を待たず、各段の壁打ちが終わるたびに spec.md へ追記する。冒頭に `status: draft` と各セクションの確定/未確定を明記し、fix 時に外す。セッション断で壁打ち結果を失わないため、また会話の中にしか存在しない合意を作らないため
- **逐次追記でも正準フォーマットを崩さない**。追記は既存セクションの中身を更新する形で行い、同名セクションを新設しない（「未解決事項」が2つできる等）。書き出し後に `grep "^## "` でセクション構成が正準定義と一致することを確認する
- **spec.md を書くたびに state.yaml も更新する**。`progress`（工程内の現在地一行、例: `§3 異常系マトリクスの壁打ち中`）と `updated` を書き換え、state を常に現状と一致させる
- **一度に全部聞かない**。§1→§4 を段階ごとに壁打ちする。ただし各段の往復は1〜2回で切り上げ、疑問は「未解決事項」に積んで先へ進む
- **検証不能な要件を書かない**。「使いやすい」「高速」が出たら、観測可能な条件（操作列と期待結果、数値閾値）への言い換えをその場で提案する

## 手順

### 1. ゴール定義

元ドキュメント（要件メモ、issue、会話ログ）があれば読み込む。その上で:

- この実装で**何を実現するべきか**（1〜3文）
- **何が満たされればゴールと言えるか**（観測可能な条件の箇条書き）
- **スコープ外**（やらないことの明示 — 後工程のスコープクリープを防ぐ防波堤）

たたき台を提示し、壁打ちで確定する。

### 2. 正常系 — 操作・入力のバリエーション

ユーザー操作や API 入力のバリエーションを列挙する。操作シナリオとして
「どのような操作の結果、何が起きるべきか」をシミュレーションする。
たたき台を提示し、抜けをユーザーに指摘してもらう。

### 3. 異常系 — 状態変数マトリクス

結果に影響する**変数・状態**を列挙する（例: ログイン有無、ネットワーク状態、事前操作の有無、
データの空/満杯、権限）。次に **各変数 × §2 の正常系シナリオ** のマトリクスを機械的に走査し、
挙動が変わる交点をすべて拾う。これがエラーハンドリングパスの洗い出しになる。

直感で「ありそうなエラー」を挙げるのではなく、マトリクスから導出すること。
エージェントはエラーパスの列挙を体系的にサボる — この工程を独立させているのはそのため。

### 4. 振る舞い仕様化 + ゲートタグ付け

§1〜3 の結果を Given/When/Then 形式に落とす。**各行に品質ゲート種別をタグ付けする**:

| タグ | 意味 |
|------|------|
| `unit` | ユニットテストで担保 |
| `e2e` | 結合・E2E テストで担保 |
| `visual` | レイアウト・表示の確認で担保 |
| `manual` | ユーザー目視・手動操作で担保 |
| `formal` | プロパティベーステスト・形式検証で担保（拡張タグ） |

**`formal` の発動条件は主張の形**: Then が「どんな入力でも」「どの実行順でも」という
全称主張で、`unit` を付けようとして「代表入力を数個試すだけでは不安」と感じたら
`formal` を検討する。詳細はリポジトリの `docs/appendix-formal.md` を参照。

`manual` が多すぎる場合は警告する — 最終ゲートの検証コストがユーザーに跳ね返る。
自動化可能な形への仕様の言い換えを提案すること。

### 5. 確定

spec.md 全体をユーザーに提示し、最終確認を得て fix。state.yaml を更新する。

## spec.md 形式（正準定義）

```markdown
# <feature> 仕様

## ゴール
<1〜3文> + 満たすべき観測可能な条件の箇条書き

## スコープ外
- ...

## 振る舞い仕様

### B-1: <タイトル>
- Given: <前提状態>
- When: <操作・入力>
- Then: <観測可能な期待結果>
- Gate: unit

### B-2: ...（異常系も同形式。Given に異常状態を置く）

## 未解決事項
- ...（壁打ちで確定しなかった点。plan までに解消するか、明示的にスコープ外へ）
```

B-n の ID は以降の全工程（plan のマイルストーン対応、gate のトレース表）から参照される。
**fix 後の改番は禁止**。追加は末尾、削除は取り消し線 + 理由。

## 短縮コース時

短縮コース（state.yaml の `course: short`）では §1〜4 を1パスで通し、壁打ちは
最後の全体確認1回に圧縮する。ただし §3 の状態変数マトリクスは省略しない — 
短縮コースで事故るのは大抵ここ。完了後は dandori-plan / dandori-ground を飛ばし
dandori-impl へ直行する（マイルストーン1個の plan を暗黙生成する扱い）。

## 完了条件

- spec.md がユーザー確認済みで fix されている
- 全 B 行にゲートタグが付いている
- state.yaml: `phase: ground`（短縮コースは `phase: impl`）、`phases_done` に spec 追加
