# dandori-doctor — 前提条件診断

dandori の各工程は機械検査を前提に設計されているが、その基盤（テストコマンド、
lint/format、ミューテーションテスト等）が揃っているかはプロジェクト次第。
この工程外スキルは前提条件を**実行して**診断し、整備の要/不要をユーザーに提案する。
survey と同じ足場系で、フィーチャーに依存しない（プロジェクト単位で一度やれば足りる）。

**診断まで**がスコープ。整備作業そのものはやらない — 提案をユーザーが裁定した後、
通常の作業（または dandori の1フィーチャー）として実施する。

## いつ使うか

- プロジェクトへの dandori 初適用の前
- 久しぶりに再適用するとき（基盤が腐っていないかの再確認）
- dandori 側に新しい機械検査が追加されたとき（例: codereview のミューテーションテスト）

## 診断項目

| 項目 | 確認内容 | 影響を受ける工程 |
|------|----------|------------------|
| チェッカー実行環境 | `node <dandori-repo>/skills/dandori/scripts/check-docs.ts`（引数なし）が usage を表示して終了するか — Node が TS をフラグなしで素実行できるバージョン（22.18+ / 23.6+）か | spec / plan / review / codereview / gate / survey（機械検査全般） |
| リソースマップ | `.dandori/resources.md` の有無と、下記コマンド類の宣言状況 | 全工程 |
| unit / e2e | 正準テストコマンドが存在し、現行コードで緑か | impl / codereview / refine / gate |
| lint / format | 正準コマンドの有無 | refine（機械検査先行） |
| ミューテーションテスト | 言語に対応するツール（TS: Stryker, JVM: pitest, Python: mutmut, Rust: cargo-mutants 等）の導入状況と diff スコープ実行の可否 | codereview（機械検査先行） |
| B-ID 命名規約 | 既存テストにゲートタグ（B-ID）入りの命名が浸透しているか grep で確認 | gate（機械トレース） |
| デザイン指示の参照手段 | Figma エクスポート・デザインスペック・画像等を、エージェントが読める形で取得できるか（UI を持つプロジェクトのみ） | sketch |
| UI 検証手段 | 対象プラットフォームで画面の状態を観測する手段（スクリーンショット取得、レイアウト計測、プレビュー起動）が resources.md に宣言され、実際に1回動くか。最適なツールはプラットフォームごとに違う（Web / モバイル / デスクトップ） | sketch / impl / gate（visual ゲートと不可侵領域の非変化確認） |
| 足場マップ | `.dandori/map/` の有無・鮮度（詳細検証は dandori-survey verify に委譲） | 全工程 |

## 診断の原則 — 実行して確かめる

dandori の教義どおり、「読解のみ」の診断をしない:

- コマンドは**実際に叩いて**成否を見る（設定ファイルにあるだけの壊れたコマンドを
  「ある」と報告しない）
- ただし重い実行はしない。テストは実行するが、ミューテーションテストは
  ツールの存在確認 + 最小対象での動作確認（1ファイル等）まで
- 確認できなかった項目は「未確認」として明示する。「たぶんある」を書かない

## 出力 — 診断表と整備提案

```markdown
| 項目 | 状態 | 整備 | 理由 |
|------|------|------|------|
| unit | ✅ `make test` 緑 | 不要 | — |
| lint/format | ❌ なし | 要 | refine の機械検査先行が空振りし、書き味指摘が全部エージェント行き |
| ミューテーションテスト | ❌ なし | 不要 | 短縮コース中心の運用でテストも遅い。費用対効果が薄い |
| B-ID 命名 | ⚠️ 既存テストの2割のみ | 要 | gate の機械トレースが grep で対応テストを見つけられない |
```

**「不要」も積極的に提案する**。全項目を揃えるのが目的ではない — プロジェクトの
テスト速度、言語エコシステムの成熟度、運用コース（短縮のみならミューテーション
テストの費用対効果は薄い）を踏まえて裁定材料を出す。判断はユーザーに委ねる。

「要」と裁定された項目は、`.dandori/resources.md` への正準コマンド追記までを
ゴールに含めて提案する（コマンドが存在しても宣言されていなければ工程からは見えない）。
