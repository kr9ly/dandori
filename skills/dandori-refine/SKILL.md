# dandori-refine — リファインメント

codereview と gate の間で、**正しさではなく書き味**を整える工程。
codereview とは検証の経済が違う — 正当性指摘は真偽が争点なので反証フェーズが要るが、
リファインメント提案の検証は「適用してゲートが緑のままか」そのもの。
全ゲート緑という資産を安全網に、振る舞いを変えない改善だけを適用する。

## 入口条件

- `codereview.status: passed`（または escalated 後のユーザー裁定で通過）。
  短縮コースではこの工程はデフォルトでスキップする（`refine.status: skipped`）
- 対象 diff は codereview と同じ（working tree の全変更 + untracked）

## 機械検査を先に

`.dandori/resources.md` に formatter / linter の正準コマンドがあれば**レーン起動前に実行**し、
指摘を機械的に潰す。エージェントレビューは機械が捕まえられない残余に限定する —
formatter が直せるものにレビューレーンを使わない。
正準コマンドが無ければこのステップはスキップする（導入の提案はしてよいが、
この工程の中で導入作業はしない — スコープ外の変更を diff に混ぜない）。

## 3レーン並列ディスパッチ

各レーンの問いが狭いため、モデルは Sonnet クラスで足りる。
spec.md / design.md は**どのレーンにも渡さない** — この工程の照合先は仕様ではなく
既存コードの現物。仕様との適合は impl / gate が、テストの忠実度は codereview が担う。

| レーン | 照合先 | 問い |
|--------|--------|------|
| イディオム一致 | 改変ファイルの周辺既存コード | 命名・コメント密度・エラーハンドリングの流儀が周囲と揃っているか |
| 再発明・簡素化 | コードベース全体 | 既存ヘルパー・ユーティリティの再実装はないか。デッドコード、不要な抽象、防衛過剰な実装はないか |
| テスト品質 | 既存テストの流儀 | 過剰モック、脆い assertion、重複フィクスチャ、テストとして読みにくい構造はないか |

再発明レーンを独立させるのは、実装エージェントがコンテキストマニフェスト外の
既存コードを知らずに再実装する失敗が構造的に起きやすいため。

共通テンプレート（レーン固有の照合先と問いを差し替える）:

```
あなたは実装コードのリファインメントレビューアです。コードベースへの読み取りアクセスがあります。
レビュー対象の差分: <取得コマンド>
<レーン固有: 照合先 + 問い>

ルール:
- 提案には必ず接地を付けること — 既存コードの実例（ファイル:行）、または
  resources.md / .dandori/map/ 記載の規約のどちらか。接地のない好みの表明は報告しない
- 振る舞いを変える提案はしない。仕様・公開 API・永続データ形式に触れるものは対象外
- 各提案に、対象箇所（ファイル:行）と具体的な変更内容を付けること
```

## 採否と適用

反証フェーズは置かない。代わりに次のフィルタと検証で決着させる:

1. **接地フィルタ** — 既存コードの実例・規約への接地がない提案は棄却
2. **振る舞い保存フィルタ** — spec の振る舞い・公開 API・永続データ形式に触れる提案は棄却。
   それはリファインメントではなく変更 — 必要なら design.md の発見ログに記録して
   正規のフロー（ユーザー裁定）に乗せる
3. **適用** — 生き残った提案を適用する（量が多ければ worker にディスパッチしてよい）
4. **ゲート再実行** — 全ゲート緑なら採用確定。赤になったら該当適用を revert して棄却

## 収束設計 — 1ラウンド限り

この工程には反証のような客観的な決着装置がないため、ループにすると命名談義で発散する。

- **1ラウンドのみ**。適用後の再レビューはしない
- 提案は適用 or 棄却の**二択**。「議論して磨く」をしない
- 採否に迷ったら棄却（保守側に倒す）。棄却にユーザー裁定は要らない

## Workflow スクリプト（任意 — 対応環境のみ）

この工程のフロー（機械検査 → 3レーン → 採否フィルタ → 適用 + ゲート再実行、1ラウンド固定）は
決定的なので、サブエージェントのオーケストレーション機構（Claude Code の Workflow ツール等）が
使える環境では、同梱の `workflow.js`（このスキルと同じディレクトリ）で機械駆動してよい。
利益は情報隔離の構造的な強制 — 「spec.md / design.md をどのレーンにも渡さない」が
プロンプトテンプレートに固定される。採否フィルタは適用エージェントから独立させてある
（適用者が自分の仕事量を減らす方向にフィルタを甘くする利益相反を避ける）。

- args: `diffCommand` / `gates`（正準コマンド配列）、任意で `mechCommands`（formatter/linter）/
  `resources` / `mapDir` / `workRoot`。JSON オブジェクトで渡す（文字列で届く環境でもスクリプト側で
  JSON.parse に正規化される）
- `gates` には**全量スイート**の正準コマンドを渡す（フィーチャースコープに絞らない）。
  この工程の適用は既存部品の再利用による import 追加を含み、影響がフィーチャー外に
  波及し得る（barrel 循環 import で無関係スイートが収集エラーになった実績あり）。
  フィーチャースコープのゲートでは素通りする
- `workRoot`: コードの作業ルート。worktree 並列レーン等、コードがセッションの主作業ディレクトリと
  別の場所にあるとき指定する（サブエージェントは主作業ディレクトリで動くため、指定しないと
  相対パスのゲートが失敗し、他 worktree を誤読・誤編集するリスクがある）。機械検査・レビュー・
  採否・適用のプロンプトに「コードの読み書きとゲート実行を workRoot 内に閉じる」指示が注入される
- 戻り値は `done`（適用・棄却一覧つき）/ `gate_red` / `blocked`（入口ゲート赤）。振る舞い変更として棄却された提案は
  `behaviorChanges` で返る — design.md の発見ログへの記録と、採否一覧のユーザー提示、
  state.yaml の更新はメインエージェントが行う
- 使えない環境では本文の手順をそのまま手で回す。挙動は同一（スクリプトは本文の写しであり、
  乖離したら本文が正）

## 完了条件

- 採用分が working tree に反映され、全ゲート緑
- 適用 / 棄却の一覧（棄却理由つき）をユーザーに提示済み
- state.yaml: `refine.status: done`、`applied` / `rejected` 件数、`phase: gate`

map への知見還流はこの工程では行わない — gate クローズ後の dandori-survey promote が
担う（既存経路）。
