# dandori-cleanup — プロセス言及の除去とクローズ

B-ID や spec/design への参照は**プロセスの追跡装置**であって、プロダクトコードの語彙ではない。
gate のコミットで「トレース可能なスナップショット」が履歴に固定された後、この工程が
プロセス由来の言及を取り除き、フィーチャーをクローズする。dandori を使っていない読者が
テスト名やコメントを読んで、前提知識なしに意味が取れる状態がゴール。

cleanup は不可逆な「店じまい」— B-ID を剥がすと以降の再 gate は機械突合が効かなくなる
（git 履歴経由の差分トレースに落ちる）。だから **gate 直後ではなく、feedback 工程で
「実装は完全に fix — 改訂はもう来ない」の裁定が出てから**実行する。

## 入口条件

- state.yaml: `phase: cleanup`（gate 通過・コミット済み + feedback で完全 fix の裁定済み）
- trace.md が存在する — B 行↔テストの対応の正であり、この工程の作業リスト。
  処分はこの工程の最後（§5）まで行わない。改訂サイクルを回したフィーチャーでは
  最後の gate の trace.md が全生存 B 行をカバーしている

## 手順

### 1. 作業リストの確定

trace.md の根拠列（file:line）から対象ファイル集合を確定し、残存言及を機械列挙する:

```
node <dandori-repo>/skills/dandori/scripts/check-docs.ts residue <対象ファイル|ディレクトリ...>
```

B-ID トークンと dandori への言及が file:line つきで列挙される。**対象は今回の
フィーチャーが作成・変更したファイルに限る** — 並行フィーチャーのテストは
まだ gate 前で B-ID が現役のため、プロジェクト全体への一括適用はしない。

### 2. 言及の除去

- **テスト名**: B-ID トークンを剥がし、残りが自然文として成立すればそれを採用する。
  成立しなければ spec の該当 B 行（タイトル・Given/When/Then）から、振る舞いを
  説明する名前に書き直す
- **コメント**: dandori 言及（B-n / R-n / C-n / spec.md / design.md / dandori）を含む
  コメントは、内容が読者に必要なら言及なしの自然文に書き換え、プロセス管理のため
  だけなら削除する
- **機能的依存は削除しない**: dandori 成果物をデータとして読むコード・テスト
  （例: `.dandori/` 配下のマップをパースする整合テスト）は削除でなく**ユーザー裁定に
  浮上させる** — 成果物を長寿命ドキュメントとして `.dandori/` 外へ昇格するか、
  依存を残すか。残す裁定なら該当行に `dandori-ok: <理由>` コメントを付ける
  （residue 検査の除外マーカー — 裁定の記録を兼ねる）
- **振る舞いに触れない**: 変更はテスト名・コメントに限る。アサーション・ロジック・
  テストの追加/削除は禁止（それは cleanup ではなく別フィーチャーの仕事）

### 3. 検証（機械確認）

- residue を再実行して **exit 0** を確認する（`dandori-ok:` 行は除外される）
- ゲート（unit / e2e — `.dandori/resources.md` の正準コマンド）を再実行して
  **緑 + skipped/todo 0 件**を確認する — テスト名の変更が test filter や
  snapshot 名に波及していないことの確認

### 4. cleanup コミット

gate のコミットとは**分離した専用コミット**にする — 直前の履歴に B-ID つきで
トレース可能なスナップショットが残り、B 行↔テストの対応は後からでも git で
再現できる（trace.md 処分後の唯一の再現手段）。

### 5. クローズ

コミット完了後、**この順序で**行う（昇格元の処分が先行すると知見が失われる）:

1. `.dandori/map/` が存在すれば、dandori-survey の **update + promote** の実行を
   提案する — update は今回のコード変更の map への反映、promote は design.md /
   trace.md に蓄積されたフロー知見（実行検証済の既存コード挙動、不変条件、
   既存理解の訂正）の map への昇格。昇格元は次のステップで処分されるため、
   **必ず処分より前に**実行する。状態マップがあれば、design.md の「共有状態への影響」で
   宣言した writers/readers の増減を states.md に反映するのも update の一部
2. sketch.md・plan.md・trace.md・review-ledger.md は役目を終える（削除 or アーカイブ。リポジトリ方針に従う）
3. design.md は発見ログの spec 還流（dandori-gate §3）を済ませた上で同様に処分してよい
4. spec.md は長寿命ドキュメントとして残す
5. state.yaml: `phase: done`、`cleanup.status: done`、`phases_done` に cleanup 追加

## スキップ

B-ID つきの名前をコード上に残す方針のプロジェクト（トレーサビリティを履歴でなく
現物で保ちたい等）では、言及除去（§1〜4）をスキップして `cleanup.status: skipped`
（理由つき）とし、§5 のクローズだけ実施する。**クローズは省略不可** — スキップしても
フィーチャーは done にする。

## 完了条件

- residue exit 0 + ゲート緑 + cleanup コミット済み（または skipped 裁定）
- クローズ手順完了（使い捨てドキュメント処分・spec 残置）
- state.yaml: `phase: done`
