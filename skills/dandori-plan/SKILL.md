# dandori-plan — 実装計画

fix 済みの spec.md / design.md を、実装エージェントが自律的に消化できる
マイルストーン列に変換する工程。計画の質は「各マイルストーンに**何を渡さないか**」で決まる — 
ドキュメント全文を渡すと肝心の制約が希釈されて読み飛ばされる。

## 手順

### 1. マイルストーン分割

以下の条件を満たすようにマイルストーンを切る:

- 各マイルストーンは **spec.md の B 行の部分集合**に対応する（対応する B 行がない
  マイルストーンは存在してはならない — それはスコープ外の作業）
- 各マイルストーン完了時点で**単体で検証可能**（ゲートを通せる）
- 順序は依存関係を満たしつつ、**リスクの高いもの・設計の中核を先に**
  （後半での前提崩れは手戻りが大きい）
- 1マイルストーン = 実装エージェント1回分の作業量を目安にする

### 2. コンテキストマニフェスト作成

各マイルストーンについて、実装エージェントに渡すドキュメントセクションを明示する:

- 対応する B 行（spec.md の該当セクション）
- design.md の関連する改変箇所・新規実装・土台のエントリ
- **不変条件は常に全文**（部分開示の例外。これだけは希釈させない）
- sketch.md があるフィーチャーでは、UI に触れるマイルストーンに該当画面の変更マップを
  含める。**不可侵領域は常に全文**（不変条件と同じ部分開示の例外 — 「変えてはいけない
  場所」を知らない実装エージェントが巻き添え変更の主因）
- `.dandori/resources.md`（リソースマップ）に規約・参照ドキュメントがあれば、
  そのマイルストーンに関係するものの**パス**をマニフェストに含める

### 3. ゲート割り当て

各マイルストーンに、対応する B 行のゲートタグから導出した検証手段を割り当てる。
`unit`/`e2e` は実行コマンドまで具体化する。`visual`/`manual` はマイルストーン内では
実装エージェントの自己確認に留め、最終ゲート（dandori-gate）でユーザー確認に回す。

### 4. 確定

plan.md を書き出したら、カバレッジ検査を実行する:
`node <dandori-repo>/skills/dandori/scripts/check-docs.ts plan <spec.md> <plan.md>`
（未カバー B 行・spec に存在しない B-ID への参照・削除済み B 行への参照・
対応 B 行ゼロのマイルストーンを機械検出する）。exit 0 にしてから
ユーザーに概要（マイルストーン一覧と順序の根拠）を提示して確認を得る。
ここは spec ほどの壁打ちは不要 — 分割の妥当性を一目で判断してもらう程度。

## plan.md 形式（正準定義）

```markdown
# <feature> 実装プラン

## マイルストーン一覧
| # | タイトル | 対応 B 行 | ゲート |
|---|---------|----------|--------|
| M1 | <中核データパス> | B-1, B-2 | unit: `npm test -- xxx` |

## M1: <タイトル>
- 対応: B-1, B-2
- マニフェスト:
  - spec.md: §B-1, §B-2
  - design.md: 改変箇所 <エントリ名>, 新規実装 <エントリ名>, 不変条件（全文）
- 手順概略: <2〜5行。詳細手順ではなく方針>
- ゲート: <実行コマンド or 確認手順>
- 依存: なし

## M2: ...
```

## 完了条件

- plan.md が存在し、全 B 行がいずれかのマイルストーンにカバーされている
  （カバー漏れの B 行 = 実装されない仕様。カバレッジ検査 `check-docs.ts plan` の exit 0 で確認する）
- ユーザーが分割を確認済み
- state.yaml: `phase: impl`、`phases_done` に plan 追加、
  `impl.milestones_total` を設定
