# dandori-survey — コードベースの足場調査

既存コードベースの分析結果を `.dandori/map/` 配下のドキュメント群として整備・維持する工程。
spec の壁打ちや ground の土台調査が毎回ゼロから探索する代わりに、蓄積済みの分析から
出発できるようにする。プロセスフロー（spec→…→gate）の工程ではなく**足場整備** —
成果物は resources.md 経由で各工程に疎結合に供給される。

## 原則: 事実と解釈を分ける

| 種別 | 例 | 扱い |
|------|-----|------|
| ツールで再生成可能な**事実** | モジュール依存、宣言ライブラリ一覧 | 散文化しない。**再生成手段**を resources.md に記録し、必要時に都度取得する |
| コードから直接読み取れない**解釈** | アーキテクチャ、状態一覧、振る舞いフロー | `.dandori/map/` に Markdown として永続化する |

事実の散文化はドリフトの温床になる。数秒で再生成できるものはドキュメントにしない。

## 原則: 主張には証拠アンカーを付ける

map 内のすべての主張に、根拠となる `ファイルパス（必要なら :行 や シンボル名）` を併記する。
各 map ファイルの冒頭に生成時の commit hash を記録する。アンカーがない主張は
verify で機械的に検査できず、腐っても気づけない。

ground と同じ検証等級を主張単位で付ける: `[実行検証済]`（動かして確認、方法を併記）/
`[読解のみ]`（コードを読んだだけ）。

## モード判定

$ARGUMENTS とファイルの存在から判定する:

- `.dandori/map/` が存在しない → **generate**（初回生成）
- 存在し、鮮度確認や「腐ってないか見て」の依頼 → **verify**
- 存在し、コード変更後の反映依頼 → **update**
- 存在し、dandori フロー完了時の知見反映依頼（gate クローズからの遷移） → **promote**
  （update と同時実行が自然 — どちらも gate 直後がタイミング）

## generate — 初回生成

### 1. ツール選定

分析には構造把握系のツール（モジュール依存グラフ、ライブラリ一覧、コード骨格の
一覧など）が有効。**その場のコンテキストで利用可能なツールを確認し**、対象言語・
ビルドシステムに適したものがあればそれを使う。

適したツールがない場合は、勝手に劣化手段で進めず選択肢をユーザーに提示する:

- 手動探索で代替する（Grep/Read の組み合わせ。精度と網羅性は落ちる）
- 既存ツールを探す（言語エコシステムの依存可視化ツール等）
- 使い捨ての分析スクリプトを書く

### 2. 事実の収集（メインが実施）

モジュール依存・ライブラリ一覧を選定したツールで取得し、**内容ではなく再生成手段を**
resources.md に記録する（例: `` `skeleton modules .` — モジュール依存グラフ``）。
取得結果はこのあとのサブエージェント解析と裏どりの参照材料として使う。

### 3. 解釈の解析（サブエージェントに委任）

以下の観点ごとに独立サブエージェントへ解析を委任する。**メインの仮説・解釈を
プロンプトに混ぜない** — 渡すのは観点・対象パス・出力形式のみ:

| 観点 | 解析内容 | 出力先 |
|------|---------|--------|
| アーキテクチャ | レイヤ構成、依存の向きのルール、モジュールの責務分担、横断的関心事の扱い | architecture.md |
| 状態 | システムに存在する状態の一覧（エンティティのライフサイクル、ステータス列挙、フラグ）、種別、遷移、保存場所 | states.md |
| 共有状態 | 機能境界をまたぐ状態の台帳（保存場所×区分、書き手/読み手）。形式は `docs/appendix-state-map.md` の `dandori-state-map` ブロック | states.md 内 `## 状態マップ` |
| 振る舞いフロー | 基本的なユーザー行動フロー、入力→処理→出力の経路、外部システムとの接点 | flows.md |

各サブエージェントには「すべての主張に証拠アンカーを付けること」「確認できなかった
ことは推測で埋めず『未確認』と書くこと」を指示する。

共有状態の区分（owned/shared/borrowed）と SSOT の指定はコードから読めない
インフラ・組織知識を含む — サブエージェントには候補と根拠だけを出させ、
**確定はユーザー裁定で行う**（`docs/appendix-state-map.md`）。

### 4. 裏どり（メインが実施）

サブエージェントの主張を全件は検証しない。**外れたときの被害が大きい順**に傾斜をつける:

1. 状態一覧・振る舞いフロー — spec/ground が直接依拠するため最優先。可能な限り
   実行して確認（テスト実行、使い捨てスクリプト、アプリ起動）
2. アーキテクチャの依存ルール — 事実（§2 の依存グラフ）と突合して矛盾を検査
3. その他 — アンカー先の存在確認（spot check）

裏どりの結果で等級を確定する。検証しなかった主張は `[読解のみ]` のまま残してよい —
等級が付いていることが重要で、全件 `[実行検証済]` にすることが目的ではない。

### 5. 書き出し

- `.dandori/map/architecture.md` / `states.md` / `flows.md` を作成
- resources.md に map 各ファイルと再生成コマンドのエントリを追記
  （resources.md が無ければ作成を提案する）

## verify — 鮮度検査

1. 各 map ファイル冒頭の commit hash と現在の HEAD を比較し、その間の変更ファイル
   一覧を取得する（`git diff --name-only <hash>..HEAD`）
2. map 内の証拠アンカーを走査し、変更ファイルに載っている主張・アンカー先が消滅した
   主張を列挙する
3. 腐った主張の一覧を「観点 / 主張 / 腐り方（アンカー先変更・消滅）」の表でユーザーに
   提示する。**サイレントに修正しない** — 修正するかは update として裁定を仰ぐ

## update — 差分更新

verify の結果（または明示された変更範囲）を入力に、影響を受ける主張だけを再解析・
再検証して書き換える。全面再生成はしない。更新した map ファイルの commit hash を
現在の HEAD に更新する。

dandori-gate 通過後（= コードが変わった直後）が自然な実行タイミング。gate 完了時に
map が存在すれば、survey の update を提案してよい。

## promote — フロー知見の昇格

dandori フローの各工程（ground / spike / impl / gate）で判明した事実のうち、
**次のフィーチャーでも真であり続けるもの**を map に昇格させる。昇格元は使い捨て
ドキュメント（design.md, trace.md）— gate のクローズで処分される前に実行すること。

### 1. 候補の収集

design.md を走査し、以下を昇格候補として抽出する:

- **土台**の `[実行検証済]` 前提 — 既存実装の挙動を実際に動かして確認した記録。
  map に無い主張、または map で `[読解のみ]` だった主張の等級を上げるもの
- **不変条件** — 他モジュールが依存する挙動、データ形式、互換性制約
- **発見ログ**のうち「既存コードの理解と現実のズレ」系 — map の既存主張の
  訂正を含む（今回のフィーチャー実装のつまずき記録は対象外）
- spike の検証結果（design.md に還流済みのものはそちら経由で拾える）

### 2. 選別基準

昇格するのは既存コードベースについての一般的な事実・解釈だけ:

- ✅ アーキテクチャ・状態・振る舞いフローについて、フィーチャーをまたいで通用する主張
- ❌ 今回のフィーチャー固有の実装詳細（spec.md とコード自体が正）
- ❌ ツールで再生成可能な事実（原則どおり、散文化せず resources.md の再生成手段に委ねる）

### 3. 提示と反映

昇格候補を表でユーザーに提示し、承認を得てから書き込む（サイレントに map を書き換えない）:

```markdown
| 主張 | 昇格先 | 等級 | 既存主張との関係 |
|------|--------|------|------------------|
| <主張> | states.md | [実行検証済: ...] | 新規 |
| <主張> | flows.md | [実行検証済: ...] | 訂正（旧: <旧主張>） |
```

- 等級と検証方法は design.md のマークをそのまま引き継ぐ。検証が使い捨てスクリプトに
  よるものだった場合、スクリプト自体は残らないので**検証方法を根拠欄に書き残す**
- 反映した map ファイルの commit hash を現在の HEAD に更新する（update と同時実行なら一度で済む）

## map ファイル形式（正準定義）

```markdown
# <観点名>

<!-- generated-at: <commit hash> / <日付> -->

## <セクション>
- <主張> — 根拠: `src/foo/bar.ts:42` [実行検証済: テスト xx 通過]
- <主張> — 根拠: `src/baz/` [読解のみ]
- <調べたが確認できなかったこと> — 未確認
```

## 完了条件

- generate: map 3ファイルが存在し、全主張に証拠アンカーと等級が付いている。
  resources.md に map と再生成コマンドのエントリがある
- verify: 腐った主張の一覧（ゼロ件ならその旨）をユーザーに提示した
- update: 影響範囲の主張が更新され、commit hash が HEAD と一致している
- promote: 昇格候補の表をユーザーに提示し（ゼロ件ならその旨）、承認分が map に
  反映されている。commit hash が HEAD と一致している
