# dandori-spike — 検証スパイク

最もリスクの高い前提を、本実装の前に最小の使い捨てコードで検証する工程。
前提の誤りを最終ゲート（工程の最後）ではなく plan の手前で検出するための保険。
ドキュメントレビューは静的検証にすぎず、机上で整合的に間違った設計はレビューを通過する — 
それを現実に突き合わせるのがこの工程。

## 手順

### 1. スキップ判定

design.md のリスクランキングを読む。以下なら **スキップ**:

- `[読解のみ]` の前提が存在しない（すべて実行検証済み）
- 残る `[読解のみ]` 前提がすべて「被害=低」

スキップ時は state.yaml に `spike.status: skipped` と理由を記録し、dandori-plan へ。
**スキップは根拠つきの判定であり、省略ではない**。

### 2. スパイク実施

ランキング上位（原則1件、被害=高が複数あれば最大3件）について:

- **検証したい前提を1文で言語化**し、「何が観測できれば前提は正しいと言えるか」を先に決める
- 使い捨ての最小コードで検証する。場所はスクラッチディレクトリまたは
  `specs/<feature>/spike/`（.gitignore 推奨）。プロダクションコードには触れない
- 動かして観測する。読んだだけの結論は不可

### 3. 結果の還流

結果を design.md に反映する:

- 前提が**正しかった** → 該当前提のマークを `[実行検証済: spike]` に更新、発見ログに1行
- 前提が**崩れた** → 発見ログに記録し、影響範囲を判定:
  - design.md の修正で吸収できる → 修正して続行
  - spec.md のゴール・振る舞いに波及する → **フェーズを逆行**（state.yaml の
    `phases_done` から該当フェーズを外す）し、ユーザーに報告して spec/ground からやり直す。
    逆行はプロセスの失敗ではなく、最終ゲートでの手戻りを未然に防いだ成功

### 4. 後始末

スパイクコードは**本実装に流用しない**（品質基準を満たしていない前提で書かれたコード）。
知見だけを design.md に残し、コード自体は削除するか spike/ に隔離したまま放置する。

## 完了条件

- state.yaml: `spike.status: done`（または skipped + 理由）、`phase: plan`、
  `phases_done` に spike 追加
- 検証結果が design.md の発見ログに記録されている
