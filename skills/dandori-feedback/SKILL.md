# dandori-feedback — 継続改善（フィードバック受け入れと巻き戻し）

gate と cleanup の間に立つ**裁定点**。他者レビュー・運用・仕様変更の議論はセッションの
外で起き、この工程にはその**結論だけ**が持ち込まれる。結論を台帳化し、項目ごとに
巻き戻し先をトリアージして、フィーチャーを改訂サイクル（revision）として通常フローに
再合流させる。改訂がもう来ないと確定したときだけ cleanup（不可逆な店じまい —
B-ID 除去・使い捨てドキュメント処分）へ送る。spec.md が長寿命ドキュメントとして
残るのはこの工程のため — 改訂は常に「fix 済み spec への差分」として入る。

賛否の再討議はこの工程の仕事ではない。結論の**翻訳**（spec の語彙への落とし込み）と
**配線**（どのフェーズからやり直すか、いつ店じまいするか）に徹する。

## 入口条件

いずれかの文脈で入る:

- **gate 直後**（通常経路）: state.yaml が `phase: feedback`（gate 通過・コミット済み、
  cleanup 未実施）。B-ID はテスト名に現役で残っている
- **done からの再開**: state.yaml が `phase: done`（cleanup 済み）のフィーチャーに
  ユーザーが結論を持ち込んだ。B-ID は剥がし済み — 再 gate は差分トレース（§5）になる

done 前（gate 未通過）のフィードバックはこの工程の対象外 —
review / codereview / gate 裁定の守備範囲。

## 手順

### 1. 裁定 — 取り込むか、待つか、店じまいするか

改訂サイクルを開く前に、**ユーザーに裁定を仰ぐ**。この工程が推測しない。

- **取り込む結論がある**（仕様変更・改善指示・不具合報告の裁定結果）→ §2 へ
- **実装は完全に fix — 改訂はもう来ない** → state.yaml を `phase: cleanup` にして
  **dandori-cleanup へ遷移する**。以降のフィードバックは done からの再開（差分トレース）に
  落ちることを一言添える
- **まだ結論待ち**（外部レビューが進行中 等）→ **そのまま待機**。`phase: feedback` のまま
  セッションを終えてよい（安定点 — 再開時にルーターがこの工程へ戻す）。
  done からの再開でこの選択肢に相当するのは「何もせず終了」（state.yaml は触らない）

### 2. 改訂サイクルの開始と台帳化

state.yaml を更新する: `revision: <前回 + 1>`（初出は 2）。done からの再開では
`phase: feedback` にも更新する。

持ち込まれた結論を review-ledger.md に 1 項目 1 行の **F-n 行**で記録する。
gate 直後の経路では前サイクルの台帳（R/C 行入り）が生きているのでそこへ追記、
done からの再開では再生成する。形式は review / codereview と同じ台帳
（`| ID | Rd | 深刻度 | 論点 | 処置 | 根拠・理由 |`）で、**Rd には今回の revision 番号**を書く。
形式・処置の完全性は `check-docs.ts ledger` が機械検査する（F 系列対応 — 収束判定の
対象外で、完了条件は未処置ゼロ）。

### 3. トリアージ — 項目ごとの分類と巻き戻し先

各項目を分類し、処置列を埋める:

| 分類 | 巻き戻し先 | 処置の書き方 |
|------|-----------|-------------|
| **仕様変更**（観測可能な振る舞いが変わる） | spec（壁打ちが必要な場合）/ ground・impl（結論が B 行に機械的に落ちる場合 — §4 でこの工程が spec を改訂して短絡） | 反映済 + 根拠に追加した B-ID |
| **前提・設計の変更**（振る舞いは同じだが土台・方式が変わる） | ground | 反映済 + 根拠に改訂スコープ一言 |
| **振る舞い不変の改善**（リファクタ・性能・可読性） | impl。ただし可能なら観測可能な B 行に言い換えて仕様変更として扱う（性能なら測定条件を Then に） | 反映済 + 根拠に改訂スコープ一言 |
| **別フィーチャー**（この spec の守備範囲外・別 spec に波及） | — 新規フィーチャーとして切り出し | 却下 + 理由「別フィーチャー <name> へ」 |
| **取り込まない** | — | 却下 + 理由（保留は minor のみ） |

巻き戻し先は取り込む全項目の**最深部**に揃える（spec > ground > impl）。

**取り込む項目がゼロ**（全項目が却下、または minor の保留のみ）になったら、
改訂サイクルは開かない:

- 却下項目は理由ごと spec.md の「スコープ外」へ、保留項目は「未解決事項」へ還流する
  （台帳はいずれ cleanup で処分される — 残すべき判断は長寿命ドキュメントへ）
- `revision` を入口前の値に戻す（初回なら削除）— B 行を 1 本も足さずに終わった
  サイクルは採番を消費しない
- gate 直後の経路なら §1 の裁定（待機 or cleanup へ）に戻る。done からの再開なら
  再生成した台帳を処分して `phase: done` に戻す

### 4. spec 改訂（結論が機械的に落ちる場合）

結論が B 行へそのまま翻訳できる（要件の再検討・壁打ちが不要な）場合は、
この工程で spec.md を改訂して spec フェーズを短絡してよい。fix 済み spec の編集規約に従う:

- **改番禁止**。振る舞いの変更は旧 B 行に取り消し線 + 理由、新 B 行を末尾追加
- 改訂で追加した B 行には **`- Rev: <今回の revision>`** フィールドを付ける
  （どのサイクル由来かの記録。cleanup 済み後の再開では gate の差分トレースが
  「今回検証すべき行」を識別する目印になる）
- 編集後は改番検知 + 形式検査を通す:
  `git show HEAD:<spec.md> > /tmp/base.md` →
  `node <dandori-repo>/skills/dandori/scripts/check-docs.ts spec <spec.md> --baseline /tmp/base.md`。
  状態モデルがあれば `check-state-model.ts` も再実行する
- 改訂版 spec はユーザー確認を得て fix

要件の再検討が必要な項目が 1 つでもあれば spec.md はここでは触らず、
dandori-spec へ巻き戻す（`Rev:` 付与の規約は spec 工程が引き継ぐ）。

### 5. コース再判定と再 gate 範囲の裁定

- **コース再判定**: 改訂スコープに対してルーター §2 のトリアージ基準を再適用する。
  元がフルコースでも、改訂が局所的なら短縮コースでよい（逆もある — 局所修正のつもりが
  スキーマ変更を含むなら改訂はフル）。`course:` を更新し、変更したら台帳の根拠列に記録
- **再 gate 範囲**（**done からの再開のみ** — gate 直後の経路では B-ID が現役なので
  通常の全行トレースが機械で効き、この裁定は不要。`feedback.trace_scope` も書かない）:
  既定は**差分トレース**（`feedback.trace_scope: delta`）— 今回の revision の B 行だけを
  フルトレースし、旧 B 行はスイート緑の回帰で担保する（旧 B 行↔テストの対応の正は
  cleanup 前の最終 gate コミット。git 履歴で再現できる）。以下のいずれかに該当したら
  **フル再トレース**（`trace_scope: full`）を提案してユーザーと裁定する:
  - 既存 B 行の過半に取り消し線が入る
  - 状態モデルの軸・状態が変わる
  - 不可逆変更（DB スキーマ・公開 API・データ移行）を含む
- full の場合、旧 B 行↔テストの対応は最終 gate コミット時点の trace.md と
  cleanup コミットの rename 差分から再構築する。テスト名に B-ID を復元する必要はない —
  gate は根拠明記の手動対応付けを許す

### 6. 巻き戻し（state.yaml の改訂サイクル初期化）

state.yaml を巻き戻し先に合わせて初期化し、該当工程スキルへ遷移する:

- `phase:` を巻き戻し先（spec / ground / impl）に設定
- `phases_done:` をリセット — spec 巻き戻しなら `[]`、§4 で spec 改訂済みなら `[spec]`
- 再走する工程の per-phase セクション（review.rounds / impl.milestones_* /
  codereview / refine 等）を初期状態に戻す — 前サイクルの記録は git 履歴が正。
  sketch は改訂スコープで再トリアージする（UI に触れるか）
- `feedback.items:`（台帳の F 行数）を記録。done からの再開なら `feedback.trace_scope:` も
- `node <dandori-repo>/skills/dandori/scripts/check-docs.ts state <state.yaml>` を通す

以降は通常フローに再合流し、gate を通ってこの工程（§1 の裁定）に戻ってくる。
フィーチャーが done で閉じるのは、§1 で「完全に fix」の裁定が出て
cleanup が完了したときだけ。review-ledger.md の F 行は改訂サイクルの間、
取り込んだ結論の追跡台帳として生き、処分は cleanup が行う。

## 完了条件

以下のいずれか:

- **cleanup へ遷移**（§1 で完全 fix の裁定）— state.yaml が `phase: cleanup`
- **待機**（§1 で結論待ち）— `phase: feedback` のまま中断。state.yaml は整合していること
- **クローズ維持**（done からの再開で取り込みなし）— `phase: done` のまま
  （§3 経由なら却下・保留の spec 還流と revision の巻き戻しを済ませる）
- **巻き戻し完了** —
  - 全項目が台帳で処置済み（`check-docs.ts ledger` の未処置ゼロ、却下は理由つき）
  - state.yaml: revision 採番済み・phase が巻き戻し先・phases_done とセクションのリセット済み・
    `check-docs.ts state` グリーン
