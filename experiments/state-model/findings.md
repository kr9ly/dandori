# 実験所見: cart-payment §2 の状態モデル化と静的検査

日付: 2026-07-06
対象: ec-replace/.dandori/specs/cart-payment/spec.md（fix 2026-07-04、レビュー R1〜R23 済み）
方法: §2 正常系バリエーションを状態モデル（cart-payment.model.ts）に全訳し、
checker.ts の 7 検査を「現行」と「fix 時点復元（--as-of=fix）」の 2 モードで実行。
過去レビューラウンドの指摘（spec 内 R 番号注釈）を正解ラベルとして照合した。

## 結果サマリ

23 ラウンドのレビューを通過した spec に対して、機械検査が**実質的な新規指摘 6 件**を出した。
一方、バックテスト（レビュー指摘の後追い再発見）は限定的 — 組み合わせ系の検査で捕まるのは
レビュー指摘のうち特定の型だけで、大半の型は守備範囲外。「レビューループの代替」ではなく
「レビューが構造的に見ないものを埋める補完」と位置づけるのが正しい。

## 新規指摘（現行モデルでの真陽性）

1. **宣言済み交点 S5×S7（pending × クーポン）に覆う B 行がない**（検査2）。
   B-4 の Then は「ERP 連携なし」を主張するが Given にクーポンがない —
   クーポンなしでは ERP はそもそも呼ばれず、主張は空虚に真。pending が ERP を
   スキップすることを観測するテストは spec 上存在しない。
2. **同 S5×S21（pending × ピックアップ）**（検査2)。B-4 は「案内メールなし」を主張するが
   Given に PickupItem がなく、B-20 は明示的に「pending でない決済」。交点はどちらの行でも
   実行されない。R2 major-2（在庫更新は無条件）はこの交点の意味論の指摘であり、
   fix 時点でこの検査が交点 B 行を要求していれば、専用行の執筆時点で
   「メールはスキップ・在庫更新は無条件」の分解を強制できた蓋然性が高い（部分的ヒット）。
3. **B-19 の only 制約違反**（検査4）。order_no=preassigned は Wallet Pay 限定
   （S20→S4/S5）だが、B-19 の Given は決済メソッド無指定（基準 = card に展開される）。
   card カートに order_no を直接シードする「到達不能な状態のテスト」を書いてもパスしてしまう。
4. **stock 軸（商品単位）の混在が未検討**（検査7）。offline-store-order の spec は
   「B × C: 商品単位で独立 — 混在が成立する」を宣言しているが、cart-payment §2 は
   received/reservable 混在に言及なし。異常系 B-41 には混在 Given があるのに正常系にない。
5. **coupon × fixed_discount が未分類だが実は依存**（検査3 真陽性）。B-6 の Then 内(3)
   「固定額割引は対象商品リスト × 決済方法条件を満たす最初の 1 商品にのみ加算」は
   この 2 軸の相互作用そのもの。軸間関係に未宣言で、両方を Given に持つ B 行もない。
6. **coupon × promo が未分類だが実は依存**（検査3 真陽性）。B-6 の Then 内(2)
   「プロモコードのみの商品は除外」は組み合わせ挙動なのに、宣言もカバーもない。

5・6 は「Then の奥に散文で埋まった相互作用が、軸間関係という契約面に昇格していない」
という型で、ペア全数性検査がなければ今後も沈んだままだった。

## 摩擦とフォーマットへの示唆

- **ペア全数性の指摘 30 件は生のままでは多すぎる**。12 軸 = 66 ペアのうち 36 が既宣言、
  30 が未分類。ただし大半（route × pickup 等）は自明な直交で、一括宣言構文があれば
  1〜2 行で 25 件以上消える。必要なのは:
  - グループ直交宣言（「軸5系ブールは相互および他の全軸と直交、例外は個別列挙」）
  - あるいは「直交がデフォルト・依存だけ列挙」モード。ただし cancel-order-product の
    B-38/B-39（レビューで発覚した交点）を見る限り、明示裁定を挟まないデフォルト直交は
    危険 — グループ宣言で「裁定の痕跡」を残す方が dandori の思想に合う
- **軸の分解はモデル化の主作業**。spec の軸2（排他4値）・軸5（排他6値のふり）は
  そのままではモデルにならず、独立軸への分解が必要だった。この分解作業自体が
  仕様バグ（5・6）の発見源になっており、単なる転記コストではない
- **observation / modifier の語彙は必須**。S14/S22 を軸として直積に入れると
  無意味な組み合わせ要求でノイズが爆発する。spec の散文（「S14 は観測」「S22 は修飾」）を
  種別として書けるだけで足りた
- **検出できない型の境界は明確**: (a) 軸内逐次遷移（連鎖）、(b) 分類述語の値域網羅、
  (c) 未発見の状態変数、(d) 値内部のセマンティクス。(b) は formal レーンに接続可能
  （B-53 と同様の PBT/全列挙）。(a) はフォールバック軸に遷移グラフ宣言を要求する拡張で
  一部救える可能性があるが、今回は未検証

## 実験1: ロールバック横断のモデル化（cart-payment-rollback.model.ts）

§3 の横断関心事「失敗タイミング × 確保済みリソース」を 3 軸
（failure_phase / shipping_state / cart_recreated）でモデル化し、バックテストを実施。

**結果: R4 blocker-2 の両方を fix 時点復元で完全再発見**。

- 値カバレッジが「p6_stock_shipparam をカバーする B 行がない」を指摘
  = 旧 B-46 の「フェーズ0〜5」がフェーズ6 を取り落とした件そのもの
- 同じく「shipping_state=without_id をカバーする B 行がない」
  = 旧 B-47 が shipment_id=None の 2 態目を欠いた件そのもの
- 現行モデル（R4 反映後）はグリーン

反実仮想の公正性: failure_phase の値域は §3 マトリクスのフェーズ構造から、
shipping_state の値域は §2 の S17/S18 の記述（hold_payed / failed Shipping は
shipment_id なし）から、いずれも fix 時点の spec 本文の機械的転記で書けた。
optout 型（未発見変数）と違い、モデルさえ書いていれば捕まえられた指摘と言える。

**含意**: 正常系の軸だけでなく、異常系の横断関心事（ロールバック・補償処理）こそ
モデル化の投資対効果が高い。§3 が散文で宣言する「失敗タイミング × 確保済みリソース」は
そのまま 2 軸 + only 制約に写り、フェーズ分割の写経だけで取り落とし検出が機械化される。

## 実験2: グループ直交宣言によるノイズ収束

orthogonal_groups 構文と `軸×'*'` 宣言を追加し、未分類 30 ペアへの裁定案
（spec 本文からの推定 — ユーザー未確認マーク付き）を投入した。

**結果: ペア指摘 30 件 → 3 件、全体 34 件 → 10 件に収束**。

- 30 件中 27 件は spec 本文の記述から即裁定できた（直交 23 / 依存 4）
- 残 3 件（promo×fixed_discount / fixed_discount×pickup / wms×pickup）は
  spec 本文から裁定不能 — コード確認（ground 相当）が必要な真の開いた問い。
  「チェッカーの指摘残 = ground の確認項目リスト」という接続が成立する
- 裁定の過程で発見された真の依存 4 件（coupon×fixed_discount、coupon×promo、
  coupon×pickup、sheets_only×stock）は依存宣言に昇格 → 検査2 が
  「宣言済みだが覆う B 行がない」5 件を要求リストとして出力する状態になった。
  未分類（何も分からない）が「宣言済み・未カバー」（何をすべきか明確）に変換される

## 実験4: 既存 spec 3 本への遡及適用（2026-07-06 — ec-replace 全 plan 済み spec）

対象: modelx-cart-payment（軸セクションあり・軸間関係なし）/ offline-store-order（軸 +
軸間関係あり）/ cancel-order-product（軸セクションなし・フェーズ構造）。各 spec を独立
エージェントがモデル化 → checker 実行 → メインが旧実装の確認を挟んでトリアージ。

### 成果（実質的な新規指摘 → spec 反映済み）

- **modelx: 3 件** — 軸間関係宣言の全面欠落（社割×引き取りは ground で完全独立を確定）/
  PickupQuota 3 分岐が B-4 の Then に埋没（S4a〜c に命名）/ シングルクォート除去の
  Given バリエーション欠落（除去がシート書き込みのみである事実も ground で確定）
- **offline: 2 件** — quantity 混在の per-item 未テスト（旧実装の確認で配送グループIDの
  商品跨ぎ共有 + 仮ID グローバル連番という**実依存**を発掘 — 宣言済み「独立」の反例側）/
  「B×C 独立」宣言に混在 Given のカバー行がない（B-42/B-43 追加）
- **cancel: 3 件** — 大型/小型混在の合成挙動（include_unconfirmed_cancel の OR・Slack 合算）/
  注記に埋まった交点 2 件の B 行昇格（B-44/B-45）
- レビュー通過済み spec からの追加 B 行: cart 5 + offline 2 + cancel 3 = **10 本**。
  レビュー実績（modelx R8 / offline R23 / cancel R8）と無相関に一定率で出る — 検査の
  守備範囲がレビューと直交している傍証

### 摩擦（3 エージェントの実測 + トリアージ側の所見）

1. **per-item 軸の粒度問題が最重要**（offline）: order レベル軸に押し込むと checker の
   base-value 省略ルールで交点がトリビアルに covered 判定され、真の欠落を検出できない。
   per-item 軸を別立てすると依存関係が複雑化 — 語彙設計の宿題
2. **base 値の暗黙宣言が検査をすり抜ける**（offline）: 基準シナリオが軸を明言していない場合
   base 選定自体が推定になり、「全 B 行が base を自動カバー」の規則が未検証前提を隠す。
   base に「宣言済み/推定」のマークが必要
3. **軸なし spec の逆算は恣意性が高い**（cancel）: フェーズ=軸変換で gate_error 型の排他軸に
   押し込むと交差構造（B-39 の 2 独立条件）が見えなくなる。16 軸 120 ペアで分類作業が
   破綻寸前 — 軸セクションを spec 執筆時に書かせる（integration-design の本来の運用）が正で、
   遡及適用は「注記に埋まった交点の発掘」までが実用範囲
4. **語彙の不足 3 種**（modelx/offline）: (a) B 行 Then から発掘された S 番号なし多値軸、
   (b)「値は存在するが出力に反映されない」（強制上書き）、(c) 単発バグ再現 B 行を
   直積検査から除外する別カテゴリ（軸を専用に立てて '*' 直交を量産する現状はコスト過剰）
5. **機械的直交裁定の圧力**（modelx）: 軸間関係のない spec では「全数性を満たすため」に
   吟味なしで直交宣言してしまう — チェッカーの意図（裁定の存在の強制）が
   「裁定のふりの量産」に転化しうる。グループ直交宣言 + ground 送りの明示が対策

### 統合設計への含意

- 導入ステップ 1（新規 spec 1 本で人力運用）の摩擦実測はこの遡及適用で先取りできた。
  次の新規 spec（admin-gift-order 予定）では軸セクション執筆と同時にモデルを書く本来運用を試す
- checker 改修候補: per-item 軸の交点判定（base 省略ルールの適用除外）、base の
  宣言済み/推定マーク、one-off カテゴリの追加

## 実験 5: admin-gift-order — 本来運用の初投入（spec 執筆時に B 行と同時にモデルを書く）

2026-07-06。ec-replace の admin-gift-order（POST /admin/gift/、単純 CRUD 系）で
実施。壁打ち §2 の軸出し → §4 の B 行 + Covers 同時執筆 → チェッカー → fix の本来運用。
チェッカーは製品版（skills/dandori-spec/scripts/check-state-model.ts）を使用。

### 経過

- §2 で軸 5 本（ラベル解決 / memo / postal 表記 / phone 表記 / EAW 境界）を散文で出し、
  §4 のモデル化時に EAW 軸を name/addr の独立 2 軸に分解（ノウハウ 1 の「排他値のふりをした
  独立変数」を執筆時に自己適用）。最終 6 軸 13 値・B 行 29 本（モデル対象 8 / one-off 21）
- 初回チェッカー実行から exit 0（指摘ゼロ、ground 送り 1 件）。同時執筆では
  「Covers を書くために軸との対応を考える」行為自体が B 行の Given を精密化するため、
  後付け遡及で出た「注記に埋まった交点」型の指摘が原理的に出にくい
- 差分プライシングの手動試行 1 回: 「memo 空文字対応」の要望を模擬 → 軸値 +1 だけ入れて
  再実行 → 検査 1 が「値 B3 をカバーする B 行がない」を 1 件出す。
  値札 = **+軸 0 / +値 1 / +依存 0 / 要求 B 行 +1**。B-29 を 1 本書いてグリーン復帰。
  往復 2 分弱 — 複雑性スコア化の最小単位として機能する感触

### 摩擦（本来運用 vs 遡及適用の差）

1. **異常系 B 行の Covers ルールが未整備**: 裁定「エラーマトリクス行は形式化しない」と
   検査 0「全 B 行に Covers 必須」の間に隙間があり、エラー系 B 行 21 本に
   `Covers: one-off — 異常系 EXX` を貼る運用で埋めた。one-off の本来意味（単発バグ再現）
   から「モデル対象外の汎用マーカー」へ意味が拡張されている。レポートの one-off 列挙も
   21 件でノイズ気味。**語彙候補: `Covers: out-of-model — 理由`（one-off と区別）**
2. **値 ID の命名衝突**: §2 の軸記号（A/B/C/D/E + 数字）が §3 異常系マトリクスの
   E 番号（E1a〜E5c）と衝突。EAW 軸の値を name_lt25 / addr_eq64 型に改名して回避。
   S 番号統一（appendix の推奨）を §2 執筆時から守るべきだった — 本来運用でも
   「§2 の記号設計」への指針が要る
3. **摩擦らしい摩擦はこの 2 点のみ**: 同時執筆のオーバーヘッドは体感ほぼゼロ
   （8 本の Covers 執筆 < 1 分）。遡及適用の摩擦 3（軸なし spec の逆算破綻）は
   本来運用では発生しない — integration-design の見立てどおり
4. **単純 spec でも元は取れた**: 軸 6 本と少なめだが、モデル化の過程で
   (a) EAW 軸の分解、(b) memo 空文字の第 3 値（差分プライシング経由で正式採用 —
   旧実装の TextField の ""/NULL 別値は互換観測点）の 2 つが執筆時に出た。
   発動条件「軸 3 つ以上」は妥当な下限

### 統合設計への含意

- checker 改修候補（追加）: `out-of-model` カテゴリの新設（one-off との意味分離）、
  one-off 列挙の折りたたみ（件数のみ表示オプション）
- 差分プライシングは「軸値の追加 → 検査 1 の要求本数」で最小実装が既に成立している。
  スコア化はチェッカー出力の diff を取るだけで開始できる
