# 構想メモ: モバイル適用時の状態モデル — 2.5次元（時間交差する状態）の扱い

## 前提: 状態空間の次元という見方

- **2次元（スナップショット）**: ある瞬間にどの状態の組み合わせが妥当か。現行の
  状態モデル（軸 × 値域、直交/依存宣言、直積カバレッジ）が扱っているのはこれ
- **3次元（軌道）**: 時間軸上でどの遷移列が妥当か。状態が時間軸で交差する。
  ゲームプログラミングが典型 — 毎フレーム全状態が交差しうるので軌道空間が生成的
- **2次元で回る条件はマルコフ性**: 次の状態が「現在のスナップショット + 入力」だけで
  決まり、履歴が漏れていないこと。バックエンドの request/response はアーキテクチャが
  時間を外部化している（毎サイクルリセット + 軌道問題は DB トランザクションに委譲）
  ので、現行モデルの2次元前提が成立していた
- **構造化とは履歴をスナップショットに畳み込む操作**: ステートマシン化（履歴→現在状態
  enum）、イベントソーシング（軌道→再生可能なデータ）、フェーズ順序の固定

## モバイルが 2.5 次元である理由

時間交差を起こせる主体が**有限で、プラットフォームが定義している**:
ライフサイクル遷移、config change、プロセス死と復元、接続変化、push / deep link 着弾、
二重タップ、応答順の逆転。割り込みの語彙はほぼこれで閉じる。

軌道の自由度が丸ごとあるのではなく、「スナップショット的な本体 × 有限個の割り込み源」
の積で書ける — 割り込み点の選択だけが時間軸に残る。それが 0.5 次元分。

さらにモバイルフレームワークの歴史は畳み込み装置の積み上げでできている:

| 装置 | 畳み込んでいるもの |
|------|-------------------|
| back stack | ナビゲーション履歴 → データ |
| ViewModel | config change をまたぐ生存 → 履歴の区別を消す |
| SavedStateHandle | プロセス死 → スナップショット化 |
| structured concurrency（viewModelScope 等） | 非同期の生存期間 → 画面スコープに縛る cancellation |
| 宣言的 UI / UDF | レンダリング過程 → UI = f(state) の純関数 |

**残る 0.5 次元 = どの装置もカバーしていない交差**。

## 現行モデルに既にある片鱗（2つ）

1. **異常系の「失敗タイミング軸」**（フェーズ構造をそのまま値域に写す）— 時間を
   スナップショット軸に畳む操作そのもの。ただし軌道が1本の線形フェーズ列である
   バックエンド前提
2. **「確保済みリソース」** — 割り込み点の同値分割の基準（どのリソースを持った状態で
   割り込まれるか）を先取りしている

拡張はこの延長線上で行う。新機構の発明ではなく既存2機構の一般化として設計する。

## 拡張候補（3つ）

### 1. 割り込み軸のプラットフォームカタログ化

ライフサイクル・プロセス死・接続変化はフィーチャーごとに発明する軸ではなく、
アプリ全体で共有される軸。states.md の共有状態マップ（appendix-state-map.md）と
同じ扱いで survey 層に一度だけ定義し、各 spec は `ref:` で参照する。
値域はフレームワークが決めているので `partition_of:` の全域性検査がそのまま効く。

### 2. 割り込みタイミングの同値分割

「操作の途中で X が起きる」の「途中」を、**確保済みリソースの集合で商を取って**
同値クラス化する（リクエスト発行前 / in-flight / 応答後描画前 / …）。
これで軌道の網羅が E 網羅性の語彙（分割の全域性）に還元される。
失敗タイミング軸の一般化: フェーズ列が1本でなくても、リソース保有状態の変化点で
区切れば同じ形式になる。

### 3. 直交理由の新しい型: 装置による直交（構造的直交）

現行の `orthogonal:` の reason は意味論的（「出庫結果は WMS レスポンスのみで決まる」）。
モバイルでは「viewModelScope の cancellation がこの交差を潰す」のような
**構造的直交**が主役になる。装置名を理由として書かせると:

- ground 工程で「その装置が本当に効いているか」をコードで検証できる —
  直交宣言が徳目でなく検査可能になる（例: scope の張り方を実コードで確認、
  cancellation を落とすミューテーションで検出）
- どの装置名も書けないペアが dependent 交点か ground 送り —
  **その一覧がそのまま 0.5 次元の全リスト**になる

記法案（未検証）: `orthogonal: [{ axes: [fetch, navigation], device: viewModelScope-cancellation }]`
— `reason:` と別キーにするかは実地で判断。

## DRIVE 側の対応 — R の軌道拡張

fixture 語彙に「**基準軌道 + 割り込みクラス k で中断**」という軌道差分が要る。
成立条件は virtual time・ディスパッチャ注入（TestDispatcher 等）・プロセス死
シミュレーション。Covers のデルタ表現も `base + interrupt=in_flight` のような
軌道差分を許す形に拡張が要るはず。

## 実地で確認する問い

- [ ] 割り込みカタログは本当に閉じるか。アプリ固有の割り込み源（例: プレイヤーの
      バックグラウンド再生、WebView とのブリッジ）はカタログ拡張で吸えるか
- [ ] 確保済みリソースによる商は粗すぎないか / 細かすぎないか。in-flight の中を
      さらに割る必要が出る操作はあるか（例: 楽観更新の適用前後）
- [ ] 構造的直交の検証は ground の作業量として現実的か。装置ごとの検証手順は
      定型化できるか
- [ ] 応答順逆転・多重発火（操作 × 操作の交差）は割り込み軸で表現しきれるか。
      操作ペアの dependent 交点として別扱いが要るか
- [ ] 2.5 次元でも直積は爆発しないか。割り込み軸 × 全操作軸の積を only 制約で
      どこまで刈れるか
